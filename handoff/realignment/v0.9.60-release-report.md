# Release Completion Report — v0.9.60

Boot-time npm package detection. Pre-v0.9.60 the dashboard told operators "Install: npm install -g @playwright/mcp" for every awaiting-pool entry that had a known canonical package, but never checked whether the package was already on disk. v0.9.60 adds a lazy first-snapshot scan of known npm-root candidates, caches yes/no per known entity id, and flips the dashboard line from "Install: ..." to "Already installed: <package> [DETECTED]" when the canonical package is present.

## Scope completed

### `performNpmPackageDetectionLocked` + `packageDetected(entityId)`

Lazy-cached detection on first snapshot call. The scan logic:

1. **Infer the operator's npm root from existing pool args.** Walk `workerSupervisor_->listPools()`, find any pool whose `template.args` contains `node_modules`, and clip the path at that point — capturing the operator's actual install root (the four already-pooled MCPs all carry their full path through the global node_modules tree, so this almost always succeeds).

2. **Fallback roots.** Add `%APPDATA%\npm\node_modules` (resolved via `GetEnvironmentVariableW` per `.claude/rules/10-windows-native-cpp.md`) and `C:\Program Files\nodejs\node_modules` to cover fresh installs that don't have any pool yet.

3. **Test each known canonical package.** For each entry in the same six-package map used by `kKnownInstallCommand` (playwright, chrome-devtools, filesystem, memory, sequential-thinking, sqlite), test `std::filesystem::exists(<root>/<package>)` against every candidate root. Cache yes/no in `detectedPackagesCache_`.

The scan happens once per process under `detectedPackagesMutex_`; `detectedPackagesScanned_` guards re-scans. Snapshot calls are O(1) map lookups thereafter.

### `installPackageDetected` populated in the snapshot

`enrichRuntimeStatDiagnostics` calls `packageDetected(entityId)` on the `awaiting_pool_registration` path when `installCommand` is non-empty, and stores the result in `stat.installPackageDetected`. The `installHint` trailer text branches:

- **Detected:** *"Canonical npm package detected on disk; skip the install and register a pool."*
- **Not detected:** *"Known install command: \`npm install -g <package>\`."* (v0.9.58 behaviour)

### Dashboard card branches the install line

`renderRuntimeStatDiagnostic` in `app.js`:

- **Detected:** *"Already installed: `<package>` <span>DETECTED</span>"* — a green chip pill alongside the package name (the `npm install -g ` prefix is stripped to show only the package).
- **Not detected:** *"Install: `npm install -g <package>`"* — original v0.9.58 line.

The chip uses `.subagent-diag-detected-pill` styled with the same green accent as `.subagent-diag-pill.good`.

### Compatibility with the `getenv` deprecation

The detection uses `GetEnvironmentVariableW(L"APPDATA", ...)` instead of `std::getenv("APPDATA")` — the former is Win32-native (matches the project rule) and avoids the CRT-deprecated `getenv` warning that would otherwise fail the build under `/W4 /WX`.

### Detection on hosts with the package present

If the operator runs `npm install -g @playwright/mcp` and restarts MCOS, the next snapshot will flip `playwright`'s `installPackageDetected` to `true` and the dashboard will show:

> **Already installed:** `@playwright/mcp` [DETECTED]

instead of:

> **Install:** `npm install -g @playwright/mcp`

## Risks and blockers

* **Detection cache is one-shot per process.** A `npm install` performed after MCOS startup is not picked up until the next service restart. Acceptable for v0.9.60; revisit if operators flag it. A `/api/diagnostics/refresh-detection` endpoint would be a small addition.
* **Inferred npm root may miss multi-user setups.** When MCOS runs as Local System, `%APPDATA%` resolves to the service profile, not the operator's. The pool-args inference handles the common case where Flynn pre-registered the four working pools, but a clean install without any pool yet falls back to per-service-account `%APPDATA%`. Documented as a known limitation; revisit if operators report mismatches.
