# Release Completion Report — v0.9.92

Supervisor lifecycle events now dual-emit to the persistent `MasterControl::Diagnostics` module.

## Scope completed

### Persistent diagnostic mirror for supervisor lifecycle

The operator's original directive on the supervisor surface explicitly named the diagnostic module:

> *"If a test fails, then it MUST be reported in the Overview Error Frame and reported to the diagnostic module."*

That mandate was partially satisfied by:
* The Overview Error Reporting card (already harvesting `statusCode >= 400` events from `/api/activity`).
* The in-memory activity ring's spill to `<dataDirectory>/activity.jsonl` (ring-trimmed to ~5000 events).

The gap: supervisor lifecycle events landed in the activity ring but **never in the persistent Diagnostics module** at `include/MasterControl/MasterControlDiagnostics.h`. That module writes append-only per-component `.jsonl` files under `%PUBLIC%\Documents\Master Control Orchestration Server\logs\<component>\events.jsonl` that survive every restart and never trim. The runtime already uses it for the `dashboard-snapshot` telemetry record; supervisor events were missing.

v0.9.92 dual-emits supervisor lifecycle events into the `supervisor` component path:

| Event kind | Severity | When |
|---|---|---|
| `supervisor_select` | `info` (ok) / `error` (rejected) | POST /api/supervisor/assignment/select |
| `supervisor_revoke` | `warn` | POST /api/supervisor/assignment/revoke |
| `supervisor_connect` | `info` | POST /api/supervisor/connect/confirm (success) |
| `supervisor_connect_rejected` | `error` | POST /api/supervisor/connect/confirm (capability / provider / config mismatch) |
| `supervisor_disconnected` | `warn` | supervisor watchdog flipped Connected → Disconnected (120s heartbeat threshold) |

Each event carries structured `data`:

* `providerId`, `assignmentId`, `configId`, `actor`
* Connect events additionally include `clientId` + `capabilities[]`
* Watchdog disconnects include `thresholdSeconds`
* Revoke includes the operator-supplied `reason`

### Live verification

Immediately after v0.9.92 deploy:

```
POST /api/supervisor/assignment/select { providerId: "grok" }
→ Activity ring: supervisor_select (200)
→ <PUBLIC>\Documents\Master Control Orchestration Server\logs\supervisor\events.jsonl gets a fresh row:
   { "capturedAtUtc":"2026-05-11T01:14:52.044Z",
     "component":"supervisor",
     "data":{ "actor":"operator",
              "assignmentId":"SUP-20260511T011452Z-2f67",
              "configId":"CFG-20260511T011452Z-ef95",
              "providerId":"grok" },
     "event":"supervisor_select",
     "message":"Supervisor selected: Grok (assignment SUP-...).",
     "severity":"info" }
```

Persistent log captured the event in the expected JSONL shape. A `LOG-LOCATION.txt` companion file was also written naming the events / telemetry paths — the operator can `cat` that file to find where the supervisor audit lives.
