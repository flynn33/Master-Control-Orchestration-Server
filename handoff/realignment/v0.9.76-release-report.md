## Scope completed

### 1. Bind-address consistency on the WinUI Shell

The dashboard fix in v0.9.74 routed every gateway-URL render through `resolveDisplayUrl` so wildcard binds (`0.0.0.0` / `::`) substituted to the LAN-routable primary IP. The shell already had wildcard substitution for the configuration narrative line, but four display-only cards still rendered `snapshot.bindAddress` raw:

* **Overview**: `ApplyApisAndServicesCard` (the "Bind:" line on the APIs and Services card)
* **Telemetry**: `EnvironmentProfileText` (the "Bind address:" line)
* **Security**: `BindAddressText`
* **Security narrative**: `formatSecurityNarrative` in `ShellFormatting.h`

When the operator configured `bindAddress=0.0.0.0` and `primaryIpAddress=192.168.1.7`, they saw the literal `0.0.0.0` on those four surfaces while every other gateway-URL card showed the resolved `192.168.1.7`. v0.9.76 adds `resolveDisplayBindAddress(bindAddress, lanIp)` and `isWildcardBindAddress(bindAddress)` to `ShellFormatting.h` and routes the four sites through the helper. Each site also emits a trailing `(configured 0.0.0.0)` or `Listening on: 0.0.0.0` line so the operator sees both the configured listen value and the resolved client-facing value.

Editor surfaces — `SettingsSectionControl::PopulateEditorFromSnapshot`, `SettingsSectionControl::UpdateSummary`, and `MainWindow::SetupWizard bindAddressBox.Text` — keep the raw value so operator edits round-trip. Substitution is display-only.

### 2. Telemetry MCP-server / sub-agent cards rendered at compact density

The user reported "MCP Servers on the telemetry page must have smaller cards. They must be close in size to the current Sub-Agent Cards." Both card kinds already shared the same `EndpointStatCardGrid::renderEndpointStatCardGrid` template, so they were already structurally identical — but the absolute size was the operator concern. v0.9.76 adds an optional `compact` template parameter (default `false`) that shrinks:

* card padding from 14 / 12 to 10 / 8
* inner spacing from 6 to 4
* header font (name) from 16 to 13
* spec font from 12 to 11
* utilization label from 20 to 16
* progress bar height from 10 to 8
* util-row column widths (64/80 → 50/64)
* hides the empty-state "No active clients leasing this..." line when there are no clients

`TelemetrySectionControl::PopulateMcpServerCards` and `PopulateSubAgentCards` pass `compact=true`. `RuntimeSectionControl` and the Overview surface continue to use the default wide density for primary inventory.

### 3. WinUI-safe system tray icon with hide-on-close

The v0.9.72 attempt subclassed the WinUI 3 main window's `WndProc` via `SetWindowLongPtrW(GWLP_WNDPROC, ...)`, which crashed the shell on launch with `EXCEPTION_ACCESS_VIOLATION` before the first frame painted (verified by minidump analysis at `%LOCALAPPDATA%\CrashDumps\MasterControlShell.exe.*.dmp`). WinUI 3 reserves the main-window WndProc for the XAML compositor; replacing it leaves the compositor calling into a stale function pointer.

v0.9.76 implements the WinUI-supported pattern:

* `MainWindow::EnsureTrayHostWindow()` registers a class `MCOSTrayHostWindow` and creates an `HWND_MESSAGE`-parented stub window. Its `TrayHostWndProc` owns the Shell_NotifyIcon callback and the right-click popup-menu logic. The XAML compositor's WndProc is never touched.
* `MainWindow::HookAppWindowClosingForTray()` registers a handler on `Microsoft::UI::Windowing::AppWindow::Closing`. When the operator clicks the close button, the handler sets `args.Cancel(true)` and calls `appWindow.Hide()` — the window minimizes to the tray and the running MCOS service is left running.
* `Close MCOS Shell` from the tray right-click menu sets `exitRequested_=true` and posts `WM_CLOSE`; the AppWindow.Closing handler observes the flag and lets the window actually close, removes the tray icon, and destroys the stub window.

Right-click menu commands: **Show MCOS Shell**, **Stop Server** (`net stop MasterControlProgram` via UAC `runas`), **Restart Server** (`net stop && net start`), **Close MCOS Shell**. `InstallTrayIcon` and `HookAppWindowClosingForTray` are now called from `ConfigureWindow` once per session start.

### 4. Supervisor Agent Assignment Wizard end-to-end

Implemented per the Implementation Package at `C:\Users\Flynn\Downloads\MCOS_Supervisor_Agent_Assignment_Wizard_Package`. Strict task-manifest order:

* **TASK-001 / TASK-002 — Backend data model + service.** New header `include/MasterControl/SupervisorAssignment.h` carries `SupervisorProvider` (chatgpt | claude | grok | unknown), `SupervisorMode` (5 values per `SECURITY_AND_CAPABILITY_MODEL.md`), `SupervisorState` (off | config_generated | pending_connection | connected | disconnected | revoked | error), and the request / response payload structs. New service in `src/MasterControlApp/SupervisorAssignmentService.{h,cpp}` (≈600 LOC) owns the assignment lifecycle behind an `ISupervisorAssignmentService` interface. Mutex-guarded; persists the latest assignment to `<dataDirectory>/supervisor-assignment.json` on every mutation; rehydrates on construction so a service restart preserves the operator's selection.
* **TASK-003 — Config generation.** `selectAndIssue()` mints `SUP-{compactUtcStamp}-{hex}` assignment ids and `CFG-{compactUtcStamp}-{hex}` config ids, computes a `sha256:`-prefixed FNV-folded server fingerprint, and synthesizes the `mcos.supervisor.config.v1` JSON document with provider-specific instructions block, server endpoint metadata (mcpEndpoint, discoveryEndpoint, fingerprint), supervisor block (providerId, role=supervisor, mode, exclusive), the 8 autonomous-supervisor capabilities, and `auth.tokenRef` referencing the freshly-issued token id.
* **TASK-004 — API routes.** Six routes added to `MasterControlRuntime::handleHttpRequest` and registered in the route advertisement table:
  * `GET  /api/supervisor/assignment` → current full `SupervisorAssignment`
  * `POST /api/supervisor/assignment/select` → revoke prior + issue new + return result
  * `POST /api/supervisor/assignment/revoke` → revoke active
  * `POST /api/supervisor/config/generate` → shorthand for select-or-regenerate (accepts `providerId` to select or empty body to re-emit current)
  * `POST /api/supervisor/connect/confirm` → server-side capability gate for remote client connect
  * `GET  /api/supervisor/status` → compact status for UI cards
  Every state transition emits an `ActivityEvent` to `globalActivityRing()` with `kind=supervisor_select` / `config_issue` / `config_regenerate` / `revoke` / `connect` / `connect_rejected`.
* **TASK-005 — UI popup and single-selection toggles.** New Supervisor Agent card on `OverviewSectionControl.xaml` (operator memory: WinUI is the priority surface) between Self-Tests and Error Reporting. Three rows with `ToggleSwitch` controls (`SupervisorChatGptToggle`, `SupervisorClaudeToggle`, `SupervisorGrokToggle`); single-selection radio behavior implemented in `SetSupervisorSelection`.
* **TASK-006 — File save flow.** `Generate Config & Save` button calls `GenerateSupervisorConfigAsync(providerId)` which posts `/api/supervisor/config/generate`, opens an `IFileSaveDialog` (Win32 `IID_FileSaveDialog` COM, native Windows save UX) defaulting to `%USERPROFILE%\Documents\MCOS\SupervisorConfigs\` (folder auto-created via `SHCreateDirectoryExW`), and writes the embedded config object as UTF-8 (no BOM). The dialog title is "Save Supervisor Configuration"; the file-type filter is `*.json` then `*.*`; the suggested filename matches the package's required pattern (`mcos-supervisor-{providerId}.config.json`).
* **TASK-007 — Connection confirmation flow.** `confirmConnection(SupervisorConnectionClaim)` validates assignment exists, provider matches, configId matches, expiration not reached, and every requested capability is in the mode's allowed set and not in the forbidden set. On success: state → Connected, clientId saved, connectedAtUtc + lastHeartbeatUtc set. Forbidden tools (`worker.run_shell`, `admin.disable_governance`, `admin.deploy`, `admin.rotate_secrets`, etc.) are rejected with an explicit errorMessage. Provider mismatch and config-id mismatch each produce distinct error messages.
* **TASK-008 — Status display.** `ApplySupervisorCard` populates the headline, status text, and dot tone from `ShellSnapshot.supervisorStatus` on every snapshot tick. Status text mirrors the active provider, mode, and last-heartbeat (Connected) or expiration timestamp (PendingConnection). Generate button is enabled when any provider is selected; Revoke is enabled when the server-side state is active. `Revoke Active` button calls `RevokeSupervisorAsync` which clears the local UI and POSTs `/api/supervisor/assignment/revoke`.
* **TASK-009 — Tests.** 12 backend tests added to `MasterControlOrchestrationServerTests.cpp`:
  1. `testSupervisorProviderRoundTrip` (chatgpt/claude/grok/unknown enum mapping)
  2. `testSupervisorStateRoundTrip` (all 7 states bidirectional)
  3. `testSupervisorDefaultCapabilitiesAreAutonomousScoped` (8 capabilities, no forbidden in allowed)
  4. `testSupervisorSelectAndIssueChatGpt` (full happy-path JSON shape per schema)
  5. `testSupervisorSelectRejectsUnknownProvider`
  6. `testSupervisorReassignmentSupersedes` (new assignment id, prior revoked)
  7. `testSupervisorRevokeClearsActive`
  8. `testSupervisorConfirmConnectionHappyPath`
  9. `testSupervisorConfirmRejectsForbiddenCapability` (`worker.run_shell` denied)
  10. `testSupervisorConfirmRejectsProviderMismatch`
  11. `testSupervisorPersistenceSurvivesServiceRecreation`
  12. `testSupervisorParseSelectRequestRejectsBreakGlass`

Each test uses a unique temp directory under `temp_directory_path() / "mcos-supervisor-{tag}"` so on-disk state from one test never leaks into another.
