# Release Completion Report — v0.9.5

Maintenance release in the same series as v0.9.3 + v0.9.4 (operator-directed defect cleanup). Closes four real bugs surfaced by the v0.9.4 follow-up bug-hunt probe and three deferred items from v0.9.4.

## Scope completed

### Worker crash recovery (Bug A.4, dominant in the v0.9.4 follow-up probe)

Pre-v0.9.5 the supervisor learned about child death only when an outbound `sendStdioJsonRpc` write failed, and even then it just returned the error — the pool's instance listing kept advertising the dead instance as Ready until the next operator-driven scale or remove. The bug-hunt probe surfaced four related symptoms:

| Symptom | Observation |
|---|---|
| Stale instance state after kill | Killed `filesystem#2` (pid 4640); `GET /api/pools/filesystem` reported `instances=1, state=ready` for ~minutes after. |
| `Pool already at or above minInstances` returned for empty pool | After the pool was reactively reduced to 0 instances, `POST /api/pools/filesystem/scale` still claimed it was at min. |
| Operator API stalls during cleanup | `/api/pools/sequential-thinking` timed out twice while the supervisor was in the middle of detecting/handling the death of one of its children. |
| Governance bundle endpoint timeouts | `/api/governance/bundles/windows` and `/macos` timed out (resolved automatically once the dead worker was reaped — symptom of the same root cause). |
| No activity event for death | Activity ring has zero `gateway_lifecycle` / pool events when a worker dies. |

v0.9.5 fixes all five via three coordinated changes:

1. **`reapDeadInstancesLocked()`** — walks every pool's instances, calls `GetExitCodeProcess` on each child handle, reaps any process whose exit code is no longer `STILL_ACTIVE` (closes Job Object + pipe + process handles, drops `children_` entry, erases the instance from `pool.instances`, queues a `DeathLogEntry` for future telemetry consumption).
2. **Auto-respawn pass** at the bottom of `reapDeadInstancesLocked()` — for every pool whose `healthyInstanceCountLocked()` fell below `scalePolicy.minInstances`, spawns replacements until the pool is back at its contracted floor. Operators no longer need to `POST /api/pools/{poolId}/scale` to bring a crashed worker back.
3. **`healthyInstanceCountLocked(pool)`** — counts only `{Starting, Ready, Busy}`. `ensureMinInstances` and `scaleUpOnce` use this instead of `pool.instances.size()`. Pre-v0.9.5 a pool with one Failed instance and zero Ready instances was reported as "at minInstances" because the Failed one counted as one, blocking respawn.

The reap is wired into every operator-API entry that reads or modifies pool state (`listPools`, `findPool`, `ensureMinInstances`, `scaleUpOnce`) so any operator observation triggers cleanup. `sendStdioJsonRpc`'s `WriteFile`-failure path also now atomically tears down the dead child's handles + `children_` entry + erases the instance from `pool.instances` + queues a death event, so the very next listPools call sees correct state without waiting for the next reap pass.

### VERSION.json wired into CMake's depend graph (Deferred from v0.9.4)

```cmake
set_property(DIRECTORY APPEND PROPERTY CMAKE_CONFIGURE_DEPENDS
    "${CMAKE_CURRENT_SOURCE_DIR}/VERSION.json")
```

Bumping `VERSION.json` now triggers a configure step on the next build, which regenerates `MasterControlVersion.h` with the new `MASTERCONTROL_VERSION` literal so the running binary reports the right version through MCP `initialize` + `serverInfo` + the operator dashboard's version chip. v0.9.4 had to touch `CMakeLists.txt` manually to force a reconfigure. Verified at v0.9.5 build time:

```
CMake is re-running because ... VERSION.json is newer than ... generate.stamp.depend
-- Master Control version from VERSION.json: 0.9.5
```

### tools/list catalog cache TTL (Deferred from v0.9.4)

`NativeHttpSysGatewayAdapter` gained `toolCatalogCacheValidUntil_` and a 30-second TTL constant. `refreshToolCatalogLocked` now serves from `toolCatalogCache_` when the TTL hasn't expired, dropping per-LAN-client `tools/list` cost from O(pools) stdio round-trips to a single pointer copy on cache hit.

Measured **33x speedup** on a five-pool deployment:

| Call # | Tool count | Elapsed |
|---|---|---|
| 1 (cold) | 33 | 223 ms |
| 2 (warm) | 33 | 7 ms |
| 3 (warm) | 33 | 7 ms |
| 4 (warm) | 33 | 6 ms |
| 5 (warm) | 33 | 7 ms |

The TTL is short enough that operator-driven pool changes surface within 30s. v0.9.6 candidate: explicit cache invalidation on POST `/api/pools` and per-pool action routes for immediate visibility.

## Public contracts changed

- `GET /api/pools` and `GET /api/pools/{poolId}` now reflect dead-worker reaping immediately rather than on a delay. Operators reading these endpoints during a worker crash see the correct healthy-count, no longer the stale Ready value. (Backward-compatible — the schema is unchanged; only the freshness of `instances[]` improved.)
- `POST /api/pools/{poolId}/scale` semantics: post-reap a pool that had its only worker killed is now correctly reported as `current < minInstances` and respawned. Pre-v0.9.5 the same call returned `Pool already at or above minInstances` and left the pool dark.

No wire-protocol changes. No new endpoints. No deletions.

## Risks and blockers

- **`mcpInitialized` flag is per-`ChildProcess`, reset on respawn.** When a worker crashes and is respawned, the new instance has `mcpInitialized=false` and the next `sendStdioJsonRpc` call performs the lazy initialize handshake again. Correct behavior. Watch-item: if a misbehaving worker crashes repeatedly, every operator API call triggers the 3s handshake deadline; consider a per-instance crash-counter that opens a circuit breaker after N rapid failures.
- **Reap is opportunistic, not proactive.** A pool with no traffic and a dead worker stays dark until the next operator query. v0.9.6 candidate: 2-second background watchdog tick.
- **Cache TTL drift.** Operator pool changes surface within 30s; immediate visibility requires explicit invalidation from `upsertPool` / `removePool` / pool action routes. v0.9.6 candidate.
- **Activity-ring telemetry events for pool death + respawn are recorded but not yet drained.** `deathLog_` / `respawnLog_` accumulate `DeathLogEntry` / `RespawnLogEntry` records; no consumer drains them. v0.9.6 should wire a `TelemetryAggregator` pointer into `WorkerSupervisor`'s constructor and emit `Telemetry::Pool::{Error,Info}` events. Until then, operators learn about deaths via the pool listing rather than the activity stream.
- **DNS-SD self-test SKIPs by default** — mDNS resolution requires Bonjour / mDNSResponder on the host. The advertisement code path is untested by the new script when Bonjour is absent. Watch-item: install Bonjour on a CI runner or add a Win32 DNS-SD listener path.

## Ready for next phase?

Answer: **yes.** v0.9.5 leaves MCOS in a state where:

1. A LAN AI client can connect, complete the MCP handshake, list tools (with cached fast path), call tools end-to-end, and get persisted state back across worker restarts.
2. A worker process crash (any cause: OOM, segfault, operator kill, OS reboot of just that process) is detected on the next operator query and auto-recovered; no operator action is required.
3. Operator dashboards see fresh state on every query (no more "Ready" lies).
4. Version bumps propagate automatically without a manual CMake nudge.
5. Discovery surfaces (well-known / api / beacon / DNS-SD / per-client onboarding) are validated by a single-command self-test.

The next iteration (v0.9.6) should pick up the watchdog thread + telemetry-aggregator wiring + explicit cache invalidation, then start the architecture decision for the MCOS sub-agent runner. The existing live deployment is fully usable as-is; remaining work is incremental polish.
