## Scope completed

Three coordinated fixes, all required to fully solve the problem:

### 1. Prober walks the union of probe targets
The dashboard snapshot doesn't only iterate `inventoryService_->listEndpoints()` — it also synthesizes platform-gateway endpoints inline (lines ~10417 in the snapshot). The v0.9.15 prober walked only the inventory list, so the synthetic gateway endpoints were always cache misses. v0.9.16 expands the prober's walk to the union of inventory + `platformServiceCatalogService_->listGateways()`. Governance servers route through gateway host:port pairs so probing the gateways covers them too.

### 2. Snapshot path is non-blocking on cache miss
`probeReachabilityCached` gained an `allowSynchronousProbe` parameter (default false). Snapshot callers never block: a cache miss returns the last-known-good cached value or `false` if no prior probe. The prober explicitly passes `true` to opt into the synchronous-probe path so it can populate the cache. End result: the dashboard snapshot is always O(map lookup) regardless of TTL state.

### 3. Inventory refresh moved off the snapshot thread
This was the dominant slow path. `AdminApiService::snapshot()` was calling `inventoryService_->refresh()` which does its own serial 400ms-per-endpoint TCP probes inside `RuntimeInventoryService::probeEndpoint`. With ~30 endpoints that's 12 seconds blocking on every dashboard read. The contract comment on `refreshAsync()` literally says it exists "so the admin API doesn't block for 10+ seconds while N endpoints are TCP-probed sequentially" — but it was only used in mutating handlers, not the snapshot. v0.9.16 switches the snapshot to `refreshAsync()`. The probe still happens, just on a detached background thread.

## Risks and blockers

- **First call after fresh service start may show partial reachability data** because the prober's first sweep takes 6-15s and the snapshot returns false-or-stale during that window. Operator dashboard cards may briefly show "unreachable" for endpoints that are actually online; the next refresh after the prober completes its first sweep (10-20s after boot) shows the true state. Acceptable trade-off vs the alternative (every dashboard call blocks for 12s).
- **`refreshAsync()` is fire-and-forget** — the snapshot doesn't wait for the inventory refresh to complete, so consecutive snapshots taken in fast succession may read very slightly stale inventory state. In practice the inventory refreshes itself every few seconds; the staleness window is sub-second.
