# Release Completion Report — v0.9.13

Maintenance release. Closes the activity-ring loses-on-restart issue documented in v0.9.10's risk list.

## Scope completed

### Persistent activity ring

Pre-v0.9.13 the global `ActivityEventRing` was in-memory only. A wedge / crash / SCM force-stop lost all forensic data, including the v0.9.10 `supervisor_shutdown_wedge` events that were specifically designed for post-mortem analysis. Operators reading `/api/activity` after a restart saw an empty audit trail.

v0.9.13 persists every append to `AppPaths::dataDirectory/activity.jsonl` (resolved at runtime — observed at `C:\ProgramData\MasterControlOrchestrationServer\activity.jsonl` on the test machine). Append-only JSON-lines. On runtime construction the ring loads up to `kPersistedTailMax = kCapacity = 512` most-recent events from the file before the in-memory ring starts accepting new appends; `nextSequence_` is bumped to the highest persisted id so new events get fresh ids that don't collide with the loaded ones.

Bounded on-disk size: when the file crosses `kFileBytesSoftMax = 8 MB` the ring rewrites the file from its in-memory contents (truncating older history). 8 MB ~= weeks of typical traffic.

Persistence is best-effort: I/O failures (corrupt file, permission issues, disk full) are swallowed inside the locked path so they cannot disrupt the in-memory append. Worst case is the in-memory ring continues working with no disk record, which is the pre-v0.9.13 baseline behavior.

Wired into `MasterControlApplication::Impl::initialize` as the **first** line so the ring is persistence-armed before any subsequent service constructs (gateway boot, configuration load, pool registration). Pre-v0.9.13 those early appends would have been lost; now they're persisted from the very first event.

**Verified live** with a stop/start cycle:

| Phase | Activity events visible |
|---|---|
| Before `Stop-Service` | 10 |
| After `Stop-Service` + `Start-Service` | 21 |
| → events from prior run survived; new run added boot/scale events on top |

The file at `C:\ProgramData\MasterControlOrchestrationServer\activity.jsonl` was 2,844 bytes after the test cycle.

## Risks and blockers

- **No persistence-health surface yet.** If disk fills or permissions break, `appendToDiskLocked` silently catches the exception. Operators have no signal that persistence is failing. Watch-item; v0.9.14 candidate to surface a `persistedToDisk: bool` field on each event response or a `/api/activity/health` endpoint.
- **Loaded events lack a "this was loaded, not produced this run" marker.** If operators want to filter a post-mortem read by source, that's a v0.9.14 candidate.
