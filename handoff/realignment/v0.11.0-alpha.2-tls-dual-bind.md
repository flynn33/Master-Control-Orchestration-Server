# Release Completion Report — v0.11.0-alpha.2 — LAN-trust TLS dual-bind

Second internal alpha cut of the v0.11.0 line. Adds optional **HTTPS support on the MCP gateway** so strict clients (ChatGPT connector-edge, Claude.ai web, security-conscious browser extensions) that refuse plain HTTP can connect. No app-layer auth — transport TLS is a separate concern from authentication; LAN-trust posture intact.

## 1. Operator-stated trigger

> "Other models, connectors and apps are refusing to connect because they cannot connect via `https://`. That capability needs to be added."

Pre-`v0.11.0-alpha.2` the runtime served only `http://+:8080/mcp`. Many modern MCP clients enforce a no-plain-HTTP policy at the transport layer (browser extensions, ChatGPT's connector-edge, Claude.ai's MCP surface). For them MCOS was unreachable on the trusted LAN even though the network itself was fine.

## 2. Scope

| Surface | TLS in alpha.2 | Why |
|---|---|---|
| **MCP Gateway** (port 8080 today) | ✅ **Dual-bind: HTTP stays on 8080 + new HTTPS on 8443** | This is where strict AI clients refuse plain HTTP — the actual blocker. HTTP.sys natively supports dual-prefix registration. |
| **Admin HTTP listener** (port 7300) | ❌ HTTP-only this iteration | Uses Winsock `SimpleHttpServer`, not HTTP.sys. Adding TLS there requires an SChannel handshake rewrite — out of scope for alpha.2. Admin is operator-facing from the trusted host. |
| **UDP discovery beacon** (port 7301) | ❌ Unsigned JSON | LAN trust posture treats this as acceptable. Future iteration may sign the payload. |

## 3. Three operator-confirmed design choices

Recorded for traceability (each was the recommended option in the scoping question):

1. **HTTP + HTTPS dual-bind.** HTTPS additive on new port. HTTP keeps working for in-LAN curl / dashboard / Shell tooling that already points at `http://`. Fully reversible (`tlsEnabled=false`).
2. **Self-signed cert per host.** `scripts\Configure-LocalServerCert.ps1` generates the cert with DNS + IP SANs. Each LAN client trusts the exported `.cer` once. Upgrade paths to a local CA documented for the next iteration.
3. **Port 8443 for the MCP gateway HTTPS prefix.** Standard convention (HTTP+443 mapping). The scoping question also discussed a future 7443 admin HTTPS port; **alpha.2 does not add an admin TLS schema field** — admin port 7443 will be introduced when the Winsock-based `SimpleHttpServer` is rewritten to use HTTP.sys / SChannel. The only new TLS port field in this PR is `McpGatewayConfiguration.tlsListenPort`.

## 5. Public contracts changed

| Change | Impact |
|---|---|
| `McpGatewayConfiguration` JSON schema | 3 additive optional fields (`tlsEnabled`, `tlsListenPort`, `tlsCertThumbprint`). Older configs deserialize cleanly via `NLOHMANN_DEFINE_TYPE_NON_INTRUSIVE_WITH_DEFAULT`. |

No removed contracts. No status-code regression. No removed runtime field.

## 6. Operator quickstart — enable HTTPS

```powershell
# 1. From an elevated PowerShell on the MCOS host:
cd "C:\Program Files\Master Control Orchestration Server"
# Or whichever path holds the scripts/ folder you cloned/extracted.

# 2. Generate cert + bind + open firewall + restart service:
scripts\Configure-LocalServerCert.ps1 -RestartService

# Script prints the JSON snippet you need:
#   "tlsEnabled": true,
#   "tlsListenPort": 8443,
#   "tlsCertThumbprint": "<SHA1 from the cert>"

# 3. Open mcos.config.json (under %PUBLIC%\Documents\Master Control Orchestration Server\config\)
#    and merge that block into cfg.mcpGateway. Restart the service if you skipped -RestartService.

# 4. Verify the gateway dual-binds:
Invoke-WebRequest -UseBasicParsing -SkipCertificateCheck https://127.0.0.1:8443/health
Invoke-WebRequest -UseBasicParsing -SkipCertificateCheck https://192.168.1.7:8443/mcp -Method POST `
    -Body '{"jsonrpc":"2.0","id":1,"method":"tools/list"}' -ContentType application/json

# 5. Distribute the public-key .cer to LAN clients so they trust the cert:
#    %PUBLIC%\Documents\Master Control Orchestration Server\certs\mcos-server-public.cer
#
# Windows clients: certmgr.msc -> Trusted Root Certification Authorities -> Import.
# macOS clients:   security add-trusted-cert -d -r trustRoot -k <keychain> mcos-server-public.cer
# Linux clients:   sudo cp mcos-server-public.cer /usr/local/share/ca-certificates/ && sudo update-ca-trust

# 6. Strict clients (ChatGPT connector-edge, Claude.ai, etc.) now use the HTTPS URL emitted by
#    GET http://<host>:7300/api/onboarding/<client-type>.
```

Rollback: `scripts\Remove-LocalServerCert.ps1 -RestartService`.

## 8. Risks and blockers

- **Self-signed cert trust friction.** Every LAN client must add the exported `.cer` to its trust store once. Acceptable for an internal alpha; a local-CA path will land in a follow-up so adding a new MCOS host doesn't re-trigger the per-client trust step.
- **Admin listener (port 7300) is still plain HTTP.** Operators reaching the dashboard / Shell from a remote LAN machine over `http://host:7300/api/...` will not see TLS until the SChannel refactor lands. Practical impact bounded because the Shell + browser dashboard normally run on the host itself.
- **Bind failures are not loud.** If the operator sets `tlsEnabled=true` without running `Configure-LocalServerCert.ps1`, the runtime records the failure in `/api/gateway/status` `message` and `/api/health/summary` carries the same string — but it does **not** flip to a Failed state. HTTP fallback keeps serving. Surface the message in the Shell's gateway card for a sharper signal (follow-up).
