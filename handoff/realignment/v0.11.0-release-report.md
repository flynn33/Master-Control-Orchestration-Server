# Release Completion Report — v0.11.0 — PHASE-14 Slice A

Closes **item 7** from the operator's 8-item testing-phase audit. PHASE-14 (Comprehensive Diagnostics Module) is delivered as a slice plan per the phase's existing structure; Slice A — the HTTP route surface — ships in v0.11.0. Slices B–E land in subsequent iterations.

After v0.11.0, items 1–7 from the operator's audit are all closed. Item 8 (LAN-trust application-layer auth) remains indefinitely deferred to the retail build per explicit operator directive.

## 1. Scope completed (Slice A)

Five operator-visible HTTP routes added inline to `MasterControlRuntime.cpp::handleHttpRequest`. All five aggregate against the existing per-component `events.jsonl` persistent logs (runtime + supervisor + installer) plus their rotated `.1.jsonl` siblings (v0.10.21 rotation).

| Route | Method | Purpose |
|---|---|---|
| `/api/diagnostics/events` | GET | Query the merged event stream with `?max=` / `?severity=` / `?source=` filters. Sorted recent-first. |
| `/api/diagnostics/summary` | GET | `totalEvents` + `bySeverity` map + `byComponent` map + `latest5` array + `generatedAtUtc`. |
| `/api/diagnostics/self-test` | GET | Latest in-memory `SelfTestSnapshot` (passed/failed counts + per-test rows). |
| `/api/diagnostics/export?format=markdown\|json` | GET | Markdown export with structured sections per severity (capped at 50 events/section with a truncation note) **or** JSON dump of the full event set. Content-type matches the format. |
| `/api/diagnostics/clear` | POST | Returns 501 with a clear message: clear-with-retention lands when the SqliteDiagnosticsStore arrives in Slice E. |

## 2. Findings → resolution

**Pre-v0.11.0 state:** Operators had no programmatic way to query, search, or export the persisted events. The persistent log files live at `<PUBLIC>\Documents\Master Control Orchestration Server\logs\<component>\events.jsonl` (plus `.1.jsonl` after the v0.10.21 rotation), and the only access path was reading them from disk by hand. The PHASE-14 plan called for a centralised Diagnostics Service surfaced via HTTP + MCP + WinUI Shell + browser dashboard.

**v0.11.0 delivers:** the HTTP-route portion of that plan as Slice A. The aggregation is done inline against the existing jsonl files (no new SqliteDiagnosticsStore yet — that's Slice E). This unlocks operator queries, exports, and MCP integration (Slice B) without adding a new database dependency.

## 6. Slice B–E plan (deferred to subsequent v0.11.x iterations)

- **Slice B — `mcos-bridge` MCP plugin tools.** Six tools in `.claude-plugin/mcos-control/mcp-servers/mcos-bridge/server.py`: `mcos_diagnostics_summary`, `_events`, `_self_test`, `_export_markdown`, `_export_json`, `_clear`. Each maps to the corresponding HTTP endpoint. Pattern matches existing tools.
- **Slice C — WinUI Shell `DiagnosticsSectionControl`.** Three new files in `src/MasterControlShell/`: XAML + header + .cpp. Severity filter ComboBox, source filter ComboBox, message search TextBox, results ListView (severity-color-coded), Export Markdown / Export JSON / Refresh / Clear buttons, status TextBlock. The Export buttons construct `Windows::Storage::Pickers::FileSavePicker` with operator-suggested filename `mcos-diagnostics-<timestamp>.{md,json}`.
- **Slice D — Browser dashboard Diagnostics tab.** Add a `Diagnostics` tab button + content section to `resources/web/index.html`, JS handlers in `app.js`, styles in `styles.css`. Browser Export = `fetch` URL → `Blob` → programmatic `<a download>` click (browsers can't open the native file picker; this is the standard browser-export pattern, acceptable per the PHASE-14 plan's risk table).
- **Slice E — `SqliteDiagnosticsStore` + tests + completion report.** This is where the persistent store lands. Will replace the inline `aggregateEvents` walks with a SQLite query. Will enable `POST /api/diagnostics/clear` (currently 501). Tests for store CRUD + query filters + ring eviction + export rendering + HTTP route envelopes.

Each slice is independently shippable. Slice A alone closes the HTTP-API gap that was the operator's primary item-7 ask.

## 7. Risks and blockers

- **In-memory ring not yet implemented.** Slice A reads from disk on every query. Acceptable for current event volumes (46 events at audit time, growing slowly except during pool churn). At higher volumes, Slice E's persistent store will add an in-memory ring for the recent-N path.
- **No retention pass.** Without a persistent store, retention is whatever the per-component v0.10.21 rotation provides (50 MB live + 50 MB rotated per component). Slice E's `SqliteDiagnosticsStore` will add a `retain_last_N_boots` pass for self-test snapshots.
- **Disk read on every request.** The aggregation walks ~3 files (runtime + supervisor + installer events.jsonl) per query. For small files this is microsecond-class; for the rotated files (~50 MB each) it's a few milliseconds. Acceptable for the operator-query path. The dashboard's polling can be tuned in Slice E.
