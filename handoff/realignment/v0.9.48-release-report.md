# Release Completion Report — v0.9.48

**CRITICAL fix.** Maintenance release. Invalid UTF-8 bytes in any `/mcp` request body crashed the entire MCOS service.

## Scope completed

### Invalid UTF-8 in `/mcp` request body no longer crashes the service

The v0.9.48 bug-hunt sent a stray Latin-1 `0xE9` byte (the `é` char in CP1252 — extremely common from non-UTF-8 terminals or older Windows clients) embedded in a `tools/list` `params` field and watched `MasterControlServiceHost.exe` exit with code **1067 (ERROR_PROCESS_ABORTED)**. Every probe with invalid UTF-8 reproduced. Reproducibly fatal.

**Root cause path**:

```
client sends body:        {"jsonrpc":"2.0","method":"tools/list","params":{"x":"<0xE9>"}}
                                                                                ^ invalid UTF-8

serveLoop                 receives byte-perfect copy via HttpReceiveRequestEntityBody
  └─ handleMcpRequest
      └─ json::parse(body)               throws parse_error.101 ("ill-formed UTF-8 byte")
          └─ catch (std::exception& ex)  builds error message including ex.what()
              └─ buildJsonRpcError(-32700, "Parse error: " + ex.what(), id)
                  └─ envelope.dump()     **THROWS type_error.316** because the
                                         message string itself contains the invalid byte
                                         from ex.what()'s "last read: '<0xE9>'" tail
                  └─ throw escapes buildJsonRpcError
              └─ throw escapes handleMcpRequest
          └─ throw escapes serveLoop                                    
      └─ serveLoop is the entry of an std::thread
          └─ uncaught exception in thread function -> std::terminate -> service dies
```

The `nlohmann::json::dump()` default error handler is `error_handler_t::strict` which throws on any invalid UTF-8 in any string field. The error-response envelope being constructed *contained* an invalid UTF-8 byte (in the `message` field, copied from the parser's diagnostic tail), so dump() threw the moment it tried to serialize it. The throw had no handler and took the entire service down.

Operator-visible: any LAN client that sent a request with mojibake / Latin-1 / CP1252 / accidentally-malformed-UTF-8 bytes could crash MCOS at will. This is a remote DoS on a trusted-LAN port — not exploitable for code execution, but trivial to trigger and immediate in effect.

**Two-layer fix**:

1. **`buildJsonRpcError` uses `error_handler_t::replace`**. Invalid UTF-8 bytes in the envelope's strings become U+FFFD (`�`) in the output — the standard "lenient JSON serializer" behavior. A try/catch around `dump()` falls back to a hand-rolled minimal valid JSON-RPC error envelope (`-32603 "Internal serialization error."`) if any future nlohmann version throws on something else.

2. **`serveLoop` wraps `handleMcpRequest` in try/catch at the dispatcher boundary**. Any uncaught exception out of the dispatcher now produces HTTP 500 with a `-32603` JSON-RPC error envelope and the gateway thread keeps running. Defense-in-depth — even if a future code path throws past every other safety net, the service stays up.

**Verified live** (bug-hunt's reproducer + several malformed variants + load test):

```
1. Latin-1 0xE9 byte in tools/list params:
   POST /mcp body:'{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{"x":"\xe9"}}'
   -> 200 OK
   {"error":{"code":-32700,"message":"Parse error: ...ill-formed UTF-8 byte; last read: '\"\\ufffd\"'"},"id":null,"jsonrpc":"2.0"}
   service state: RUNNING

2. Latin-1 0xE9 in tools/call args:
   -> 200 OK with parse-error envelope; service RUNNING.

3. Valid UTF-8 (0xC3 0xA9 = é) in tools/call:
   -> 200 OK
   {"id":1,"jsonrpc":"2.0","result":{"content":[{"text":"é","type":"text"}]}}

4. 5× malformed-UTF-8 requests in a row:
   all 200; service RUNNING after; tools/list still returns 33 tools (18991 bytes).
```
