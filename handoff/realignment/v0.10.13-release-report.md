# Release Completion Report — v0.10.13

Server-side reachability self-check + Verify Endpoints button. Driven by a second ChatGPT supervisor-candidate connection-test report (`MCOS_Supervisor_Connection_Test_Report_Attempt_2`) that returned `Connection refused` on all seven probed endpoints.

## 1. What Attempt 2 reported

The cloud ChatGPT runtime tested the new v0.10.8/v0.10.12-style supervisor config and got connection-refused on every URL:

| URL | Method | Result |
|---|---|---|
| `http://192.168.1.7:7300/api/health` | GET | `URLError: <urlopen error [Errno 111] Connection refused>` (19.7 ms) |
| `http://192.168.1.7:7300/.well-known/mcos.json` | GET | Errno 111 (0.8 ms) |
| `http://192.168.1.7:7300/api/supervisor/status` | GET | Errno 111 (0.6 ms) |
| `http://192.168.1.7:8080/mcp` | GET | Errno 111 (0.4 ms) |
| `http://192.168.1.7:8080/api/health` | GET | Errno 111 (0.5 ms) |
| `http://192.168.1.7:8080/.well-known/mcos.json` | GET | Errno 111 (0.4 ms) |
| `http://192.168.1.7:7300/api/supervisor/connect/confirm` | POST | Errno 111 (1.5 ms) |

The report's headline hypothesis: "MCOS may still not be bound to the LAN interface" or "HTTP.sys URL ACL may still be localhost-scoped."

## 2. Why that hypothesis is wrong

Two diagnostic signatures in the report contradict it:

1. **POSIX errno 111** = `ECONNREFUSED` (Linux). Windows would return `WSAECONNREFUSED` (10061) wrapped in `OSError` / `WinError`, not `urlopen error [Errno 111]`. The probe ran from a Linux box -- the ChatGPT cloud runtime, not a same-LAN client.
2. **Sub-millisecond response time** on every probe after the first 19.7 ms attempt. A real TCP RST coming back from a refused-connection across the internet would be ~1 RTT (10-50 ms minimum on a healthy link). Sub-ms refusals are the local kernel saying "I have no route to this host" without putting a packet on the wire. The cloud runtime's network namespace cannot route to RFC-1918 destinations.

Verified locally on the MCOS host:

| Probe | Result |
|---|---|
| `netstat -ano \| findstr :7300` | `TCP 0.0.0.0:7300 LISTENING` (admin) |
| `netstat -ano \| findstr :8080` | `TCP 0.0.0.0:8080 LISTENING` (gateway) |
| `GET http://192.168.1.7:7300/api/health` | **HTTP 200** |
| `GET http://192.168.1.7:7300/.well-known/mcos.json` | **HTTP 200** |
| `GET http://192.168.1.7:7300/api/supervisor/status` | **HTTP 200** |
| `GET http://192.168.1.7:8080/mcp` | **HTTP 405** (Method Not Allowed -- gateway requires POST; listener alive) |
| `GET http://192.168.1.7:8080/api/health` | **HTTP 404** (route lives on admin port; listener alive) |
| `GET http://192.168.1.7:8080/.well-known/mcos.json` | **HTTP 404** (same; listener alive) |

MCOS is healthy. The cloud runtime cannot route to the LAN IP. This is structurally identical to the Attempt 1 finding documented in v0.10.8.

## 3. What v0.10.13 adds

A definitive server-side reachability surface so the operator never has to argue against a remote-network failure mode again.

### New route

`GET /api/supervisor/reachability-check`

Probes MCOS's own URLs from inside the runtime via the existing `sendJsonRequest` WinHTTP helper and returns:

```json
{
  "ok": true,
  "allReachable": true,
  "resolvedLanIp": "192.168.1.7",
  "adminPort": 7300,
  "gatewayPort": 8080,
  "generatedAtUtc": "...",
  "interpretation": "...",
  "probes": [
    { "url": "...", "method": "GET", "ok": true, "statusCode": 200,
      "errorMessage": "", "interpretation": "..." },
    ...
  ]
}
```

The interpretation block carries operator-facing language explaining that "all reachable" means MCOS is healthy, and a remote client reporting connection-refused is "either on a network that cannot route to RFC-1918 private addresses (cloud-hosted ChatGPT runtime is the canonical case) or being blocked by a perimeter firewall outside this host. Use a LAN tunnel (Tailscale, ngrok, Cloudflare Tunnel) to publish the MCOS endpoint to clients that cannot reach the LAN IP directly."

### Probe set

Six probes total. Two are synthetic (admin port self-probes -- the single-threaded SimpleHttpServer deadlocks if it WinHTTPs back to itself from inside a route handler, so we surface the tautological "this response is being served from the admin port, so the listener is alive" instead). Four are real outbound probes against the gateway port (a different HTTP.sys listener, no deadlock risk):

| Probe | Expected | Why |
|---|---|---|
| `GET <loop>:7300/api/supervisor/reachability-check` (synthetic) | 200 | Tautological |
| `GET <lan>:7300/api/supervisor/reachability-check` (synthetic) | 200 | Tautological |
| `GET <loop>:8080/mcp` | 405 | /mcp requires POST; 405 proves listener alive |
| `GET <lan>:8080/mcp` | 405 | Same on LAN IP |
| `GET <loop>:8080/health` | 200 | Gateway health endpoint |
| `GET <lan>:8080/health` | 200 | Same on LAN IP |

Live verification on this host:

```text
allReachable=True
resolvedLanIp=192.168.1.7
adminPort=7300, gatewayPort=8080

OK  HTTP 200 GET http://127.0.0.1:7300/api/supervisor/reachability-check
OK  HTTP 200 GET http://192.168.1.7:7300/api/supervisor/reachability-check
OK  HTTP 405 GET http://127.0.0.1:8080/mcp
OK  HTTP 405 GET http://192.168.1.7:8080/mcp
OK  HTTP 200 GET http://127.0.0.1:8080/health
OK  HTTP 200 GET http://192.168.1.7:8080/health
```

### New Shell UI

`Verify Endpoints` button on the Overview Supervisor card, next to `Generate Config & Save` and `Revoke Active`. Click it and the runtime renders the full per-probe roster in a Consolas-font TextBlock with a copy-paste-friendly summary including resolved LAN IP, admin/gateway ports, and the interpretation block. Operators can paste that verbatim into bug reports or share it with a supervisor-candidate operator so the "MCOS is alive on the LAN" vs "your network can't route to it" distinction is unambiguous.

## 7. Risks and blockers

* **The reachability self-check cannot detect a remote-network routing problem.** That's by definition outside the host. Operators with cloud supervisor clients still need a LAN tunnel; this release just gives them definitive server-side proof so they know where to point the fix.
* **Admin port self-probes remain synthetic.** If the admin server ever moves to a thread-pool model, we can re-enable the real outbound admin probes. For now the synthetic entries explicitly document why ("this very HTTP 200 response is being served on the admin port, so the listener is alive") so the operator isn't surprised.

## 8. Operator answer to Attempt 2

The cloud ChatGPT runtime's connection-refused result is its own Linux network stack rejecting the route to RFC-1918 -- `errno 111` and sub-millisecond response times are the signature, and no Windows MCOS instance returns those error codes. MCOS is healthy: ports 7300 + 8080 are both `0.0.0.0`-bound, all admin endpoints return HTTP 200 on the LAN IP, and the gateway port returns HTTP 405/404 on non-/mcp paths (which proves the listener is alive). v0.10.13 gives you a definitive surface for proving this: click **Verify Endpoints** on the Supervisor card or `curl http://192.168.1.7:7300/api/supervisor/reachability-check`. For the cloud ChatGPT client to actually reach MCOS, expose the LAN IP through a tunnel (Tailscale, ngrok, Cloudflare Tunnel) and stamp the tunnel URL into the supervisor config in place of `192.168.1.7`.
