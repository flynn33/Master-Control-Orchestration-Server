# Release Completion Report — v0.9.31

Maintenance release. Real operator-visible bug surfaced by the v0.9.31 bug-hunt.

## Scope completed

### `/api/activity` now honors `?max=N`

The v0.9.31 bug-hunt probed `/api/activity?max=5` expecting a small "recent 5 events" payload and got back **128 986 bytes / 455 events** — the entire ring buffer. Pre-v0.9.31 the route handler parsed only `?since=N` and silently dropped any other query parameter:

```cpp
if (request.method == "GET" && request.path.rfind("/api/activity", 0) == 0) {
    std::string sinceId;
    /* parse only since= */
    const auto snap = globalActivityRing().read(sinceId);  // no maxCount
    /* return whole ring ... */
}
```

The bug is asymmetric with the structurally adjacent `/api/telemetry/events` route (just a few lines below in the same handler), which has honored `max=N` correctly since v0.6.8 and uses the exact same query-parsing pattern. The `ActivityEventRing::read(sinceId, maxCount)` helper has supported the limit parameter since the original PHASE-09 ring; the route just wasn't wiring it through.

Operator impact: a dashboard polling `/api/activity?max=5` to render a small "recent activity" widget pulled the whole ring on every poll. With the activity ring at v0.9.13's `kCapacity = 512`, that's ~128 KB per poll, every 5 seconds, per dashboard instance — wasted bandwidth on the LAN trust boundary, wasted CPU serializing JSON, and a slower first-paint for the widget.

The fix parses `max=N`, validates it (positive integer ≤ `kCapacity`; bad/missing values fall through to the default), and passes it to `read()`. Default behavior (no `max=`) is unchanged — clients that intentionally want the whole ring (forensic poll after a wedge, log archival job) keep working with no migration.

**Verified live** (v0.9.31 deployment, ring at 455 events):

```
max=1        ->     335 bytes,   1 event
max=5        ->   1 486 bytes,   5 events     (was 128 986 / 455 — 87× smaller)
max=20       ->   5 791 bytes,  20 events
max=100      ->  27 375 bytes, 100 events
since=450&max=3 -> 906 bytes,   3 events     (combined filtering works)

(default)    -> 131 041 bytes, 455 events    (backward compat preserved)
max=garbage  -> 131 041 bytes, 455 events    (bad max= falls through to default)
max=0        -> 131 041 bytes, 455 events    (zero falls through)
```
