# Release Completion Report — v0.9.9

Maintenance release. Closes both items deferred from v0.9.8.

## Scope completed

### 1. Bounded watchdog shutdown (fixes v0.9.8 deploy wedge)

The v0.9.8 deploy reproduced a real production-blocking bug: SCM showed the service in `StopPending` for 60+ seconds while trying to stop the v0.9.7 binary, and had to be force-killed before the v0.9.8 binary could be installed. The pre-v0.9.9 destructor:

```cpp
~WorkerSupervisor() override {
    watchdogRunning_ = false;
    if (watchdogThread_.joinable()) {
        watchdogThread_.join();   // <-- BLOCKS INDEFINITELY if watchdog wedged
    }
    (void)shutdownAll();
}
```

`std::thread::join()` blocks until the thread exits. If the watchdog is mid-reap and `CreateProcessW` is hanging (or any other path takes longer than the watchdog's natural exit cycle), join() never returns. SCM eventually escalates to a forced terminate, but operators see a stuck StopPending state and may force-kill the binary, which is what happened during v0.9.8 deploy.

v0.9.9 bounds the shutdown:

```cpp
~WorkerSupervisor() override {
    watchdogRunning_ = false;
    if (watchdogThread_.joinable()) {
        std::atomic<bool> joinComplete{false};
        std::thread joiner([this, &joinComplete]() {
            if (watchdogThread_.joinable()) watchdogThread_.join();
            joinComplete.store(true, std::memory_order_release);
        });
        const auto deadline = std::chrono::steady_clock::now()
                            + std::chrono::milliseconds(kShutdownJoinTimeoutMs);
        while (std::chrono::steady_clock::now() < deadline
               && !joinComplete.load(std::memory_order_acquire)) {
            std::this_thread::sleep_for(std::chrono::milliseconds(50));
        }
        if (joinComplete.load(std::memory_order_acquire)) {
            joiner.join();           // clean path
        } else {
            watchdogThread_.detach(); // wedged path -- accept thread leak; process is exiting
            joiner.detach();
        }
    }
    (void)shutdownAll();
}
```

`kShutdownJoinTimeoutMs = 5000` is generous enough to absorb a single full reap cycle (~2.1s worst case = 2s sleep + ~100ms reap) plus the drain pass, while staying well below SCM's typical patience threshold.

**Verified live:** `Stop-Service -Name MasterControlProgram` measured **5.4 seconds** on the v0.9.9 binary — at the join deadline + a small buffer. The pre-v0.9.9 binary in the same probe took 60+ seconds and had to be force-killed.

The fix is **safety-critical for in-place upgrades** because SCM cannot install a new binary while the old one holds the file open. A 5-second bounded shutdown means upgrade attempts always succeed within ~10 seconds.

### 2. Dedicated `QuarantineLogEntry` struct (cleaner than the v0.9.8 ad-hoc encoding)

v0.9.8 packed quarantine events into `DeathLogEntry` with a sentinel: empty `instanceId` + `previousState == "QUARANTINED"` + `exitCode = failure-count-in-window`. That worked but conflated two distinct events in one struct.

v0.9.9 splits them:

```cpp
struct QuarantineLogEntry {
    std::string poolId;
    int failureCountInWindow = 0;
    int windowSeconds = 0;
    int cooldownSeconds = 0;
};
std::vector<QuarantineLogEntry> quarantineLog_;
```

The reap path pushes into `quarantineLog_` instead of conflating with `deathLog_`. `drainAndEmitSupervisorEvents` has a dedicated quarantine emit loop. The consumer message reads cleanly:

```
Pool 'sequential-thinking' QUARANTINED -- 5 worker crashes within 60s window;
auto-respawn paused for 60s. Operator can force respawn via
POST /api/pools/sequential-thinking/scale.
```

**Verified live:** kill storm (6 rapid kills) → after the 5th the pool is QUARANTINED; the message above appeared in BOTH `/api/activity` (as `kind=pool_quarantine`) and `/api/telemetry/events` (as `severity=error`). Operator-visible message shape is unchanged from v0.9.8 — only the internal encoding got cleaner.

## Risks and blockers

- **Watchdog wedge ROOT CAUSE not pinned.** v0.9.9 makes shutdown bounded, but doesn't diagnose WHY the wedge happened. Hypothesis: `CreateProcessW` for an npx-launched node taking longer than the SCM patience threshold while the watchdog held mutex_, then the destructor's old `join()` waited for that to complete. Worth a v0.9.10 follow-up: log the watchdog state on detach so we can spot the wedge pattern in production.
- **Detached watchdog leaks until process exit.** Acceptable since detach only fires when the process is shutting down anyway, but it does mean the OS handle table briefly carries the leaked thread. Don't apply this pattern to long-lived contexts.
