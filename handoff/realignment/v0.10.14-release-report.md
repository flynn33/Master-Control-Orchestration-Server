# Release Completion Report — v0.10.14

Remediation of `AUDIT-FINDINGS-2026-05-11` (Grok-authored repo audit against the then-current v0.10.11 source tree). The audit confirmed the core runtime, contracts, and architecture are sound and identified three classes of surface-level defects: hardcoded developer paths, retired-gateway documentation, and a fragile test build. v0.10.14 ships fixes for all three plus one sentinel-flagged downstream drift.

Audit also stated the C++ hardcoded path was in `MasterControlApp/MasterControlRuntime.cpp`; verification revealed the actual offending file was `MasterControlBaselineToolsWorker/main.cpp` line 2814, in the `scribe.list_release_reports` MCP tool. Both files were grepped; only the worker has the literal. **Fix:** derives the handoff directory via `GetModuleFileNameW(nullptr)`, probes `<exe-dir>\handoff\realignment` and `<exe-dir>\..\handoff\realignment`, returns a structured "not available" JSON when neither resolves. Works on any install location.

`.claude/mcp-state/register-pools.ps1` line 28 carried the same anti-pattern (`$projectRoot = "G:\Claude\..."`); replaced with `$projectRoot = (Resolve-Path (Join-Path $PSScriptRoot '..\..')).Path` so the script derives its own root.

## 2. Sub-agent dispatch (per audit's recommended workflow)

Two sub-agents ran in parallel:

* **`build-resolver`** — fixed `.mcp.json`, `register-pools.ps1`, `tests/CMakeLists.txt` + test source, and the C++ hardcoded path. Ran the full validation chain itself (cmake build 0 errors / 0 warnings, ctest 4/4 pass).
* **`documenter`** — retired the in-process HTTP.sys adapter references in the four plugin files. Verified with a clean grep (zero matches for `the in-process HTTP.sys adapter|the in-process HTTP.sys adapter|the in-process HTTP.sys adapter` across `.claude-plugin/mcos-control/`).

After both returned, **`sentinel`** ran a hard-rule + audit-scope compliance review. Flagged one additional drift outside the audit's stated scope: `.claude/rules/30-mcp-gateway-discovery.md` still named the in-process HTTP.sys adapter as the "preferred first gateway" and showed a `NativeHttpSysGatewayAdapter` topology. Fixed inline: the rule now reads "Current implementation (v0.9.0+): Native Win32 HTTP.sys listener... wrap the listener behind `IMcpGateway` / `NativeHttpSysGatewayAdapter`" with a historical-note paragraph documenting why the substrate changed.
