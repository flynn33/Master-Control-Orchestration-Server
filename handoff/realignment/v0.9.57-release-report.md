# Release Completion Report — v0.9.57

Dashboard JS surface for the v0.9.56 honest-unavailable diagnostic block. The data has been flowing through `/api/dashboard` since v0.9.56; v0.9.57 wires it into the browser cards so operators see the diagnostic on launch instead of having to hit `/api/diagnostics/runtime-stats` separately.

## Scope completed

### Per-card diagnostic block on every sub-agent + MCP-server card

`resources/web/app.js` `renderSubAgentCard` now appends a compact diagnostic block built by a new helper `renderRuntimeStatDiagnostic(stat)`. The block carries:

* **Install-state pill** with five rendered labels mapped from `stat.installState`:

  | installState | pill label | tone |
  |---|---|---|
  | `installed_and_supervised` | "supervised" | good (green) |
  | `online_via_admin_port` | "online (admin port)" | warn (amber) |
  | `online_via_external_listener` | "online (external)" | warn (amber) |
  | `supervised_pool_not_ready` | "pool stuck" | critical (red) |
  | `awaiting_pool_registration` | "no pool registered" | critical (red) |
  | `unknown` / empty | "unclassified" | info (gray) |

* **Reason chip** carrying the categorized failure reason (`timeout`, `connection_refused`, `dns_failed`, `host_unreachable`, `socket_error`, `no_listener`, `no_endpoint_registered`, `supervised_pool_not_ready`).

* **Last-probe error line** — the human-readable line stored in `stat.lastErrorMessage`, with the relative-time stamp from `stat.lastErrorAtUtc`. Example: *"TCP connect to 192.168.1.7:7104 timed out after 200ms - target may be firewalled or down. (just now)"*

* **Next-step hint line** — the operator-actionable text from `stat.installHint`. Example: *"Catalog placeholder for MCP server 'computer-use'. No process listens on 192.168.1.7:7104. Register a managed pool to make this entry live: POST /api/pools with poolId=\"computer-use\" and an executable template."*

The block only renders when at least one of the five new fields is present, so older runtimes (pre-v0.9.56 servers serving the dashboard JS) fall back to the legacy red dot without breakage.

### MCP-server cards inherit the block automatically

The dashboard's MCP-server card grid synthesizes a sub-agent-shaped object and delegates to `renderSubAgentCard` (line 1331 comment: *"reuses renderSubAgentCard's exact visual structure"*). One JS edit covers both grids — no per-grid duplication.

### CSS chrome matching the existing TRON style

`resources/web/styles.css` adds:

* `.subagent-diagnostic-block` — ~80px compact frame, cyan-edged on the existing dark gradient.
* `.subagent-diag-pill` with four tones (`good` / `warn` / `critical` / `info`) using the existing card palette (`#1cf2c1`, `#ffaf3a`, `#ff3a5a`, neutral gray).
* `.subagent-diag-reason` — monospace category chip alongside the pill.
* `.subagent-diag-error` / `.subagent-diag-hint` — readable body lines with bolded "Last probe:" / "Next step:" labels in cyan accent.

## Risks and blockers

* **Older dashboard cached JS** — operators with a long-lived browser tab may have the v0.9.55 `app.js` cached. A hard refresh (Ctrl-F5) shows the new block. No server-side behaviour difference.
* **Localization** — pill labels and "Last probe:" / "Next step:" copy are English-only. Out of scope for this iteration; the i18n pattern is not yet established for the dashboard.
