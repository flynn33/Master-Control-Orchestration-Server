# Release Completion Report — v0.9.50

Maintenance release. Substantial bug: query strings broke route matching.

## Scope completed

### `?cache=bust` and other query strings no longer return 405

The v0.9.50 bug-hunt sent `GET /api/dashboard?cache=bust&t=12345` and got back **405 Method Not Allowed**. Same for `GET /api/version?v=1`. Investigation traced the bug to the request-path parser:

```
SimpleHttpServer::parseRequest stored the full request target
verbatim:                request.path = "/api/dashboard?cache=bust&t=12345"

Route handler used strict equality:
                         request.method == "GET" && request.path == "/api/dashboard"
                                                    ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                                                    fails because of the query suffix
```

The v0.9.28 `supportedMethodsForPath()` table DID strip the query string before lookup, so the path appeared "known" — and the v0.9.28 fall-through generated a 405 with `Allow: GET, HEAD, OPTIONS`. The worst possible response: a generic GET with a cache-bust parameter (standard browser behavior, dashboard polling, retry-with-jitter clients) got 405, the v0.9.28 logic confidently asserting that GET wasn't allowed when in fact GET *was* the method requested.

This affected **every GET endpoint** — `/api/version`, `/api/dashboard`, `/api/health`, `/api/governance/bundles/{platform}`, all of them — the moment any query string was attached.

The fix splits the request target at `'?'` in `parseRequest`:

```cpp
const auto queryPos = request.path.find('?');
if (queryPos != std::string::npos) {
    request.query = request.path.substr(queryPos + 1);
    request.path = request.path.substr(0, queryPos);
}
```

`request.path` is now the URI path only; new `request.query` holds the raw query string (no leading `?`) for the two routes that consume it (`/api/activity` and `/api/telemetry/events`). Every other route's strict equality comparison now works under arbitrary query strings.

The two query-aware routes simplified from `rfind(prefix, 0) == 0` (a substring match that absorbed the query suffix) to strict equality, since the query string is no longer in `request.path`.
