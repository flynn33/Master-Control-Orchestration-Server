# Release Completion Report — v0.9.52

Maintenance release. **Real injection bug found**: `poolId` accepted slashes / whitespace / control bytes, creating pools that were unreachable for removal.

## Scope completed

### Pool registration input validation

The v0.9.52 bug-hunt sent `POST /api/pools` with `poolId="foo/bar"` and got back `200 "Pool registered."`. The pool was then **stuck**:

```
GET /api/pools/foo/bar         -> 400 "Unknown pool sub-resource; expected leases|saturation."
                                   (the slash made the route parser see poolId="foo" subresource="bar")
POST /api/pools/foo%2Fbar/remove -> 200 "Unknown pool id."
                                   (URL-encoded slash decodes to literal '/' which doesn't match)
```

The pool was registered in memory AND persisted to the configuration file (because every successful `upsertPool` triggers `persistSupervisedPoolsToConfiguration`), but no API path could reach it for removal — `/api/pools/foo/bar/...` always parses the second slash as a sub-resource boundary. The same registration bug accepted:

- `poolId=""` → 200 with succeeded:false (should be 400)
- `poolId="foo bar"` (whitespace) → registers (would've been similarly stuck)
- `scalePolicy.minInstances=5, maxInstances=2` → registers (`ensureMinInstances` would loop trying to spawn workers up to an unreachable target)
- `scalePolicy.minInstances=-1` → registers (no sanity bound)

The fix adds four validation gates in `WorkerSupervisor::upsertPool`:

1. `poolId` is ASCII letters, digits, `-`, `_`, `.` only. This is exactly what every existing operator pool uses (`baseline-tools`, `sequential-thinking`, etc.); no migration impact.
2. `scalePolicy.minInstances >= 0`
3. `scalePolicy.maxInstances >= 1`
4. `scalePolicy.minInstances <= scalePolicy.maxInstances`

Failures route through HTTP 400 (was 200/false). The boot-path persistence loader also runs `upsertPool`, so any previously-persisted invalid pool (the v0.9.52 bug-hunt's stuck `foo/bar`) is now silently dropped on next boot — `live` no longer contains it, the next operator-driven `persistSupervisedPoolsToConfiguration` overwrites the disk file, and the bad entry is purged. Verified post-deploy: pool count back to the canonical 5.
