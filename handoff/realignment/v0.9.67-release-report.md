# Release Completion Report — v0.9.67

Resolved the long-standing "MCOS sub-agent runner architectural decision" by registering all 7 default sub-agents (sentinel, architect, forge, scribe, recon, nexus, watchtower) as supervised stdio MCP servers using the multi-spec worker pattern. Sub-agent reachable count goes **0/7 → 7/7**; total reachable across both kinds: 24/28 + 7/7 = 31/35. Gateway tools/list grows from 71 to 92.

## Scope completed

* `Specialization::AgentSentinel` / `AgentArchitect` / `AgentForge` / `AgentScribe` / `AgentRecon` / `AgentNexus` / `AgentWatchtower` enum values + CLI parser branches.
* Each sub-agent exposes 3 tools that bridge to admin endpoints + filesystem + structured templates:
  * **sentinel**: `sentinel.list_rules` → /api/forsetti/surface; `sentinel.recent_activity` → /api/activity?max=N; `sentinel.health_summary` → /api/health/summary.
  * **architect**: `architect.list_phases` → /api/forsetti/surface; `architect.discovery_doc` → /.well-known/mcos.json; `architect.draft_adr` returns templated ADR markdown for the LAN AI client to fill in.
  * **forge**: `forge.list_pools` → /api/pools; `forge.suggest_pool_template` builds a ready-to-POST pool template; `forge.list_specializations` is reflective — walks every catalog function in this binary and returns `{specialization, tools[]}` so an operator can see at runtime what the worker can do.
  * **scribe**: `scribe.list_release_reports` reuses `indexListFiles` pointed at the handoff dir with glob `v*-release-report.md`; `scribe.draft_release_report` returns templated markdown; `scribe.version_state` → /api/version.
  * **recon**: `recon.dashboard` → /api/dashboard; `recon.diagnostics` → /api/diagnostics/runtime-stats; `recon.gateway_tools` → /api/gateway/tools.
  * **nexus**: `nexus.health_summary` / `nexus.discovery` / `nexus.list_clients`.
  * **watchtower**: `watchtower.health_summary` / `watchtower.gateway_status` / `watchtower.activity_tail`.
* `MasterControlRuntime::registerSubAgentPool` lambda mirrors `registerWorkerPool` but with `kind=SubAgent` so the dashboard renders them in the sub-agent grid (separate from the MCP-server grid). 7 pools registered.

## Architecture decision recorded

The deferred "MCOS sub-agent runner architectural decision" item that was carried for 60+ iterations is now resolved: **sub-agents are stdio MCP servers exposing role-specific helper tools, supervised by the same WorkerSupervisor + LeaseRouter stack that backs MCP-server pools.** The role lives in the tool naming, descriptions, and the structured guidance the tools emit; the cognitive work happens at the LAN AI client. This makes sub-agents first-class citizens of the gateway aggregate alongside MCP servers.
