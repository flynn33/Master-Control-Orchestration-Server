# Release Completion Report — v0.9.32

Maintenance release. Hidden consequence of v0.9.31's `max=` fix surfaced by the v0.9.32 bug-hunt.

## Scope completed

### `/api/activity?max=N` now returns the LATEST N events, not the oldest

The v0.9.32 bug-hunt followed up on v0.9.31's `max=` parameter fix and probed `/api/activity?max=10`, expecting the 10 most recent events. The response came back with events from `2026-05-09T14:46:11Z` — the boot-time `pool_worker_respawn` records — instead of the live tail at `2026-05-09T20:55:12Z`. Six hours of staleness in a "recent activity" widget.

Root cause: `ActivityEventRing::read()` iterated the ring forward and stopped at `maxCount`:

```cpp
size_t collected = 0;
for (const auto& event : ring_) {
    if (collected >= maxCount) break;
    /* ...filter on sinceSeq, push event... */
}
```

Pre-v0.9.31 with `max=` ignored, this loop ran to completion and returned all events — the bug was hidden because the dashboard rendered everything and the operator scrolled to the bottom of the list. Once v0.9.31 honored `max=`, the loop now stopped early *at the start* of the ring, returning the OLDEST events. Exactly the opposite of what every "recent activity" widget wants.

The fix splits the read semantics by `sinceId`:

- **`sinceId` empty (the dashboard "show me recent activity" use case)**: tail mode. Compute the start index as `ring_.size() - take` and copy the slice forward. Result is the latest `maxCount` events, oldest-first within the slice — chronological inside the widget, but anchored to *now*.
- **`sinceId` non-empty (incremental polling)**: forward iteration as before. Caller already has events ≤ N and wants events > N in chronological order, so iterating from the start and capping at `maxCount` is correct (the cap is a safety net for callers that fall far behind; the next poll picks up where this one left off).

The persistence-restore path was already correct — `loadFromDiskOnce` rehydrates the ring with the persisted tail, so the in-memory ring already represents the correct slice across restarts.

**Verified live** (v0.9.32 deployment, ring at 476 events):

```
GET /api/activity?max=5
  highWaterMarkId: 476
  events returned: id=472, 473, 474, 475, 476        (LATEST 5, oldest-first)

GET /api/activity?max=1
  highWaterMarkId: 476
  events returned: id=476 (timestampUtc 20:58:23.523Z) — exactly the live tail

GET /api/activity?since=472&max=10  (incremental poll preserved)
  events returned: id=473, 474, 475, 476  (strictly > 472, chronological)

GET /api/activity  (no query — backward compat)
  476 events / 136 480 bytes (whole ring)
```

## Files changed

| File | Change |
|---|---|
| [`src/MasterControlApp/MasterControlRuntime.cpp`](../../src/MasterControlApp/MasterControlRuntime.cpp) | `ActivityEventRing::read()` branches on `sinceId.empty()`. Empty → tail mode (latest `maxCount`, oldest-first inside the slice). Non-empty → forward-iterate as before. The forward-iterate branch is byte-identical to v0.9.31, so the only behavior change is the no-`since` case. The `out.events.reserve(take)` in the new tail path avoids the silent reallocations that the forward-iterate version would have hit on the small-`max` case. |
| [`VERSION.json`](../../VERSION.json) | 0.9.31 → 0.9.32. |
