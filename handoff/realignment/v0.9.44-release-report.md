# Release Completion Report — v0.9.44

Maintenance release. Closes the four remove-route holdouts in the 500→400 / 400→404 sweeps.

## Scope completed

### Four remove routes were still returning 500 on bad bodies and 400 on missing targets

The v0.9.44 bug-hunt swept every `/remove` POST and found four sites that prior sweeps had partially missed:

| Route                                                 | pre-v0.9.44 (bad body) | v0.9.44 | pre-v0.9.44 (missing target) | v0.9.44 |
|---|---|---|---|---|
| `POST /api/platform-services/apple-hosts/remove`      | 500                    | **400** | 200/false                    | 200/false (no v0.9.x mapping yet) |
| `POST /api/runtime/mcp-servers/remove`                | 500                    | **400** | 404 (v0.9.38 already)        | 404 (preserved) |
| `POST /api/runtime/subagents/remove`                  | 500                    | **400** | 404 (v0.9.38 already)        | 404 (preserved) |
| `POST /api/runtime/subagent-groups/remove`            | 500                    | **400** | 400                          | **404** |

The pattern: each route called `adminApiService_->remove*Json(request.body)` which runs `nlohmann::json::parse(...).get<RemovalRequest>()` inline. The `parse_error` throws bubbled past the route's local 200/400 branch and hit the outer 500 handler. For routes (2) and (3), the v0.9.38 404-vs-400 message-branch logic was inside the same scope as the throwing call, so the 404 path never fired on a malformed body either — the throw landed in the outer handler before the message branch ran. Route (4) had been touched by v0.9.42 only on the upsert side; remove was unmodified.

The fix wraps each in try/catch returning 400 + the diagnostic body, and (for route 4) adds the 404-on-missing-target branch for parity with v0.9.30/v0.9.33/v0.9.37/v0.9.38/v0.9.43.

The `apple-hosts/remove` route with an unknown target id was not promoted to 404 in this iteration because the underlying service doesn't currently distinguish "removed an entry that didn't exist" from "no-op removal" cleanly — the `removeAppleRemoteHost` helper is idempotent. Carrying that to a future iteration alongside any other "missing-resource through 200/false" sites the next bug-hunt surfaces.
