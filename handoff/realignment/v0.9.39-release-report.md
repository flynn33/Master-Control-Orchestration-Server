# Release Completion Report — v0.9.39

Maintenance release. Closes the v0.9.30 deferred 503 sweep — six "service is not running" responses now return the right status.

## Scope completed

### 503 Service Unavailable for runtime-service-not-initialized responses

Six routes had defensive null-checks on internal services (CLU, worker supervisor, lease router, telemetry aggregator) and returned 200 + `succeeded:false` when the service was missing. RFC 7231 §6.6.4 reserves 503 for exactly this case: "the server is currently unable to handle the request due to a temporary overload or scheduled maintenance." The routes were tracked from v0.9.29's deferred work and surfaced again in every subsequent bug-hunt.

In normal operation these 503 responses never fire — all six services boot before the HTTP listener accepts requests, and they don't go away until shutdown. But during partial-init or graceful-shutdown windows, an inbound probe could hit them, and the right answer is 503 (try again shortly), not 200 (request succeeded with no result).

| Route                                                  | Service        | pre-v0.9.39 | v0.9.39 |
|---|---|---|---|
| `POST /api/governance/decisions`                       | CLU            | 200         | **503** |
| `POST /api/pools` (upsert)                             | Supervisor     | 200         | **503** |
| `POST /api/pools/{id}/{remove,scale,drain,leases}`     | Supervisor     | 200         | **503** |
| `POST /api/pools/{id}/leases` (acquire)                | Lease router   | 200         | **503** |
| `POST /api/leases/{id}/release`                        | Lease router   | 200         | **503** |
| `POST /api/telemetry/heartbeat`                        | Aggregator     | 200         | **503** |

The body shape is unchanged in every case (`{ succeeded: false, message: "X is not running." }`), so callers that already inspect the body keep working; HTTP-status-only callers (uptime probes, retry libraries) start seeing the right signal.
