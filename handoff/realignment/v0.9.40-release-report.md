## Scope completed

### `/api/pools/{id}` now exposes `quarantinedUntilUtc`

The v0.9.40 bug-hunt killed the baseline-tools worker 5 times in rapid succession to verify the v0.9.8 crash circuit breaker. The supervisor handled it correctly:

- Kills 1-4: `pool_worker_death` â†’ `pool_worker_respawn` (auto-recovery)
- Kill 5 (within 60s window): `pool_quarantine` event, auto-respawn paused for 60s

But the *only* API surfaces of the quarantine state were `/api/health/summary`'s `darkOrQuarantined` counter and the activity ring's `pool_quarantine` event. `/api/pools/{poolId}` showed `instances: []` â€” same shape as a deliberately-drained pool. A generic operator dashboard rendering per-pool status had to *infer* "this pool is quarantined" from the empty instance list combined with `minInstances: 1`. Possible, but the dashboard couldn't render an accurate "quarantined for Ns" badge or surface the recovery deadline.

The fix:

1. New `ManagedEndpointPool::quarantinedUntilUtc` field. Empty string when not quarantined; ISO-8601 UTC expiry timestamp when quarantined.
2. New `WorkerSupervisor::quarantineExpiryUtcLocked(poolId)` private helper that converts the in-memory `steady_clock::time_point` deadline into a wall-clock `system_clock` ISO-8601 string by adding the remaining duration to `system_clock::now()`. This stays accurate even if the monotonic and wall clocks have drifted apart.
3. `listPools()` and `findPool()` populate the field on each returned snapshot.
4. `to_json` / `from_json` for `ManagedEndpointPool` emit and accept the field. Round-trips through configuration persistence are safe â€” disk-persisted definitions never carry a real value.

The MSVC `to_time_t` overload required a `time_point_cast<system_clock::duration>` because adding `steady_clock::duration` (nanoseconds) to `system_clock::now()` yields a time-point with nanosecond precision, but the overload expects `time_point<system_clock, system_clock::duration>` (different precision per platform).
