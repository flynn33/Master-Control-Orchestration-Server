# Release Completion Report — v0.9.36

Maintenance release. **DNS-SD LAN advertisement was completely broken.** Fixed.

## Scope completed

### `DnsServiceRegister` rejected `pRegisterCompletionCallback = nullptr` with Win32 error 87

Pre-v0.9.36 both DNS-SD register sites — `PlatformServiceCatalogService::registerGatewayLocked` (the platform-gateway lanes for Windows/macOS/iOS) and `BeaconService::registerInstanceLocked` (the underlying `_mcos._tcp.local` LAN beacon) — set `request.pRegisterCompletionCallback = nullptr`. `DnsServiceRegister` rejects that with `ERROR_INVALID_PARAMETER` (Win32 87 / 0x57). The v0.9.34 bug-hunt found this in `/api/beacon`:

```json
"platformGateways": [
  { "platform":"windows","status":"registration_failed","last_register_error":"0x00000057",
    ... "serviceType":"_mastercontrol-windows._tcp.local" ... },
  { "platform":"macos","status":"registration_failed","last_register_error":"0x00000057", ... },
  { "platform":"ios","status":"registration_failed","last_register_error":"0x00000057", ... }
]
```

Operator impact: the mDNS service-type advertisements `_mastercontrol-windows._tcp.local`, `_mastercontrol-macos._tcp.local`, `_mastercontrol-ios._tcp.local`, and `_mcos._tcp.local` never reached the LAN. A client browsing by service type to discover MCOS got nothing back — exactly the failure mode the LAN-discovery rule (`.claude/rules/30-mcp-gateway-discovery.md`) is designed to prevent. The native HTTP gateway on port 8080 still worked because clients with the URL hard-coded or learned via the UDP beacon could connect, but the *primary* discovery surface was dead.

The fix at both call sites:

1. **Add a no-op completion callback.** The synchronous `DnsServiceRegister` return value carries the "registration accepted" decision; the callback fires later for completion observability. The OS handles re-broadcasting on its own. The `PDNS_SERVICE_INSTANCE` is owned by the registration record and freed in the deregister path, so the callback truly is just observability.

2. **Accept `DNS_REQUEST_PENDING` (= 0x2522) as success in addition to `ERROR_SUCCESS`.** Per Win32 docs, the async-with-callback path returns `DNS_REQUEST_PENDING` to mean "registration accepted, completion will fire later" — that's the success path for our usage pattern.

```cpp
request.pRegisterCompletionCallback = &dnsRegisterCallback;
const auto status = DnsServiceRegister(&request, nullptr);
registration.registered = (status == ERROR_SUCCESS || status == DNS_REQUEST_PENDING);
```
