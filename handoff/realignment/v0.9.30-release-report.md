# Release Completion Report — v0.9.30

Maintenance release. Hostname truncation + four 200-with-succeeded:false routes corrected.

## Scope completed

### Hostname truncation in /api/beacon, /api/dashboard, and DNS-SD records

Pre-v0.9.30 `WindowsHostTelemetryService::readComputerName()` called `GetComputerNameA(buffer, &size)` with a 16-byte buffer (`MAX_COMPUTERNAME_LENGTH + 1`). That API returns the **NetBIOS** name — capped at 15 octets by the NetBIOS protocol since Windows NT. Hosts with longer DNS names had their identity silently chopped:

```
PC-GAMING-R7-5800X (DNS hostname, 18 chars)
       ↓ readComputerName via GetComputerNameA
PC-GAMING-R7-58  (NetBIOS, truncated to 15)
```

Operator-visible symptom: the v0.9.30 bug-hunt probed `/api/beacon` and saw `hostName: "PC-GAMING-R7-58"` while `tools/call host_info` (which runs in the worker process and uses a different Win32 API) reported the full `PC-GAMING-R7-5800X`. Inconsistent identity surfacing across MCOS — bad for any LAN client that:

- Runs reverse-DNS verification of the beacon advertisement against the actual FQDN.
- Joins the dashboard's `platformGateways[].hostName` with locally-cached host records.
- Logs MCOS attribution by hostname for incident review.

The fix uses `GetComputerNameExA(ComputerNameDnsHostname, ...)` which returns the **DNS** hostname with no NetBIOS cap. RFC 1035 caps a DNS hostname at 255 octets, well within the new 256-byte buffer. The legacy `GetComputerNameA` is retained as a fallback only when the DNS query hard-fails — truncated-but-present beats nothing.

**Verified live**:

```
=== /api/beacon ===
"hostName": "PC-GAMING-R7-5800X"           (full, 18 chars)
"hostName": "PC-GAMING-R7-5800X.local"     (DNS-SD record)

=== /api/dashboard hostName references ===
7 occurrences of "PC-GAMING-R7-5800X"
0 occurrences of "PC-GAMING-R7-58"  (the old truncation)
```

### Four routes that returned 200 with succeeded:false now return the right HTTP status

The v0.9.29 release deferred fixing every "200-with-succeeded:false" site; v0.9.30 closes the four that the bug-hunt exposed:

| Route                                                | pre-v0.9.30 | v0.9.30 | rationale |
|---|---|---|---|
| `GET /api/pools/{unknown}`                           | 200         | **404** | RFC 7231 §6.5.4: resource does not exist |
| `GET /api/pools/{poolId}/{unknown-subresource}`      | 200         | **400** | route-shape error |
| `POST /api/pools/{poolId}/{unknown-action}`          | 200         | **400** | route-shape error |
| `POST /api/leases/{leaseId}` (without `/release`)    | 200         | **400** | route-shape error |

These follow the same operator-monitoring rationale as v0.9.29's parse-error fix: 200 OK responses with `succeeded:false` bodies don't surface in any monitoring system that buckets by status code (Prometheus blackbox, Datadog HTTP synthetic, browser `fetch.ok`, retry libraries that retry 4xx/5xx). The 404 and 400 status codes give operator dashboards and client-side error boundaries the right signal.

**Verified live**:

```
GET /api/pools/does-not-exist                        -> 404 ("Unknown pool id.")
GET /api/pools/baseline-tools/badsubresource         -> 400 ("Unknown pool sub-resource...")
POST /api/leases/foo                                 -> 400 ("Lease route is POST /api/leases/...")
POST /api/pools/baseline-tools/badaction             -> 400 ("Unknown pool action...")

GET /api/pools/baseline-tools                        -> 200  (existing route untouched)
tools/list via /mcp                                  -> 33 tools (gateway pipeline intact)
```
