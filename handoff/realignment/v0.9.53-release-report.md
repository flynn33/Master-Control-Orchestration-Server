# Release Completion Report — v0.9.53

Maintenance release. Same URL-segment-id injection bug found across four more registration endpoints.

## Scope completed

### `clientId`, `mcpServerId`, `subAgentId`, `groupId` now validated like `poolId` (v0.9.52)

The v0.9.53 bug-hunt followed up the v0.9.52 pool injection across the four sister registration endpoints. Same bug everywhere:

| Endpoint                              | pre-v0.9.53 with id='foo/bar' | v0.9.53 |
|---|---|---|
| `POST /api/clients`                   | 200 "LAN client registered." (then unreachable for DELETE) | **400** |
| `POST /api/runtime/mcp-servers`       | 200 "Custom MCP server settings updated." | **400** |
| `POST /api/runtime/subagents`         | 200 "Custom sub-agent settings updated." | **400** |
| `POST /api/runtime/subagent-groups`   | 400 (incidentally caught by an unrelated conflict check) | **400** (now explicit) |

Every post-registration request to the entity at `/api/<resource>/foo/bar` would split the slash into a sub-resource segment (`foo` with subresource `bar`), so the entity was unreachable for direct GET / DELETE.

The fix factors the v0.9.52 inline `poolId` character-set validator into a shared helper:

```cpp
bool isSafeUrlSegmentId(const std::string& value) {
    if (value.empty()) return false;
    for (const unsigned char c : value) {
        const bool ok = (c >= 'a' && c <= 'z') || (c >= 'A' && c <= 'Z')
                     || (c >= '0' && c <= '9') || c == '-' || c == '_' || c == '.';
        if (!ok) return false;
    }
    return true;
}
```

Used in `WorkerSupervisor::upsertPool`, `LanClientAccessService::upsertClient`, `McpServerCatalogService::upsertMcpServer`, `SubAgentCatalogService::upsertSubAgent`, and `SubAgentGroupService::upsertGroup`. Failure routes through HTTP 400 (was 200/false) with a uniform `"<Resource> ID is invalid. Id must be ASCII letters, digits, '-', '_', or '.' (no slashes, whitespace, or other URL-unsafe characters)."` message.

`LanClientAccessService` also gained a constructor-time scrub: any previously-persisted client with an invalid id is dropped from `state_->configuration.lanClients` and the config file is rewritten. The v0.9.52 bug-hunt's stuck `foo/bar` client (which the new validator would have rejected at upsert time) is purged on next boot — verified post-deploy: client count went from 1 to 0.

The other registries (mcp-servers, sub-agents, sub-agent groups) didn't need a scrub because their registration paths route through the validation gate from boot, but a `foo/bar` mcp-server registered before v0.9.53 would still need a manual remove via the body-based `/api/runtime/mcp-servers/remove` route (which works because it doesn't use the URL path for the id).
