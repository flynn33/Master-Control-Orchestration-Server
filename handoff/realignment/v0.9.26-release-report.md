# Release Completion Report — v0.9.26

Maintenance release. RFC-compliance fix found by the v0.9.26 bug-hunt.

## Scope completed

### HEAD requests on operator endpoints

Pre-v0.9.26 every `HEAD /api/*` returned 404 because `SimpleHttpServer`'s route table only matched `request.method == "GET"`. Common monitoring tools (Pingdom, Uptime Robot, healthchecks.io, even `curl -I`) probe with HEAD for cheap liveness checks; getting a 404 made every operator route look dead.

Per RFC 7231 §4.3.2 the server SHOULD process HEAD identically to GET except no body. Fix:

- `SimpleHttpServer::handleClient` rewrites `request.method` from `"HEAD"` to `"GET"` before calling the handler. Every existing GET route handles HEAD transparently.
- `HttpResponse` gained a `suppressBody` flag; the HEAD path sets it to true so `sendResponse` emits status + Content-Type + Content-Length matching the GET-equivalent body size, but skips writing the body bytes themselves.

**Verified live**:

```
                  HEAD bytes     GET bytes
/api/health       90            144
/api/version      90            149
/api/pools        92          9,693
/api/discovery    92          1,891
/api/health/summary 91         707
```

HEAD = headers only (~90 bytes — status line + headers + blank line). GET = headers + body. Status was 200 in both cases.
