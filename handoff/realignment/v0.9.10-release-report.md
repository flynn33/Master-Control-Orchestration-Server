# Release Completion Report — v0.9.10

Maintenance release. Closes one item deferred from v0.9.9 plus a real edge-case bug surfaced by the v0.9.10 bug-hunt probe.

## Scope completed

### 1. `GET /mcp` returns HTTP 405 (was: HTTP 204)

Pre-v0.9.10 a `GET` request against `/mcp` produced `HTTP 204 No Content`. Trace: the gateway didn't check the request method; an empty body parsed to `{}`; the dispatcher saw no `method` and no `id`, classified the request as a JSON-RPC 2.0 notification, and returned the empty-string sentinel which the HTTP layer then translated to 204.

That was operationally confusing — a client doing a GET (e.g., trying to open a Streamable-HTTP SSE stream) saw an opaque success for a request that should have been refused.

v0.9.10 explicitly checks the verb on the `/mcp` route and returns `HTTP 405 Method Not Allowed` with a structured body for any non-POST request:

```json
{
  "error": "MCP path requires POST",
  "path": "/mcp",
  "method_received": "GET",
  "hint": "MCP traffic is JSON-RPC 2.0 over POST. SSE upgrade is not implemented in this build."
}
```

The `method_received` field names the verb (GET / PUT / DELETE / HEAD / OPTIONS / TRACE / CONNECT / OTHER). PATCH falls through to "OTHER" because `HttpVerbPATCH` isn't in older Windows SDKs.

**Verified live:** `Invoke-WebRequest -Method GET http://localhost:8080/mcp` returned HTTP 405. POST regression: `tools/list` still returns 33 tools across 5 pools.

### 2. Watchdog detach diagnostic event

When the v0.9.9 bounded-shutdown path takes the detach branch (watchdog didn't exit within `kShutdownJoinTimeoutMs = 5000`ms), v0.9.10 now emits a `supervisor_shutdown_wedge` ActivityEvent before detaching:

- `kind`        = `"supervisor_shutdown_wedge"`
- `actor`       = `"supervisor-destructor"`
- `target`      = `"watchdog-thread"`
- `statusCode`  = `504`
- `message`     = `"Watchdog did not exit within 5000ms of watchdogRunning_=false. Detaching. supervisor-state-at-detach: pools=N children=M."`

The pool count + child count snapshot at detach time helps spot wedge patterns:
- Always wedges with a respawn-in-flight → CreateProcessW pattern
- Always wedges with high children → inherited-handle exhaustion pattern
- Always wedges with high pools → mutex_ contention pattern

A new helper `appendSupervisorWedgeActivity` is defined alongside `appendPoolLifecycleActivity` and forward-declared near the supervisor class for the same reason: the supervisor is defined earlier in the TU than `ActivityEventRing`, but the helpers can see both.

Pre-v0.9.10 the detach was silent; post-mortem analysis had no signal that a wedge had occurred.

## Risks and blockers

- **Service stop measured at 14.8 seconds during v0.9.10 deploy** — past the 5s bounded watchdog deadline, but no `supervisor_shutdown_wedge` events appeared in `/api/activity` after restart. Two non-mutually-exclusive hypotheses:
  - Most likely: `BeaconService::stop()` calls `worker_.join()` and the beacon worker sleeps for `beaconBroadcastIntervalSeconds=15s` between broadcasts. If `running_=false` is set mid-sleep, the worker doesn't observe the flag until the sleep ends; total stop time is bounded by the beacon interval, which is ~15s. v0.9.11 candidate: convert the beacon worker's `std::this_thread::sleep_for` to a `std::condition_variable::wait_for` so `stop()` can interrupt it.
  - Activity ring is in-memory and doesn't survive restart, so a wedge event from the old process is invisible to the new one. Persistent activity ring is also a v0.9.11+ candidate.
- **HttpVerbPATCH not in older Windows SDKs** — addressed by falling through to "OTHER" in the verb-name string.
