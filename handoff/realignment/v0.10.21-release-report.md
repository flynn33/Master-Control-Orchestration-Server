# Release Completion Report — v0.10.21

Three infrastructure fixes from the operator's testing-phase audit, closing items 5+6 from the original 8-item list (plus the supervisor env-block carry-over from v0.10.19).

## 1. Findings → resolution

### Fix A — Supervisor honors `pool.template_.environment`

**Pre-v0.10.21 behaviour:** `WorkerSupervisor::startInstanceLocked` at `MasterControlRuntime.cpp:8577` called `CreateProcessW(..., nullptr /*lpEnvironment*/, ...)` so every stdio-supervised child inherited the MCOS service's environment unchanged. The v0.10.19 sqlite seed set `PATH=C:\Program Files\nodejs\;...` in its template; that override was silently dropped because the supervisor never passed it through. Other pool templates with `template_.environment` entries (e.g. v0.10.18's memory pool's `MEMORY_FILE_PATH`) had the same silent-ignore.

**Fix:** Build a merged Unicode env block from `GetEnvironmentStringsW()` + the pool's overrides. The pattern mirrors the existing `runOneShotProcess` env-block code at line 1006–1039 (which already did this for one-shot child processes). Allocation via `HeapAlloc(GetProcessHeap(), ...)`, freed unconditionally after `CreateProcessW`. `CREATE_UNICODE_ENVIRONMENT` flag is OR'd into `creationFlags` only when the block is non-null. If `pool.template_.environment` is empty, the block is `nullptr` and the child inherits parent env as before (preserves the v0.10.18-era behaviour for pools that don't override).

### Fix B — Error-severity TelemetryEvents persist

**Pre-v0.10.21 behaviour:** `runtime/events.jsonl` (at `C:\Users\Public\Documents\Master Control Orchestration Server\logs\runtime\`) held 28 entries, ALL severity=`info`. The 168 telemetry errors operators saw during the v0.10.19 sqlite-crash remediation lived only in the in-memory `TelemetryAggregator::events_` deque and were lost on service restart. The aggregator's `recordEvent` path was the canonical sink for all categorized warnings/errors but emitted nothing to disk.

**Fix:** In `TelemetryAggregator::recordEvent` at line 9938, before the in-memory ring push, call `MasterControl::Diagnostics::appendEvent(L"runtime", to_string(severity), event_name, message, data)` when `event.severity` is `Warning`, `Error`, or `Critical`. The event name is `"worker_event"` for `TelemetryCategory::Worker` and `"telemetry_event"` otherwise. Data carries `category`, `poolId`, `clientId`. Wrapped in try/catch so a disk-full failure on `appendEvent` cannot bubble up and break the aggregator's storage path. `Info`-severity events stay in-memory-only because they are dominated by the dashboard-snapshot fast path that already persists to a separate file (`runtime/telemetry.jsonl`).

### Fix C — Persistent-log rotation

**Pre-v0.10.21 behaviour:** `runtime/telemetry.jsonl` had reached 25.7 MB / 101 474 lines at audit time. `appendJsonLine` opened the file in append mode and wrote without any size check or rotation. The disk-fill horizon was unbounded.

**Fix:** New `rotateIfOversizedLocked(filePath)` helper in `MasterControlDiagnostics.h` and a `kLogRotationBytes = 50 * 1024 * 1024` constant. Called at the top of `appendJsonLine` under the existing static `writeMutex`. When the live file's size exceeds 50 MB, the helper:

1. Removes any prior `<filePath>.1.jsonl` rotated file (silently if it doesn't exist).
2. Renames the live file to `<filePath>.1.jsonl`.
3. Returns so the subsequent `std::ofstream(filePath, ios::app)` opens a fresh empty file.

If the rename fails (e.g. another reader holds the file), the helper falls through and the append continues against the oversized live file. Total disk consumption per component caps at ~100 MB (live + one rotated). The threshold is configurable via the constant.

## 3. Public contracts changed

- **Pool `template_.environment` is now effective.** Operators registering pools with environment overrides see their values applied. Pre-v0.10.21 pools that set the field saw a silent no-op. Pool templates without overrides keep the prior nullptr-env behaviour (child inherits parent's env).
- **`runtime/events.jsonl` now contains Warning/Error/Critical entries.** Operators with downstream consumers that parse this file see new rows. The schema is unchanged — the existing `{capturedAtUtc, component, severity, event, message, data}` shape covers both info and non-info severities. Consumers that filter by severity continue to work.
- **`runtime/events.jsonl` and `runtime/telemetry.jsonl` rotate at 50 MB.** Operators with downstream consumers watching the file inode see a new file appear on rotation. The old data moves to `<name>.1.jsonl`. Consumers that follow the live file's tail keep working; consumers that read the file by absolute path see the rotated content move to the `.1.jsonl` sibling.

No removed routes. No removed Shell controls. No removed pool ids. No JSON schema changes.

## 5. Risks and blockers

- **Env-block performance.** Each supervised-pool spawn now walks the parent's env once and builds a merged map. For typical env sizes (~50 vars) this is microsecond-class. Acceptable for the spawn path which is already milliseconds-class for `CreateProcessW`.
- **Rotation race.** If two `appendJsonLine` callers race on rotation, the first wins the rename, the second's `appendJsonLine` opens the now-empty live file and continues. The static `writeMutex` inside `appendJsonLine` serializes the check + rename + write trio. External readers (e.g. the operator tailing the file) that have the file open during rotation may see EOF or get a short read; standard tail-on-rename behavior applies.
- **Disk-full while rotated file is being written.** If `<name>.1.jsonl` already exists and the rotation removes it, the disk transiently has less data than at peak. Acceptable — the operator's auditable history is at most kLogRotationBytes (50 MB) of rotated data; reduced retention during a disk-full event is the right trade-off vs failing to write new entries.
