# Release Completion Report — v0.9.69

Boot self-test framework. After the HTTP server starts, MCOS spawns a background thread that waits ~3s for pool handshakes to settle, then runs a 37-probe battery covering admin / gateway / every supervised pool / activity-ring persistence / telemetry sampler / on-disk worker exe presence. Failed probes route through the existing v0.8.7 Error Reporting frame on the WinUI Overview surface (which harvests `statusCode >= 400` activity events) with **no extra wiring**.

## Scope completed

* `SelfTestResult` + `SelfTestSnapshot` models in `MasterControlModels.h` with NLOHMANN serialization.
* `MasterControlApplication::Impl::runBootSelfTestsAsync` spawns a worker thread that calls `runBootSelfTestsNow` after a 3s settle. Stop hook joins the thread on shutdown.
* `runBootSelfTestsNow` runs probes in 6 categories:
  * `http_endpoint` — TCP connect to admin port (browserPort)
  * `gateway` — TCP connect to gateway port + `mcpGateway_->CurrentStatus().state == Running`
  * `pool_handshake` (kind=mcp-server) and `sub_agent` (kind=SubAgent) — for every registered pool, check `readyInstanceCount >= 1` when `minInstances >= 1`; pools with `minInstances = 0` are intentionally skipped (operator-staged scale-up).
  * `telemetry` — activity-ring persistence health + telemetry sampler liveness
  * `process` — on-disk presence of `mcos-baseline-tools-worker.exe`
* Each probe writes an `ActivityEvent` with `kind="self_test"` and `statusCode 200|500`, so the WinUI shell's existing error-event harvester (`statusCode >= 400`) picks failures up automatically.
* Aggregate `self_test_summary` event with `statusCode 200|500` at the end of each sweep.
* `selfTestTcpProbe` helper uses the same non-blocking-connect / select pattern as `AdminApiService::probeReachabilityCached`, but standalone so the self-test machinery doesn't depend on AdminApiService internals.
* Routes:
  * `GET /api/self-tests` — returns the cached snapshot. `pending=true` until the first sweep completes (~3s after service start).
  * `POST /api/self-tests/run` — re-runs and returns fresh results.

## Risks

* **3s settle is heuristic.** A pool that takes > 3s to reach Ready will false-fail. Adjust by re-running via `POST /api/self-tests/run` after the pool stabilizes; long-term we could also re-run on every pool state change.
* **TCP connect on the gateway port is a coarse probe.** It confirms the listener is bound but not that MCP `tools/list` returns anything. The full handshake check is implicit in the per-pool Ready-instance probe; a bound-but-broken gateway would flag through `gateway.state_running != true`.
