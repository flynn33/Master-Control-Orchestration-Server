# Release Completion Report — v0.9.28

Maintenance release. RFC-compliance fix found by the v0.9.27 bug-hunt.

## Scope completed

### 405 Method Not Allowed with Allow header for unsupported verbs on known routes

Pre-v0.9.28 every operator API route used the pattern `if (request.method == "GET" && request.path == "/api/...")`. Hitting a real route with the wrong verb (`DELETE /api/version`, `GET /api/install/zip`, `POST /api/health`, etc.) fell through to `staticFileResponse` which returned a generic 404 — indistinguishable from "endpoint doesn't exist." Per RFC 7231 §6.5.5 a known URI hit by an unsupported method MUST return 405 with an `Allow` header listing the methods the resource accepts; 404 is reserved for resources the server doesn't recognize.

Found by the v0.9.27 bug-hunt: `DELETE/PUT/PATCH/TRACE /api/version` all returned 404 with no `Allow` header. Fix:

- New `supportedMethodsForPath()` lookup — exact-match table of every literal-path route plus a prefix table for `startsWith()` routes — mirrors what's registered in `handleHttpRequest`.
- New `buildAllowHeader()` canonicalizes the `Allow` value: always adds `HEAD` when `GET` is supported (the v0.9.26 HEAD→GET rewrite makes that universally true), always adds `OPTIONS` (the v0.9.27 preflight short-circuit handles every path), emits verbs in stable order `GET, HEAD, POST, PUT, PATCH, DELETE, OPTIONS`.
- Before the `staticFileResponse` fall-through, if the path is known but the verb isn't in the allow set, return 405 with the `Allow` response header and a JSON body carrying `{ succeeded: false, message: "Method Not Allowed", path, method, allow }` so a misconfigured client gets enough to self-correct.
- `HttpResponse` gained an `extraHeaders` field (`vector<pair<string,string>>`) so the `Allow` header rides alongside the standard CORS block. `SimpleHttpServer::sendResponse` emits it before `Connection: close`. Forward-compat for `Retry-After`, `Location`, `ETag` etc. when later routes need them.

**Verified live**:

```
=== known path / unsupported verb ===
DELETE /api/version  -> 405  Allow: GET, HEAD, OPTIONS
PUT    /api/version  -> 405  Allow: GET, HEAD, OPTIONS
PATCH  /api/version  -> 405  Allow: GET, HEAD, OPTIONS
TRACE  /api/version  -> 405  Allow: GET, HEAD, OPTIONS

=== unknown path stays 404 ===
GET /api/does-not-exist  -> 404

=== known path / supported verb stays 200 ===
GET /api/version  -> 200
GET /api/health   -> 200

=== POST-only path with wrong verb ===
GET    /api/install/zip  -> 405  Allow: POST, OPTIONS
DELETE /api/pools        -> 405  Allow: GET, HEAD, POST, OPTIONS

=== 405 body ===
{"allow":"GET, HEAD, OPTIONS","message":"Method Not Allowed",
 "method":"DELETE","path":"/api/version","succeeded":false}
```

The path→methods table is maintained by hand alongside the route registrations in `handleHttpRequest`. The v0.9.28 commit comment near the table explicitly notes this so future route additions don't drift; long-term this could be unified into a single registration mechanism, but the duplication is small (~80 routes, ~100 lines of declarative table) and the alternative — refactoring every inline route handler to register through a router — is well outside one iteration's scope.
