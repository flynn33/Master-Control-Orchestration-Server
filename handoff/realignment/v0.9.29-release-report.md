## Scope completed

### Malformed-JSON and missing-required-field POST responses now return 400, not 200

The v0.9.29 bug-hunt fired a parse-error probe at `/api/governance/decisions`:

```
POST /api/governance/decisions  body: '{"this":'
```

and saw `200 OK` with `{ succeeded: false, message: "Invalid GovernanceEnforcementRequest JSON: ..." }`. This is the worst kind of API error â€” the *body* says it failed, but the HTTP status says it succeeded. Real consequences:

- **Browser fetch** â€” `response.ok` is true for 200; client code that branches on `.ok` treats this as success.
- **Prometheus blackbox exporter / Datadog HTTP synthetic / Pingdom** â€” all bucket by status code. A "succeeded:false 200" doesn't show up as an error in any monitoring dashboard.
- **Retry libraries** (axios-retry, http-retry, urllib3) â€” typically retry 4xx/5xx, never 2xx. A misconfigured client could hammer MCOS with parse errors forever and no automatic retry/backoff would fire.
- **nginx upstream metrics** â€” `nginx_http_upstream_requests_total{status="200"}` counts these as healthy; the operator dashboard misses the issue.

Per RFC 7231 Â§6.5.1: "The 400 (Bad Request) status code indicates that the server cannot or will not process the request due to something that is perceived to be a client error (e.g., malformed request syntax, ...)." A request body that fails JSON parsing is squarely 400.

The fix passes the explicit `400` status code on the four catch sites that returned an "Invalid X JSON" / missing-required-field result without one. The `jsonResponse(value, statusCode)` helper has supported a status-code parameter since the original PHASE-04 introduction; routes added later (`/api/clients`, `/api/runtime/mcp-servers`, `/api/setup/advanced-mode`, etc.) had been passing 400 correctly. The pre-PHASE-04 routes were the holdouts.

**Verified live**:

```
=== malformed JSON ===
POST /api/governance/decisions {"this":   -> 400
POST /api/governance/decisions <empty>    -> 400
POST /api/pools 'not-json'                -> 400
POST /api/telemetry/heartbeat {"oops":    -> 400

=== precondition ===
POST /api/telemetry/heartbeat {"clientId":""}  -> 400 ("ClientHeartbeat.clientId is required.")

=== happy path unchanged ===
POST /api/telemetry/heartbeat {"clientId":"v0.9.29-probe","clientType":"diagnostic"}  -> 200

=== regressions ===
GET /api/version       -> 200
GET /api/does-not-exist -> 404
DELETE /api/version     -> 405 (Allow: GET, HEAD, OPTIONS)
tools/list via /mcp     -> 33 tools (18991 bytes)
```

The four sites that swallow parse errors silently and fall back to a default (lease-acquire `LeaseRequest`, lease-release `reason`, setup-complete `skippedSteps`, governance-execute `payload`) were intentionally lenient pre-v0.9.29 â€” they treat empty/optional body as no body â€” and remain unchanged. Those are not parse-error-as-success; they're parse-error-as-empty-body, which is the documented contract.
