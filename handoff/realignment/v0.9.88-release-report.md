# Release Completion Report — v0.9.88

Provider-specific supervisor instructions now include a concrete curl one-liner for the connect/confirm step.

## Scope completed

### Curl example inlined in instructions block

The v0.9.83 provider-specific instructions named the connect/confirm endpoint but left the operator (on the LAN-client side) to construct the JSON body themselves. The body shape (configId / providerId / clientId / capabilities array / fingerprint) is non-obvious and not documented elsewhere on the receiving machine.

v0.9.88 emits a copy-paste-ready curl line wired with the actual values for this assignment:

```bash
curl -X POST http://127.0.0.1:7300/api/supervisor/connect/confirm \
  -H 'Content-Type: application/json' \
  -d '{"configId":"CFG-20260511T005747Z-2f23",
       "providerId":"claude",
       "clientId":"claude-supervisor-client",
       "capabilities":[
         "supervisor.get_context",
         "supervisor.list_pending_decisions",
         "supervisor.get_decision_packet",
         "supervisor.submit_decision",
         "supervisor.issue_remediation",
         "supervisor.get_worker_report",
         "supervisor.get_task_evidence",
         "supervisor.list_active_tasks"
       ],
       "fingerprint":"sha256:44d5ff9eecac4a5a"}'
```

The configId, providerId, capabilities, and fingerprint are spelled out from this exact assignment. The clientId is a `{provider}-supervisor-client` placeholder that the operator can rename. The admin URL is derived from `context_.discoveryEndpoint` (which already lives at the admin host:port) so the curl example always points at the right server for this deployment.

All three provider paths (Claude / ChatGPT / Grok) now end their step lists with `Sample curl: <command>` instead of the prior `POST /api/supervisor/connect/confirm with this config's configId` (which left the receiving operator to construct the body manually).

### End-to-end validation

Ran the EXACT curl from a freshly-generated Claude config against the live v0.9.88 service:

```
POST /api/supervisor/connect/confirm
→ 200 OK
→ { "ok": true, "state": "connected", "clientId": "claude-supervisor-client",
    "connectedAtUtc": "2026-05-11T00:57:59Z" }
```

State transitioned `config_generated` → `connected`. The path documented in the generated config actually works.
