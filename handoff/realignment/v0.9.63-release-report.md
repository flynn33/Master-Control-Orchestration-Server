# Release Completion Report — v0.9.63

Two HTTP-bridge specializations: `client-tracker` (`clients.list` → `/api/clients`) and `metrics` (`metrics.host` → `/api/host/telemetry`, `metrics.gateway` → `/api/telemetry/gateway`). Reachable count goes 10/28 → 12/28; gateway tools grow from 36 to 39.

## Scope completed

### `httpBridgeGet(url, timeoutMs)` helper

Worker-side helper that shells out to `curl.exe` (built into Windows 10+ / Server 2019+) using the existing `runProcessCaptured` infrastructure. The `-w "\n__MCOS_HTTP_STATUS__:%{http_code}"` write-out tail lets the worker split body from status code without depending on libcurl. The body is parsed as JSON; on parse failure it's passed through as `bodyText`. Captures `bridgeFailed: true` + `bridgeError` honestly when curl can't be launched or the call times out — no fake 200s.

Why curl rather than WinHttp: ~150 lines of `WinHttpOpen`/`WinHttpConnect`/`WinHttpSendRequest`/`WinHttpReadData` versus 5 lines around `runProcessCaptured`. Curl is a system component on every supported MCOS host; using it doesn't add a dependency.

### `client-tracker` specialization

* `Specialization::ClientTracker` enum + CLI parser branch.
* `clientTrackerCatalog()` exposes `clients.list` (no parameters).
* Dispatch calls `httpBridgeGet("http://localhost:7300/api/clients", 5000)` and surfaces the result via the MCP text-content shape.

### `metrics` specialization

* `Specialization::Metrics` enum + CLI parser branch.
* `metricsCatalog()` exposes two tools:
  * `metrics.host` → `GET /api/host/telemetry` (cpuPercent, memoryPercent, diskPercent, hostname, capturedAtUtc, ...). Honest-unavailable contract preserved: `capturedAtUtc=""` means the sampler hasn't taken its first reading yet (PHASE-08 §9).
  * `metrics.gateway` → `GET /api/telemetry/gateway` (gateway throughput + tool aggregate).

### Auto-registered pools

`MasterControlRuntime` upserts both `client-tracker` and `metrics` pools at boot — same `mcos-baseline-tools-worker.exe` binary, distinct `--specialization` arg, `minInstances=1`.

## Risks and blockers

* **HTTP bridge depends on curl.exe being on PATH.** Honest failure mode (`bridgeFailed: true` + `bridgeError`) is exposed; on hosts without curl the operator sees an actionable error rather than a fake success.
* **Bridge is read-only.** Writes (POST/DELETE) are out of scope intentionally: any state-changing tool would need explicit confirmation surface and lease-router integration. Read-only bridges are safe to expose to AI clients without that work.
