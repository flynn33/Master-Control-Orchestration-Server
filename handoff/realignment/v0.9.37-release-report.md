## Scope completed

### `DELETE /api/clients/{unknown-id}` returns 404, not 400

The v0.9.37 bug-hunt probed the LAN-client lifecycle and found:

```
DELETE /api/clients/never-registered  -> 400  body: { succeeded:false, "LAN client not found." }
```

Same pattern as the v0.9.30 `/api/pools/{unknown}` and v0.9.33 `/api/leases/{unknown}` fixes: a request that targets a resource that never existed (or has already been removed) is a 404 by RFC 7231 §6.5.4, not 400. 400 is reserved for malformed requests / governance denials / validation failures — cases where the *request itself* is wrong, not where the addressed resource is missing.

The fix branches on `result.message`:

```cpp
if (result.succeeded) { return jsonResponse(result, 200); }
const int status = (result.message == "LAN client not found.") ? 404 : 400;
return jsonResponse(result, status);
```

Other `removeClient` failure modes (governance denial via `enforceGovernance`, validation errors from the underlying access service) still return 400. The body shape is unchanged in both cases.
