# Release Completion Report — v0.9.43

Maintenance release. CLU approval queue route status codes — closes the v0.9.39 sweep gap.

## Scope completed

### `/api/clu/approvals/{id}/{approve,reject}` route status codes

The v0.9.43 bug-hunt found two more sites still returning 400 instead of 404 for missing-resource cases, plus two 500 → 503 sites for service-not-running cases that the v0.9.39 sweep missed.

| Probe                                                      | pre-v0.9.43 | v0.9.43 |
|---|---|---|
| `POST /api/clu/approvals/{unknown}/approve`                | 400         | **404** |
| `POST /api/clu/approvals/{unknown}/reject`                 | 400         | **404** |
| Same routes when `governanceApprovalQueueService_` is null | 500         | **503** |

The CLU approval queue stages governance decisions that returned `RequiresOperatorApproval`. An operator approving / rejecting an unknown deferred-action id is the "no such queue entry" case — 404 by RFC 7231 §6.5.4. 400 is reserved for malformed requests. The 500 → 503 transition matches the v0.9.39 sweep: an uninitialized service is "temporarily unavailable" (RFC 7231 §6.6.4), not "internal server error."
