# Release Completion Report — v0.9.6

Maintenance release. Closes the two top items deferred from v0.9.5 plus a self-test robustness fix surfaced by a live multi-host LAN.

## Scope completed

### 1. Background watchdog thread (proactive worker death detection)

Pre-v0.9.6 the supervisor's reap was opportunistic — it ran inside `listPools` / `findPool` / `ensureMinInstances` / `scaleUpOnce`. A pool with no traffic + a dead worker stayed dark indefinitely on a quiet LAN: a worker process could OOM, segfault, or be killed externally, and unless an operator queried the pool listing, MCOS never noticed and never respawned.

v0.9.6 adds a watchdog `std::thread` per `WorkerSupervisor` instance. The thread:
- Sleeps in 100ms steps so the destructor's `watchdogRunning_ = false` signal is observed promptly (`join()` returns within 100ms).
- Wakes every 2 seconds (`kWatchdogIntervalMs = 2000`) and calls `reapDeadInstancesLocked()` under `mutex_`.
- Serializes with operator API calls via the same mutex everything else uses.

Cost: a few microseconds of `GetExitCodeProcess` every 2s per supervised child. No new dependency or thread-pool infrastructure.

**Verified live:** killed `memory#3` worker (pid 10328); waited 5 seconds without any operator query; queried the pool — got `memory#6` running with a different pid. The watchdog detected and replaced the worker proactively.

### 2. Explicit `tools/list` cache invalidation on pool changes

v0.9.5 introduced a 30-second TTL on the gateway's `toolCatalogCache_`, dropping repeat-`tools/list` cost from O(pools × stdio-roundtrips) to a pointer copy. The drawback was that operator pool changes (`POST /api/pools` and the per-pool action routes) didn't surface in LAN-client `tools/list` output until the TTL expired.

v0.9.6 adds:
- **`IMcpGateway::InvalidateToolCatalog()`** — virtual method on the gateway interface with a no-op default. `FakeMcpGatewayAdapter` inherits the no-op; `NativeHttpSysGatewayAdapter` overrides it to stamp `toolCatalogCacheValidUntil_` to epoch under `mutex_`, forcing the next `refreshToolCatalogLocked` to do a fresh fan-out.
- **Wire-up in the four pool-admin POST handlers** in `MasterControlRuntime.cpp`: `POST /api/pools` (upsert), `POST /api/pools/{id}/remove`, `POST /api/pools/{id}/scale`, `POST /api/pools/{id}/drain`. Each calls `mcpGateway_->InvalidateToolCatalog()` after a successful operation, before the telemetry-aggregator emit.

**Verified live:**
- Initial `tools/list` returned 33 tools across 5 pools.
- `POST /api/pools/sqlite/remove` → next `tools/list` returned 28 tools (no sqlite tools), no TTL wait.
- Re-register sqlite + scale → next `tools/list` returned 33 tools (sqlite tools back), no TTL wait.

### 3. Discovery self-test robustness — skip foreign-host beacons

The `scripts/check-mcos-discovery.ps1` UDP-beacon check was taking the first packet off the wire and failing the test if it didn't match the expected DiscoveryDocument shape. A live probe surfaced a real-world failure mode: another MCOS instance on the LAN at `192.168.1.238` (host `PC-CREATION-R9-`) was broadcasting the legacy `BeaconAdvertisement` shape (`{browserPort, gatewayPort, ipAddress, platformGateways, ...}`), and the listener picked up its broadcast first.

v0.9.6 makes the beacon-listen loop continue receiving packets within the deadline window until it sees a DiscoveryDocument-shaped payload (`product` + `gateway.mcpUrl`). Foreign packets are counted and reported in the success message ("skipped N foreign packets"). The default deadline of 18 seconds — one full broadcast interval (15s) plus 3s buffer — accommodates this without a timeout false-alarm.

Also bumped the per-call HTTP timeout in the self-test from 5s to 20s. The first call after a service restart can take 1-3s due to PDH counter cold-init in `telemetryService_->captureSnapshot`; this surfaces transitively through the discovery doc and every onboarding profile. Pre-v0.9.6 the 5s timeout false-alarmed when running the self-test within ~10s of a service restart.

**Verified live:**
```
[?] UDP beacon broadcasts DiscoveryDocument within 18 seconds on port 7301  PASS
    received 1435-byte DiscoveryDocument from 192.168.1.7
    -- serverIp=192.168.1.7 gateway=http://192.168.1.7:8080/mcp (skipped 1 foreign packets)
```

Self-test final: 9 PASS / 0 FAIL / 1 SKIP (informational DNS-SD; Bonjour-not-installed).

## Risks and blockers

- **Watchdog mutex contention.** With 5+ pools and frequent operator API traffic, the 2-second watchdog tick acquires `mutex_` briefly (~few ms typically, longer if a respawn fires). Operator API calls during that window queue. Not observed as a problem in v0.9.6 testing; watch-item if pool count grows substantially or operators run high-frequency dashboard polling.
- **Cache-invalidation race window.** A LAN-client `tools/list` arriving in flight at the same instant as a `POST /api/pools` could observe the pre-change cache (the invalidation runs after the supervisor's `upsertPool` returns). Window is sub-millisecond; acceptable.
- **`mcpInitialized` flag is per-`ChildProcess`, reset on respawn.** When a watchdog-respawned worker starts up, the next `sendStdioJsonRpc` call performs the lazy initialize handshake again (correct behavior). If a misbehaving worker crashes repeatedly, every operator API call triggers the 3s handshake deadline. Watch-item: per-instance crash counter that opens a circuit breaker after N rapid failures.
- **Activity-ring telemetry events for death/respawn still not drained.** `WorkerSupervisor::deathLog_` / `respawnLog_` accumulate `DeathLogEntry` / `RespawnLogEntry` records; no consumer drains them. v0.9.7 candidate: wire `TelemetryAggregator*` into `WorkerSupervisor` constructor and emit `Telemetry::Pool::{Error,Info}` events.
- **The other MCOS instance on the LAN (192.168.1.238 / PC-CREATION-R9-).** Confirmed broadcasting the legacy beacon shape. Not a v0.9.6 problem (the self-test now handles it gracefully) but worth flagging: that instance is older than the gateway-first realignment and may surface in future LAN-discovery tests as similar-shape mismatches. If that host is operator-managed, a coordinated upgrade would be cleaner than skipping its packets.

## Ready for next phase?

Answer: **yes.** v0.9.6 leaves MCOS in a state where:

1. LAN AI clients reach the gateway and use 33 tools across 5 pools end-to-end.
2. Worker crashes (any cause: OOM, segfault, operator kill, OS reboot of just that process) are detected within 2 seconds and auto-recovered, regardless of whether any LAN client or operator is currently making requests.
3. Operator pool changes (register / remove / scale / drain) surface in LAN-client `tools/list` on the next request, with no TTL drift.
4. The discovery self-test is robust to a multi-host LAN environment.
5. Version bumps propagate automatically (verified live: VERSION.json 0.9.5 → 0.9.6 → CMake auto-reconfigure → MasterControlVersion.h regen → MASTERCONTROL_VERSION → serverInfo.version="0.9.6" on the live `initialize` handshake).

The next iteration (v0.9.7) should pick up the activity-ring wiring for death/respawn events, then start the architecture decision for the MCOS sub-agent runner. The existing live deployment is fully usable as-is; remaining work is incremental polish.
