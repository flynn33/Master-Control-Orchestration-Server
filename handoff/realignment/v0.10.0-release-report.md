# Release Completion Report — v0.10.0

Three operator-driven changes responding directly to the user's known-issues list:

> Remove "Docker" and "Playwright" MCP servers from MCOS.
> Cluade Code plugin is missing.
> The MCP Servers and Sub-Agents on the telemetry page MUST have smaller cards. They MUST be close in size to the current Sub-Agent Cards.

The third item shipped in v0.9.99 (compact mode now emits a single horizontal Grid row per card; ~28 px per card vs ~140 px pre-v0.9.99) but the operator was looking at a shell that had not been re-launched after the deploy. v0.10.0 ships a fresh shell binary so the v0.9.99 layout is loaded the next time the operator launches the shell. Items 1 and 2 are net-new code in v0.10.0.

## Scope completed

### 1. Docker Control + Playwright removed from the baseline catalog

`MasterControlDefaults.cpp` `baselineMcpCatalog` shrinks from 23 to 21 entries — the `docker-control` and `playwright` seeds are gone. The MCOS hard rule `.claude/rules/10-windows-native-cpp.md` already classed Docker as development-only, not the Windows production path; v0.10.0 makes the catalog match the rule.

`MasterControlBaselineToolsWorker/main.cpp` drops:

- the `Specialization::DockerControl` enum value
- the `dockerControlCatalog()` function
- the `case Specialization::DockerControl: return dockerControlCatalog();` line in the catalog switch
- the `docker.status` + `docker.list_containers` execution branch (the `if (gSpecialization == Specialization::DockerControl) { ... }` block in the tool dispatcher)
- the `else if (id == "docker-control") { gSpecialization = Specialization::DockerControl; ... }` ID-to-specialization mapping in the worker bootstrap

`MasterControlRuntime.cpp` drops the `playwright` entry from both runtime hint maps:

- `kKnownInstallCommand` (for the `installCommand` field on RuntimeStat)
- the canonical npm-package detection list (for `installPackageDetected`)

The "catalog placeholders" comment in `renderRuntimeStats` is stripped of its `playwright` example.

### 2. Startup migration prunes deprecated endpoint IDs from existing installs

A fresh-install removal from `buildDefaultSeededEndpointsForHost()` only affects fresh installs. Existing installs already carry `playwright` + `docker-control` in the persisted `seededEndpoints` array of `<ProgramData>\MasterControlOrchestrationServer\config\master-control-orchestration-server.json`, and the configuration loader normally replays whatever is on disk verbatim. Without migration the operator would have to manually edit the config JSON or clear ProgramData.

`FileBackedConfigurationService` constructor now runs a one-shot pass after loading the config: any element of `state_->configuration.activeProfile.seededEndpoints` whose `id` matches the deprecated list (`{"playwright", "docker-control"}`) is removed, and the cleaned config is persisted back to disk. The pass is idempotent — subsequent boots are no-ops once the config no longer contains those IDs. Verified live: after deploy the on-disk config has 0 matches for `playwright|docker`, the live `/api/pools` enumeration shows 0 pools matching those IDs, and the boot self-test reports 24 pool_handshake checks (down from 26 pre-v0.10.0).

### 3. Claude Code plugin (mcos-control) toggle now also flips ~/.claude/settings.json's enabledPlugins

Pre-v0.10.0 the toggle only created a directory junction at `<USERPROFILE>\.claude\plugins\mcos-control` pointing at the install's bundled plugin source. The junction is necessary but not sufficient — Claude Code only loads plugins whose key appears in `enabledPlugins` in `~/.claude/settings.json`. Operators who toggled the plugin on, restarted Claude Code, and looked for `/mcos-control:status` would find nothing because Claude Code never mounted the plugin.

v0.10.0 adds `setClaudeMcosPluginEnabled(profileDir, enabled)` which:

- reads the existing `~/.claude/settings.json` (creating an empty object if missing or unparseable)
- ensures `enabledPlugins` is an object
- on toggle ON: writes `enabledPlugins["mcos-control"] = true`
- on toggle OFF: removes the `enabledPlugins["mcos-control"]` key
- writes the merged JSON back with 2-space indent
- preserves all other keys in the file untouched

The helper is best-effort. Failure to write does not fail the toggle (the junction is the authoritative state) but is reported through new `claudeSettingsAttempted` / `claudeSettingsSucceeded` / `claudeSettingsLastError` fields on `/api/claude-plugin/toggle`'s response, plus `claudeSettingsEnabled` / `claudeSettingsReadError` on `/api/claude-plugin/status`. The UI status row can now show both "junction registered" and "Claude Code will load this plugin" honestly.

Verified live with a full toggle off → status → toggle on → status round-trip:

| Step | registered | claudeSettingsEnabled | claudeSettingsSucceeded |
|---|---|---|---|
| Toggle OFF | false | n/a | true |
| Status post-OFF | false | false | n/a |
| Toggle ON | true | n/a | true |
| Status post-ON | true | true | n/a |

Direct `Get-Content ~/.claude/settings.json` confirms `enabledPlugins.mcos-control` is `true` after toggle ON.

## Risks and blockers

The settings.json key form for the local mcos-control plugin is the bare `mcos-control` (no `@marketplace` suffix). All other entries in the operator's existing `enabledPlugins` map use `<name>@claude-plugins-official` because they came from the official marketplace. If a future Claude Code version requires a specific suffix for local-disk plugins (e.g. `mcos-control@local`), `setClaudeMcosPluginEnabled()` will need a corresponding update. The toggle remains operational either way because the junction is the authoritative state — the worst case is an unused settings.json entry.
