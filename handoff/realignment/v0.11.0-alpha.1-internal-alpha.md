## 3. Build pipeline used

```powershell
git checkout -b release/v0.11.0-alpha.1 main           # off 94cad65
# (edits to Build-Msi.ps1, VERSION.json, README.md, CHANGELOG.md)
powershell -ExecutionPolicy Bypass -File scripts\Sync-RepositoryVersionBadges.ps1
powershell -ExecutionPolicy Bypass -File scripts\Package-MasterControlOrchestrationServer.ps1 `
    -Preset release `
    -Version 0.11.0-alpha.1
```

The `Package-MasterControlOrchestrationServer.ps1` wrapper drives the full pipeline:

1. Enters the VS18 dev shell via `Launch-VsDevShell.ps1 -Arch amd64 -HostArch amd64 -SkipAutomaticLocation` so cmake/MSBuild/wix see the full Windows SDK toolchain.
2. `cmake --preset release` + `cmake --build --preset release`.
3. `ctest --test-dir build\release -C Release --output-on-failure`.
4. `cmake --install build\release --config Release --prefix build\_payload\MasterControlOrchestrationServer-v0.11.0-alpha.1-win-x64`.
5. Strips `*.pdb` from the stage; copies VC++ runtime DLLs from the toolchain redist dir.
6. Runs `MasterControlBootstrapper.exe preflight` against a throwaway target dir; writes the preflight JSON next to the bundle.
7. Emits `PACKAGE-METADATA.json`, `INSTALL.txt`, `START-HERE.txt`.
8. Invokes `installer\Build-Msi.ps1` which: harvests the stage into `installer\Fragments\Files.wxs` (one Component per file, explicit File Ids for the headline executables), then `wix build -arch x64 -ext WixToolset.UI.wixext -ext WixToolset.Util.wixext -d ProductVersion=0.11.0.1 ...` against the WiX sources.
9. Copies the produced MSI into the bundle dir.
10. Compresses the bundle dir into the `.zip` distribution.

## 6. MSI upgrade ordering caveat

`v0.11.0-alpha.1` carries MSI ProductVersion `0.11.0.1`. A future retail `v0.11.0` would carry `0.11.0.0`, which Windows Installer treats as **older** than `0.11.0.1`. This is intentional for an internal alpha that gets replaced by a `0.11.1` or `0.12.0` retail bump rather than by a same-band retail rebuild. If a same-band retail rebuild is required, it must bump to `0.11.0-rc.N` (giving `0.11.0.N` with N > 1) or `0.11.1.0` to upgrade cleanly over the alpha.

The retail track has no installations yet; this caveat is forward-looking and does not affect alpha deployment.

**Caveats acknowledged:**
- `Launch-VsDevShell.ps1` corrupts `$env:PATH` when invoked 3 times in succession from the same PowerShell process (the third call drops `System32` so `cmd.exe`, `vswhere.exe`, and `findstr.exe` become unfindable). The first packaging attempt failed at the `cmake --install` step for this reason after `cmake --build` + `ctest` had already invoked the dev shell twice. Workaround for this alpha: re-ran the packager with `-SkipBuild -SkipTests` so only the `cmake --install` invocation entered the dev shell (single PATH mutation, clean). The build artifacts produced by the first attempt's build phase (timestamps `15:15`–`15:16`) were re-used by `cmake --install`. This is a packaging-pipeline bug to triage in a follow-up — it does not affect the MSI's contents.
- The 2 `WIX1163` warnings about VBScript custom actions in `CustomActions.wxs` are carry-over from the v0.7.x packaging baseline and are not blocking. VBScript is deprecated by Microsoft but still bundled with every supported Windows host (Win10 22H2, Win11 23H2+, Server 2022). A follow-up will rewrite these as `BinaryRef` PowerShell actions.

## 9. Risks and blockers

- **No end-to-end client integration session has been observed since the last service restart.** `/api/clients` returns count 0 at alpha-cut time. The MCP protocol `tools/list` handshake works on loopback and LAN IP, but a real ChatGPT / Claude Code / Codex / Grok session driving a `tools/call` round-trip has not been observed in this runtime instance. This is the alpha-validation step that only the operator can drive.
- **DNS-SD live presence is advertised but not peer-confirmed.** `/api/discovery` advertises `dns-sd` + `udp-beacon` capabilities; `Resolve-DnsName _mcos-mcp._tcp.local PTR` is not supported from PowerShell on this host. Validate from a macOS peer with `dns-sd -B _mcos-mcp._tcp` or a Bonjour Browser on another LAN host before declaring discovery green.
- **LAN-trust application-layer auth is intentionally absent.** Per the operator's standing directive, app-layer auth lands only in the retail build. Alpha must run on the secure LAN.
- **PHASE-14 Slices B–E are deferred.** None gate alpha — they're UX polish over routes that already work.

## 10. Operator-driven alpha test plan

1. Run the MSI: `msiexec /i dist\packages\release\MasterControlOrchestrationServer-v0.11.0-alpha.1-win-x64.msi /qn /l*v install.log`.
2. Confirm the service starts and `GET http://<host>:7300/api/version` returns `0.11.0-alpha.1`.
3. From a LAN peer machine, fetch the onboarding profile for the target client type:
   - ChatGPT: `GET http://<host>:7300/api/onboarding/chatgpt`
   - Claude Code: `GET http://<host>:7300/api/onboarding/claude-code`
   - Codex: `GET http://<host>:7300/api/onboarding/codex`
   - Grok: `GET http://<host>:7300/api/onboarding/grok`
   - Generic: `GET http://<host>:7300/api/onboarding/generic`
4. Drop the returned `.mcp.json` snippet into the client's MCP config; restart the client.
5. Inside the client, run `tools/list` and confirm 126 tools appear; then call `baseline-tools__mcos.echo` for a no-dep round-trip.
6. Back on the host, `GET /api/clients` should now show the connected client with a fresh `lastSeenUtc`.
7. From a second LAN host, run `dns-sd -B _mcos-mcp._tcp` (macOS) or `bonjour-browser` (Windows) and confirm an `MCOS` advertisement appears.

Items 1–2 are alpha smoke-test. Items 3–7 are the alpha-acceptance test.
