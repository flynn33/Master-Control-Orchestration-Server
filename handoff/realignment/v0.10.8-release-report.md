# Release Completion Report — v0.10.8

Corrective release driven by an external supervisor-candidate connection-test report. ChatGPT runtime attempted to assume the MCOS supervisor role via the supplied `mcos-supervisor-chatgpt.config.json` and reported `not_connected` against all five tested URLs. The report (`MCOS_Supervisor_Connection_Test_Report.md`, `connection_test_summary.json`, `sanitized_config_snapshot.json`) is preserved in `C:\Users\Flynn\Downloads\MCOS_Supervisor_Connection_Test_Report\`.

## 1. What the report claimed

| Tested URL | Result |
|---|---|
| `GET http://127.0.0.1:7300/api/health` | connection_refused |
| `GET/POST http://127.0.0.1:7300/mcp` | connection_refused |
| `GET http://192.168.1.7:7300/api/health` | connection_refused |
| `GET http://192.168.1.7:7300/.well-known/mcos.json` | connection_refused |
| `GET/POST http://192.168.1.7:7300/mcp` | connection_refused |

Most-likely-failure hypothesis in the report: "MCOS is bound only to localhost / not listening / HTTP.sys URL ACL missing LAN binding."

## 2. What the local diagnostics actually show

Ran the suggested diagnostics on the MCOS host:

| Probe | Result |
|---|---|
| `netstat -ano \| findstr :7300` | `TCP 0.0.0.0:7300 LISTENING` (admin/browser API, PID 18008) |
| `netstat -ano \| findstr :8080` | `TCP 0.0.0.0:8080 LISTENING` (native MCP gateway, PID 4) |
| `GET http://127.0.0.1:7300/api/health` | 200 |
| `GET http://192.168.1.7:7300/api/health` | 200 |
| `GET http://127.0.0.1:7300/.well-known/mcos.json` | 200 |
| `GET http://127.0.0.1:7300/mcp` | **404 Not Found** |
| `GET http://127.0.0.1:8080/mcp` | 204 No Content (correct -- GET on gateway is a stream-open) |
| `/api/dashboard.mcpGatewayStatus.mcpUrl` | `http://0.0.0.0:8080/mcp` |

The "MCOS only listens on localhost" hypothesis is wrong. MCOS IS listening on `0.0.0.0:7300` (admin) AND `0.0.0.0:8080` (gateway). The host can hit `192.168.1.7:7300/api/health` and get a 200. The ChatGPT runtime's `connection_refused` is consistent with cloud-hosted ChatGPT runtime being unable to reach a private RFC-1918 LAN address — orthogonal to MCOS configuration.

What IS broken: the supplied config's `server.mcpEndpoint` is `http://127.0.0.1:7300/mcp`. Two real bugs:

1. **Wrong port.** `/mcp` lives on the native HTTP.sys gateway listener at `cfg.mcpGateway.listenPort` (default 8080), not on the admin/browser port. `:7300/mcp` returns 404 because the admin server has no `/mcp` route.
2. **Wrong host.** `127.0.0.1` is unreachable from any off-box supervisor client (LAN or cloud). The well-known discovery doc on the SAME host already advertises `http://192.168.1.7:8080/mcp` via `DiscoveryService::currentDocument()`'s 5-step LAN IP precedence chain. The supervisor config is the only place still emitting loopback.

## 3. Fix

### New plumbing

* **`ISupervisorAssignmentService::setEndpoints(mcpEndpoint, discoveryEndpoint, fingerprintSeed)`** — pure virtual; impl mutates the cached context under the existing mutex. Only non-empty inputs overwrite, so callers can refresh just the field that changed.
* **`MasterControlApplication::Impl::refreshSupervisorEndpoints()`** — mirrors `DiscoveryService::currentDocument()`'s 5-step LAN-IP precedence chain (bindAddress when not wildcard → preferredBindAddress → captured `primaryIpAddress` → bindAddress-literal fallback → 127.0.0.1). Composes `mcpEndpoint = http://<lanIp>:<cfg.mcpGateway.listenPort><cfg.mcpGateway.mcpPath>` and `discoveryEndpoint = http://<lanIp>:<cfg.browserPort>/.well-known/mcos.json`, then pushes both plus a refreshed fingerprint seed into the supervisor service.

### Call-sites that refresh before issuing a config

* `POST /api/supervisor/assignment/select` (line 14699 — `selectAndIssue`)
* `POST /api/supervisor/config/generate` with `providerId` (line 14758 — `selectAndIssue`)
* `POST /api/supervisor/config/generate` without `providerId` (line 14791 — `regenerateConfig`)

`POST /api/supervisor/connect/confirm` does NOT need a refresh because confirm validates the configId against the active assignment, not the endpoint URL.

### Initialize-time fallback also fixed

The supervisor service is constructed in `MasterControlApplication::Impl::initialize()` BEFORE the snapshot tick has run, so it can't see `primaryIpAddress`. Pre-v0.10.8 the constructor used `adminBase + "/mcp"` for both ports' worth of wrongness. v0.10.8 composes the initialize-time mcpEndpoint from `cfg.mcpGateway.listenPort + cfg.mcpGateway.mcpPath` — port is correct from boot, host stays loopback until the first refresh.

## 5. Public contracts changed

The `server.mcpEndpoint` value in newly issued supervisor configs changes:

| | Before v0.10.8 | After v0.10.8 |
|---|---|---|
| Host | `127.0.0.1` | LAN-routable IPv4 (e.g. `192.168.1.7`) |
| Port | `cfg.browserPort` (e.g. 7300) | `cfg.mcpGateway.listenPort` (e.g. 8080) |
| Path | `/mcp` | `cfg.mcpGateway.mcpPath` (e.g. `/mcp`) |

Existing on-disk configs from v0.10.7 and earlier carry the wrong URL but remain otherwise valid; operators can regenerate via the wizard's "Generate Config & Save" or `POST /api/supervisor/config/generate`.

`server.discoveryEndpoint` keeps the same path (`/.well-known/mcos.json`) and same port (`cfg.browserPort`), but the host changes from `127.0.0.1` to the LAN-routable IPv4.

## 7. Operator action

After deploy:

1. Open the Supervisor Agent Assignment Wizard on the Overview deck.
2. Click "Revoke Active" if a stale CFG-* assignment is bound to the old URL.
3. Select ChatGPT (or whichever provider) and click "Generate Config & Save".
4. The newly saved JSON's `server.mcpEndpoint` should read `http://192.168.1.7:8080/mcp` (NOT `127.0.0.1:7300/mcp`).
5. For a remote / cloud-hosted ChatGPT runtime, a LAN tunnel (Tailscale, ngrok, Cloudflare Tunnel, similar) is still required because cloud runtimes cannot reach `192.168.1.7` directly. That's an operator-side networking concern, not an MCOS bug.
