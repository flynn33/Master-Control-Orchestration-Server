# Release Completion Report — v0.9.62

Two more in-tree backends following the v0.9.61 multi-spec pattern: `local-git` (`git.run` wrapping `git.exe` via CreateProcessW + pipes) and `file-search` (`search.grep` using ripgrep when on PATH, with a PowerShell `Select-String` fallback when it isn't). Reachable count goes 8/28 → 10/28; gateway `tools/list` grows from 34 to 36.

## Scope completed

### `local-git` specialization

* New `Specialization::LocalGit` enum value; CLI parser recognizes `--specialization=local-git`.
* `gitRun(args, cwd, timeoutMs)` builds a quoted git command line via the new `quoteArg()` helper (CommandLineToArgvW-compatible escaping for whitespace, embedded quotes, and trailing backslashes), then defers to `runProcessCaptured`. Returns `{args, cwd, exitCode, timedOut, stdout, stderr, timeoutMs}` (+ `launchFailed`/`launchError` when CreateProcessW fails so "git not installed" surfaces honestly rather than as a fake zero-result).
* Default 60s timeout; min 100ms, max 5 min; 1 MB stream caps.

### `file-search` specialization

* New `Specialization::FileSearch` enum value; CLI parser recognizes `--specialization=file-search`.
* `searchGrep(pattern, path, glob, maxMatches, timeoutMs)` tries `rg --json --max-count <N>` first. Parses ripgrep's line-delimited JSON match envelopes into `{file, line, text}` triples.
* Falls back to PowerShell `Get-ChildItem -Recurse | Select-String -Pattern <pattern>` when `rg` isn't on PATH (rg launchFailed). The fallback emits pipe-delimited `<file>|<line>|<text>` records that the worker parses into the same `{file, line, text}` shape.
* Returns `{tool, ranFallback, pattern, path, glob, maxMatches, matchCount, matches[]}` so consumers can tell which backend served the call. Default `maxMatches=200`, max 5000.

### `quoteArg()` shared helper

Win32 single-arg quoting following the `CommandLineToArgvW` rules: wraps the arg in double quotes if it contains whitespace, doubles trailing backslashes that precede an embedded quote, and escapes embedded quotes as `\"`. Used by `gitRun` and `searchGrep` (and any future specialization that needs to pass a structured arg list through the single-string `CreateProcessW` API).

### Auto-registered pools

`MasterControlRuntime` upserts both new pools at boot — same `mcos-baseline-tools-worker.exe` binary, distinct `--specialization` arg, `minInstances=1` so they spawn immediately. Pool ids match the seeded catalog ids (`local-git`, `file-search`) so the v0.9.56 install-state classifier flips them from `awaiting_pool_registration` to `installed_and_supervised` automatically.

## Risks and blockers

* **Search regex semantics differ between rg and PowerShell fallback.** Operators with patterns containing regex metacharacters get different behavior depending on which backend ran. Documented in `ranFallback`. v0.9.63 could add `literal=true` mode that escapes metacharacters in both paths.
* **Worker process count doubles per added specialization.** Each pool spawns its own worker.exe instance under Job Object supervision. Three specializations now means three processes (plus baseline-tools) for ~12 MB resident each. Acceptable for ~25 specializations; revisit if memory becomes a constraint.
