# Release Completion Report — v0.9.27

Maintenance release. Browser/cross-origin client fix found by the v0.9.27 bug-hunt.

## Scope completed

### CORS preflight + headers on every response

Pre-v0.9.27 a browser-based client (e.g. a JS dashboard hosted on a different origin or `file://`, a future WinUI WebView2 panel served from `ms-appx-web://`, or an external diagnostic tool like Insomnia/Hoppscotch running in the page) could not call `/api/*` because:

- `OPTIONS` returned 404 (no handler in the route table).
- Successful `GET`/`POST` responses had no `Access-Control-Allow-*` headers, so the browser blocked the response from JS even when the server happily processed it.

Per the [Fetch CORS spec](https://fetch.spec.whatwg.org/#cors-protocol) a non-simple cross-origin request requires a successful preflight before the actual request fires. Fix:

- `SimpleHttpServer::handleClient` short-circuits `OPTIONS` to a 204 response before route dispatch.
- `SimpleHttpServer::sendResponse` always emits `Access-Control-Allow-Origin: *`, `Access-Control-Allow-Methods: GET, HEAD, POST, OPTIONS`, `Access-Control-Allow-Headers: Content-Type, X-MCOS-Client-Id, X-MCOS-Client-Type`, and `Access-Control-Max-Age: 86400` (one-day preflight cache).

The wildcard origin is consistent with the LAN-trust model in [ADR-001](../../docs/implementation/adr/0001-lan-trust-model.md): MCOS does not authenticate AI clients at the application layer, trust is enforced at the network level, and operator surfaces remain on a separate logical port (7300) from the AI-client gateway (8080).

**Verified live**:

```
=== OPTIONS preflight ===
  status: 204
  Access-Control-Allow-Origin: *
  Access-Control-Allow-Methods: GET, HEAD, POST, OPTIONS
  Access-Control-Allow-Headers: Content-Type, X-MCOS-Client-Id, X-MCOS-Client-Type

=== CORS headers on a regular GET ===
  status: 200
  Access-Control-Allow-Origin: *
```

### Inventory cleanup

The bug-hunt found two stale test pools (`test`, `x`) that had been registered during a prior probe and never removed. They held no instances and produced no traffic, but they cluttered `/api/pools` and the dashboard pool grid. Removed via `POST /api/pools/{id}/remove`. Pool count back to 5 supervised production pools (`baseline-tools`, `memory`, `sequential-thinking`, `filesystem`, `sqlite`).
