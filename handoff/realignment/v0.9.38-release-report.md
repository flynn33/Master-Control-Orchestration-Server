# Release Completion Report — v0.9.38

Maintenance release. Runtime registration error semantics: 500 → 400 on bad input, 400 → 404 on not-found.

## Scope completed

### Runtime registration: bad enum returns 400, not 500

The v0.9.38 bug-hunt sent `POST /api/runtime/mcp-servers` with `kind: "mcp-server"` (hyphen) instead of the expected `kind: "mcp_server"` (underscore). Pre-v0.9.38 result: `500 Internal Server Error` with body:

```json
{ "message": "Unknown enum string: mcp-server", "succeeded": false }
```

500 is the wrong status — that's reserved for server bugs (uncaught exceptions, unexpected internal state). A client sending the wrong enum spelling is a 400 by RFC 7231 §6.5.1. Root cause: `endpointKindFromString` throws `std::invalid_argument` on unknown values, the throw bubbled past the route's local handling and hit the outer `} catch (const std::exception&) { return 500; }` at the end of the handler lambda. Same throw shape applies to the subagents route.

The fix wraps both upsert call sites in try/catch returning 400 with the diagnostic body:

```cpp
try {
    const auto result = adminApiService_->upsertMcpServerJson(request.body);
    return jsonResponse(result, result.succeeded ? 200 : 400);
} catch (const std::exception& ex) {
    return jsonResponse(OperationResult{ false, false,
        std::string("Invalid runtime MCP server payload: ") + ex.what() }, 400);
}
```

### Runtime registration: remove of unknown id returns 404, not 400

Same v0.9.30/v0.9.33/v0.9.37 pattern. `POST /api/runtime/mcp-servers/remove` and `POST /api/runtime/subagents/remove` returned 400 when the target id wasn't registered; now 404 (RFC 7231 §6.5.4) for the missing-resource case, 400 reserved for parse/validation errors.

```cpp
const int status = (result.message == "The requested MCP server was not found.") ? 404 : 400;
```
