# Release Completion Report — v0.9.85

Continuation of the v0.9.84 wildcard-bind URL-leak hunt: two more routes patched.

## Scope completed

### Bugs found during v0.9.84 post-deploy probe

v0.9.84 fixed `/api/health/summary`. A wider probe of every URL-emitting admin route found two more leaks:

* `/api/gateway/status` returns the raw `GatewayStatus::mcpUrl` (= `http://0.0.0.0:8080/mcp`) plus a `status.message` line that embeds the same wildcard URL ("the in-process HTTP.sys adapter listening on http://0.0.0.0:8080/mcp...").
* `/api/gateway/health` does the same with `GatewayHealth::mcpUrl` + `health.message`.

These are direct serializations of the adapter's internal state. The native HTTP.sys adapter binds to `0.0.0.0` so HTTP.sys listens on every interface, but the externally-visible URL in the response should be a routable LAN IP for clients that consume these routes (e.g. an external supervisor that polls `/api/gateway/health` to verify connectivity).

### Fix

Same substitution chain (bindAddress → activeProfile.preferredBindAddress → 127.0.0.1) applied at both routes. Status / health structs are still emitted via `nlohmann::json`'s natural serialization, but `mcpUrl` and `message` get rewritten before serialization. An additional `mcpUrlRaw` field is appended to the response body so an operator UI can render both the listening value and the resolved value side by side.

The substitution helper walks both URL-prefix-style matches (`://0.0.0.0:`) and narrative-text-style matches (`http://0.0.0.0` anywhere in the message string), so multi-line operator status text gets the substitution too.

Re-probed every URL-emitting route after deploy: `/api/discovery`, `/.well-known/mcos.json`, `/api/onboarding/{claude-code,chatgpt,grok,generic}`, `/api/gateway/{status,health,tools}`, `/api/health/summary`. The `mcpUrl` field is now a LAN-routable URL on every one. The `mcpUrlRaw` field exists only on the three gateway-state routes (where the operator may need to see both); the discovery + onboarding paths don't need the diagnostic split because they only ever surface resolved values to begin with.

## Public contracts changed

* `/api/gateway/status` response gains an `mcpUrlRaw` field. `mcpUrl` value changes from raw wildcard URL to resolved LAN IP URL. `message` value changes wildcard URL substrings to resolved LAN IP.
* `/api/gateway/health` response gains an `mcpUrlRaw` field. Same value changes for `mcpUrl` and `message`.

No schema is published for either route, so the additions aren't schema-breaking.
