# Release Completion Report — v0.9.15

Maintenance release. Adds the background async reachability prober planned in v0.9.14's deferred list.

## Scope completed

### Background async reachability prober

`AdminApiService` now starts a background `std::thread` on construction (`StartReachabilityProber`) that walks `inventoryService_->listEndpoints()` every 10 seconds and re-probes each endpoint via the existing `probeReachabilityCached` helper, refreshing cache entries before their 30s TTL expires. Same interruptible-cv shutdown pattern as BeaconService and the supervisor watchdog so the destructor stops the prober promptly.

The intent was to eliminate the periodic 6-20s spike that v0.9.14's TTL cache left in place when entries expired. **Partial success**: cache-hit calls drop to 32-34ms (3 of 10 measured calls), but several calls still landed 5-19 seconds. Likely cause: the snapshot path probes endpoints the prober doesn't walk (a different filter on the inventory; the snapshot iterates `snapshot.endpoints` from a different service path that may include entries our prober misses). Tracking as a v0.9.16 root-cause issue.

Live measurement (10 calls over ~50 seconds, 5-second gap between calls):
- Call 1: 19001ms (cold start, prober had 15s settle)
- Call 2: 32ms ✓
- Call 3: 13538ms (likely prober/snapshot endpoint mismatch)
- Call 4: 5154ms
- Call 5: 13523ms
- Call 6: 34ms ✓
- Call 7: 5151ms
- Call 8: 13508ms
- Call 9: 31ms ✓

3 of 10 calls served from cache (32-34ms); 7 of 10 still showed multi-second probe time. **Improvement over v0.9.14**: the median is lower, the floor is much lower, but the periodic spikes are not yet eliminated. **Improvement over v0.9.13** (6.6s every call): yes, every call.
