# Release Completion Report — v0.9.77

Browser dashboard parity for the Supervisor Agent Assignment Wizard shipped in v0.9.76, plus operator-visible documentation on the CMake-install vs live-service-path mismatch that caused the "the server will not even start" report after v0.9.76 went out.

## Scope completed

### 1. Browser dashboard Supervisor Agent parity

The WinUI Shell shipped the Supervisor Agent card in v0.9.76. The browser dashboard was deferred. v0.9.77 closes the gap: the browser surface now carries the same single-selection ChatGPT / Claude / Grok toggle group, Generate Config & Save flow, and Revoke Active button — all driven off the same `/api/supervisor/*` routes the Shell uses.

**JS (`resources/web/app.js`):**

* `refreshAll()` parallel fetch picks up `/api/supervisor/status` alongside `/api/self-tests`. Stored as `state.supervisorStatus` so every refresh keeps the panel live.
* `renderSupervisorPanel()` renders the panel on the Overview deck right below the Self-Test panel. Status row reflects the six-state lifecycle (`off` / `config_generated` / `pending_connection` / `connected` / `disconnected` / `revoked` / `error`) with tone-coded headlines that match the shell's color rules. Three `<input type="radio" name="supervisor-provider">` rows for chatgpt / claude / grok with provider-specific recommendation text mirroring `MCOS_Supervisor_Agent_Assignment_Wizard_Package/docs/UI_UX_SPEC.md`.
* `generateSupervisorConfig(providerId)` POSTs `/api/supervisor/config/generate`, validates the envelope, then triggers a browser save-as via `URL.createObjectURL(new Blob([...]))` + an `<a download="mcos-supervisor-{provider}.config.json">.click()`. Cleans up the object URL after a tick. Falls back to `refreshAll()` on every error path.
* `revokeSupervisor()` POSTs `/api/supervisor/assignment/revoke` with a `Browser dashboard Revoke Active button.` reason and refreshes.
* `bindOverviewHandlers()` wires the three radios (enable/disable Generate Config button when none selected), the Generate Config button, and the Revoke Active button.

**CSS (`resources/web/styles.css`):**

* `.supervisor-panel` / `.supervisor-rows` / `.supervisor-row` / `.supervisor-row-body` / `.supervisor-row-name` / `.supervisor-row-desc` / `.supervisor-actions` / `.supervisor-status` matching the Tron CLU red-orange accent palette used by other dashboard panels. Radio accent color is the same `#ff3d2e` accent the shell card uses on its toggle border.

No new public contract: every byte the browser dashboard talks to was already exposed in v0.9.76. The browser surface is purely additive — operators who only use the shell continue to work, and remote-desktop users who happened to hit the browser dashboard now have feature parity.

### 2. Install-path mismatch documented and operator-fixed

The operator reported "the server will not even start" between the v0.9.76 release report and follow-up activity. Investigation:

* `cmake --install` writes its staged tree to `C:\Program Files\MasterControlOrchestrationServer\` (no spaces). That's the CMake default install prefix derived from `project(MasterControlOrchestrationServer ...)`.
* The Windows service binary path — recorded in the SCM by the WiX MSI installer (`MasterControlOrchestrationServerSetup.exe`) — is `C:\Program Files\Master Control Orchestration Server\MasterControlServiceHost.exe` (with spaces).
* The two paths do not auto-reconcile. `cmake --install` does not update the live-service location; it only updates the staging tree the MSI consumes.
* Result: after the v0.9.76 build and `cmake --install`, the new `MasterControlServiceHost.exe` lived in the no-spaces tree but the *running* service was still the old binary in the spaces tree. The runtime smoke test in the v0.9.76 release report passed because I manually copied the new service binary into the spaces path before the smoke. The `MasterControlShell.exe` in the spaces path was *not* manually copied. The operator launched the spaces-path shell (10:32 AM build, pre-session) and it crashed with `0xc0000005` at fault offset `0x...8` — the same crash class the v0.9.72 WndProc-subclass attempt produced, recorded in the application event log at 18:59:31 PM 2026-05-10.

v0.9.77 resolution:

* Copied the freshly-built `MasterControlServiceHost.exe`, `MasterControlShell.exe`, `MasterControlShell.pdb`, `app.js`, and `styles.css` into the spaces-path live install. The service was restarted; the shell was relaunched. Both are running cleanly under v0.9.77.
* Released the deferred work item as a known issue: the canonical deployment path is `MasterControlOrchestrationServerSetup.exe`. For hot-patching during development, the new binaries must be hand-copied from `<build>/src/MasterControl*/Debug/*.exe` and `<repo>/resources/web/*` into `C:\Program Files\Master Control Orchestration Server\` and `share\MasterControlOrchestrationServer\web\` respectively. A `scripts/Deploy-LocalLive.ps1` helper is on the v0.9.78 deferred list.
