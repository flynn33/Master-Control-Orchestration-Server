# Release Completion Report — v0.9.41

Maintenance release. Three install routes returned 500 on bad bodies; v0.9.41 makes them return 400.

## Scope completed

### `POST /api/install/{package,repo,zip}` empty/malformed body returns 400

The v0.9.41 bug-hunt exercised every POST route's empty-body handling. The runtime registration routes (v0.9.38) and client routes (v0.9.37) all returned 400 cleanly, but the three install routes returned **500**:

```
POST /api/install/package  (empty body)  -> 500
POST /api/install/repo     (empty body)  -> 500
POST /api/install/zip      (empty body)  -> 500
```

Root cause: `installPackageJson`, `installRepoJson`, and `installZipJson` all run `nlohmann::json::parse(requestBody).get<Spec>()` inline with no try/catch. An empty body throws `json::parse_error`; the throw bubbles past the route's local 200/400 branch and hits the outer 500 handler at the bottom of the request lambda. Same shape as the v0.9.38 fix for `/api/runtime/{mcp-servers,subagents}` — a malformed body is a 400 by RFC 7231 §6.5.1, not 500.

Each route gained a try/catch returning 400 with the diagnostic body:

```cpp
try {
    const auto result = adminApiService_->installPackageJson(request.body);
    return jsonResponse(result, result.succeeded ? 200 : 400);
} catch (const std::exception& ex) {
    return jsonResponse(OperationResult{ false, false,
        std::string("Invalid InstallerPackageSpec payload: ") + ex.what() }, 400);
}
```
