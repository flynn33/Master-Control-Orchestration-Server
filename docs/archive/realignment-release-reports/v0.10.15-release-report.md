# Release Completion Report — v0.10.15

Operator-facing silent-ignore defect fix on `/api/activity`, `/api/telemetry/events`, and `/api/client/activity`. Pre-v0.10.15 the route layer only accepted one canonical query-param name per concern (`max=` for the count cap, `since=` for the watermark, `kind=` for the event-kind filter); operators typing a natural-language alias such as `?limit=N` got the full 512-event ring (138 KB) with no error, no log line, and no warning header. v0.10.15 ships a single boundary-aware helper, refactors the three handlers, and grows `/api/client/activity` to mirror `/api/activity`'s pagination surface. No public-API contract is broken; the change is strictly additive.

## 4. Public contracts changed

- **Query-param surface (additive only).** Every canonical wire name continues to behave exactly as before. New aliases are accepted on the same three routes. No alias is documented as wire-stable yet — callers should still prefer the canonical name. The `kindFilter` echoes back the canonical kind name + the original wildcard suffix as before.
- **`/api/client/activity` query surface (new).** Previously the route ignored the query string entirely. It now accepts the same `max=` / `since=` / `kind=` plus aliases that `/api/activity` exposes. Callers that previously sent no query string get unchanged behaviour (full ring).
- **No removed or renamed HTTP routes, no removed or renamed Shell controls.**

## 6. Risks and blockers

- **Bare `query.find()` patterns may still exist elsewhere in the runtime.** The debugger sub-agent's Section A only identified two routes that parse query params today (`/api/activity` and `/api/telemetry/events`). Future routes that copy the old `query.find("name=")` pattern instead of calling `extractQueryParam` would reintroduce the same alias gap. A `clang-tidy` rule or pre-commit grep is a candidate follow-up (low priority — the helper is now the obvious path).
- **Alias documentation.** The wire aliases are accepted but the existing operator docs only mention canonical names. Operators who already know the canonical names see no behaviour change; the alias acceptance is silent backward-compatible. A doc update is a candidate follow-up (low priority).
- **SSE resume and POST unknown-key validation are deferred** to a follow-up patch (captured in v0.10.15 `knownIssuesDeferredToNextRelease`). Neither blocks v0.10.15 because both are read-stream / model-tolerance behaviours that current clients already cope with.
