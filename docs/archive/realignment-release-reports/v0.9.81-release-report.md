# Release Completion Report — v0.9.81

Security hardening on the supervisor connect/confirm capability validator.

## Scope completed

### Default-deny on empty allowed-capability set

Bug-hunt during the v0.9.81 iteration found a subtle short-circuit in `SupervisorAssignmentServiceImpl::confirmConnection`:

```cpp
// Pre-v0.9.81:
if (!allowedSet.empty() && allowedSet.count(cap) == 0) { reject }
```

When the assignment's mode is one whose `capabilitiesForMode()` returns an empty list — `SupervisorMode::Disabled` falls through to the empty fallback at the end of the helper — the `!allowedSet.empty()` guard skipped the allowed-list check entirely. The only remaining gate was the forbidden list, which means a Disabled-mode assignment would accept any capability that wasn't on the forbidden list (about a dozen entries). That violates the default-deny contract specified in `MCOS_Supervisor_Agent_Assignment_Wizard_Package/docs/SECURITY_AND_CAPABILITY_MODEL.md`:

> "MCOS must enforce permissions server-side. Do not rely on UI hiding or model self-restraint."

The Wizard surface itself never produces a Disabled-mode assignment (the parser rejects `mode=disabled` from POST bodies, and `selectAndIssue` upgrades `Disabled → AutonomousSupervisor` defensively). But the persisted assignment record carries the mode field directly; an out-of-band tamper or a future code path that lands on Disabled mode would expose the bypass. Hardening it now closes the gap.

v0.9.81:

```cpp
if (allowedSet.count(cap) == 0) { reject with "Capability not allowed in current mode (<mode>): <cap>" }
```

Empty allowed set now rejects every claim. The forbidden-list check still runs first so its specific error message is preferred when both gates would fire.

### Regression test

`testSupervisorConfirmDefaultDenyOnEmptyAllowedSet` exercises the gap directly:

1. Construct an autonomous_supervisor assignment via `selectAndIssue`.
2. Tamper the persisted `supervisor-assignment.json` to set `mode=disabled` and `allowedCapabilities=[]`.
3. Rehydrate a fresh service against the tampered file.
4. Confirm a benign single-capability claim (`supervisor.get_context`) — must be rejected with the not-allowed error path.

Pre-v0.9.81 this test would have passed the claim. v0.9.81 rejects it. Test now lives in the main backend test executable, run as part of every `ctest --preset debug`.

## Public contracts changed

None. The Wizard surface still produces autonomous_supervisor mode by default; that mode's allowed-capability set is unchanged. The happy path is unchanged. The change only tightens the previously-accidentally-permissive fallback when an external actor manages to set `mode=disabled` on the assignment.
