# Release Completion Report — v0.9.4

This is a maintenance release in the same series as v0.9.3 (operator-directed defect cleanup), not a numbered realignment phase. It closes the v0.9.3 deferred item ("Operator-bound external workers for the 23 named pools land in v0.9.x+") and seven MCP-protocol-correctness gaps that surfaced while probing MCOS the way Claude Code, Codex, and Grok actually touch it.

## Scope completed

1. **MCP notifications correctness** — `notifications/initialized` and `notifications/cancelled` (any JSON-RPC request without an `id` field) now get `HTTP 204 No Content` from the native gateway instead of a `-32601` envelope. Required by JSON-RPC 2.0 §4.1; strict MCP clients (TypeScript / Python SDKs) treated the prior behavior as a protocol violation.
2. **`tools/list` schema passthrough** — child-reported `inputSchema` flows from worker → gateway → LAN client verbatim. `McpToolDescriptor` gained `inputSchemaJson`. Pre-v0.9.4 every tool got an open `{"type":"object"}` placeholder.
3. **Protocol version alignment** — `initialize` advertises `protocolVersion: "2025-03-26"` (Streamable HTTP transport) instead of the stale `2024-11-05` literal, in both the gateway and the in-tree baseline-tools worker. Now matches the DNS-SD `protovers` TXT field and the transport every onboarding profile recommends.
4. **Host telemetry surface** — new `GET /api/host/telemetry` returning the existing `HostTelemetrySnapshot` (CPU / memory / disk / network plus host identity). Honors ADR-002 §9 honest-telemetry: empty `capturedAtUtc` means the sampler hasn't run yet.
5. **Discovery surfaces admin endpoints** — `DiscoveryAdmin` block listing pools, clients, activity, host telemetry, gateway status, gateway tools, and operator health URLs. New capability `mcp-2025-03-26` signals protocol-revision parity.
6. **Self-handshaking stdio bridge** — `WorkerSupervisor::ChildProcess` gained `mcpInitialized`. `sendStdioJsonRpc` performs a lazy `initialize` → `notifications/initialized` MCP handshake on the first call per instance with a 3-second deadline. Pre-v0.9.4 the supervisor sent `tools/list` without the MCP session handshake; spec-compliant servers refused, and the gateway's catalog refresh silently dropped them. (The in-tree baseline-tools worker was permissive, which masked the gap.)
7. **23-server catalog goes live, 4 pools registered** — `memory`, `sequential-thinking`, `filesystem`, `sqlite` installed globally via `npm install -g`, then registered through `POST /api/pools` with `node.exe` spawning the resolved `dist/index.js` directly. **Not via `npx`** — `npx` is a Windows shim that spawns the actual MCP server as an orphaned grandchild whose pid the supervisor cannot track and whose stdio MCOS cannot reach. Direct entry-point spawn keeps the MCP server as a direct child under the Job Object with stdio wired straight to MCOS.
8. **Workflow tooling** — 14 generic sub-agents under `.claude/agents/` (architect, sentinel, planner, reviewer, debugger, test-writer, refactorer, documenter, performance-engineer, security-auditor, git-manager, build-resolver, coordinator, multi-file-specialist) complementing the 10 existing `mcos-*` agents. Three standard MCPs added to `.mcp.json` (sequential-thinking, filesystem, sqlite) so Claude Code sessions in this repo have parity with what MCOS now exposes through the gateway. Index at `.claude/ORCHESTRATION-SETUP.md`.

## Public contracts changed

- `DiscoveryDocument` JSON now has an `admin` member (additive; backward-compatible — existing consumers ignore unknown fields per nlohmann's `NON_INTRUSIVE_WITH_DEFAULT`).
- `DiscoveryDocument.capabilities` now includes `mcp-2025-03-26` (additive).
- `tools/list` response items now carry meaningful `inputSchema` objects with `properties`/`required`/argument types instead of a bare `{"type":"object"}` placeholder. Strict consumers that previously short-circuited on the placeholder will now see the typed schema; this is forward-compatible.
- `initialize` response advertises `protocolVersion: "2025-03-26"` instead of `"2024-11-05"`. Clients that pinned to `2024-11-05` will renegotiate; the wire format is back-compatible at the JSON-RPC envelope level.
- New `GET /api/host/telemetry` endpoint (additive).
- Notifications (JSON-RPC requests without `id`) get HTTP 204 instead of HTTP 200 with a `-32601` envelope. This is a behavior change but a spec-compliance fix; clients that relied on the prior behavior were already in violation.

## Risks and blockers

- **Per-pool tools/list cost** — every gateway `tools/list` call triggers `refreshToolCatalogLocked`, which sends `tools/list` to each pool's first ready instance. With 5 pools that's 5 stdio round-trips per LAN-client request. Acceptable today; would benefit from a TTL cache (30 s) with manual invalidate on pool upsert/scale. v0.9.5 candidate.
- **`mcpInitialized` is per-`ChildProcess`, not per pool** — if an instance crashes and the supervisor spawns a new one, the new ChildProcess record's flag starts false, so the next call re-handshakes. Correct behavior. Edge case: if a supervised MCP server itself believes it has already been initialized but its process restarted (e.g., via `child.spawn` inside a wrapper), a second initialize from MCOS could return an error. Not observed in any of the 4 servers we registered; document as a watch-item.
- **VERSION.json not in CMake's depend graph** — the build system only re-runs the `file(READ "${CMAKE_CURRENT_SOURCE_DIR}/VERSION.json" ...)` step when a watched file (e.g., `CMakeLists.txt`) changes. Bumping `VERSION.json` alone won't propagate to `MASTERCONTROL_VERSION`. Workaround in v0.9.4: touch `CMakeLists.txt` after bumping. v0.9.5 candidate: add `set_property(DIRECTORY APPEND PROPERTY CMAKE_CONFIGURE_DEPENDS VERSION.json)` so the version is hot-wired into the depend graph.
- **No DNS-SD / UDP beacon end-to-end test from this host** — capabilities advertise `dns-sd` + `udp-beacon` but I had no `dns-sd.exe` / mDNS listener in the validation environment. The advertisement is presumed live based on prior phase reports. v0.9.5: add a self-test that listens for the UDP beacon and confirms it carries the new `mcp-2025-03-26` capability.

## Ready for next phase?

Answer: **yes.** v0.9.4 leaves MCOS in a state where any LAN AI client (Claude Code, Codex, Grok via Streamable HTTP, ChatGPT via the connector-edge proxy path documented in onboarding) can:

1. Resolve the MCOS gateway via `/.well-known/mcos.json` or `/api/discovery`.
2. Pull a typed onboarding profile via `/api/onboarding/{client}`.
3. Pull a governance bundle via `/api/governance/bundles/{platform}`.
4. Connect to the gateway via Streamable HTTP at the advertised `mcpUrl`.
5. Complete the MCP `initialize` → `notifications/initialized` handshake (the gateway honors the spec on both directions).
6. Discover 33 tools across 5 supervised pools via `tools/list`, with full typed input schemas.
7. Invoke any of those tools via `tools/call` and get real values back, with persistence holding across MCOS service restarts for memory + sqlite pools.

The next numbered phase (PHASE-13 Win2D rendering) is unblocked. The non-numbered v0.9.5 release should pick up the deferred items above.
