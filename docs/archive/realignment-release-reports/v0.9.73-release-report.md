## Architecture note

The new authority's document hierarchy (per `FORSETTI_CONSTITUTION.md`):

| Rank | Document | Authority |
|---|---|---|
| 1 | `FORSETTI_CONSTITUTION.md` | Highest. Foundational principles. |
| 2 | `COMPLIANCE_POLICY.md`, `CHANGE_CONTROL_POLICY.md`, `RELEASE_POLICY.md`, `DOCUMENTATION_POLICY.md` | Binding operational policy. |
| 3 | `standards/*.md` | Operational standards. |
| 4 | `contracts/*.md` | Task contract templates. |
| 5 | `core/policies/*.json` | Canonical machine-readable policy manifests. |
| 6 | `agents/*.md` | Role-specific behavior bounds (architect, builder, validator, docs-manager, release-manager). |

The five governed roles per the new authority — Architect, Builder, Validator, Docs Manager, Release Manager — each have explicit authority + boundaries. CLU's enforceAction (Phase 3) will route privileged mutations through these.

## Risks

* **Phase 1 only.** v0.9.73 makes the authority *fetchable* but does not yet *enforce* it. The existing `/api/governance/bundles/{platform}` surface still produces the v0.9.x in-source CLU profile. Phase 2 (v0.9.74+) rewrites `GovernanceBundleService::composeRulesJson` to consume Forsetti Agentic Edition documents as canonical content; Phase 3 (v0.9.75+) wires `enforceAction` against the new policy registries (`changelog-rules.json`, `compliance-rules.json`, `docs-sync-rules.json`, `repo-boundaries.json`, `versioning-rules.json`).
* **Manual provisioning of the install-side share directory.** The vendored tree was deployed via `robocopy` for testing. A future iteration adds CMake install rules + WiX installer line so a fresh install ships the governance tree out of the box.
* **WinUI shell does not yet surface the new authority.** Routes are read-only via `curl` / direct fetch for v0.9.73. Future iteration adds a SecuritySectionControl / GovernanceSectionControl surface that browses the documents.
