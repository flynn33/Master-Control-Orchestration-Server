# Release Completion Report — v0.9.80

Build-system convenience: a CMake custom target that wraps `scripts/Deploy-LocalLive.ps1` so the hot-deploy round-trip is one invocation.

## Scope completed

### `DEPLOY_LOCAL_LIVE` custom CMake target

The v0.9.78 release shipped `scripts/Deploy-LocalLive.ps1` to bridge the gap between the CMake install prefix (`C:\Program Files\MasterControlOrchestrationServer\`) and the WiX MSI service path (`C:\Program Files\Master Control Orchestration Server\`). v0.9.80 wraps the script in a CMake custom target so the developer-loop is:

```powershell
cmake --build --preset debug --target DEPLOY_LOCAL_LIVE
```

That one invocation:

1. Re-runs `cmake --build` against the active preset (`debug` or `release` via `$<CONFIG>`).
2. Depends on the full `MASTERCONTROL_RUNTIME_TARGETS` set so anything stale gets rebuilt first.
3. Then runs `scripts/Deploy-LocalLive.ps1 -BuildDir <CMAKE_BINARY_DIR> -RelaunchShell` which:
   - Stops the `MasterControlProgram` service.
   - Kills any running `MasterControlShell.exe` so the binary + PDB aren't locked.
   - Copies the freshly-built `MasterControlServiceHost.exe`, `MasterControlShell.exe`, `MasterControlShell.pdb`, `app.js`, `styles.css`, and `index.html` (if changed) into the spaces-path live install.
   - Starts the service and polls `/api/version` with a 30s timeout.
   - Probes `/api/self-tests` and `/api/supervisor/status` as sanity checks.
   - Relaunches the shell and reports the PID + exit status.

Lives in the `IDEValidationTargets` folder alongside `IDE_PACKAGE` and `IDE_ACCEPTANCE_MIXED/MANAGED/BOTH` so it shows up in Visual Studio's Solution Explorer under that grouping.

This is the one-command developer workflow alternative to running the full WiX MSI installer for every patch.
