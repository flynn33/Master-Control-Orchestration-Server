# Release Completion Report — v0.9.49

Maintenance release. Companion to v0.9.48 — admin port (7300) had the same UTF-8 issue but was masked by the outer 500 handler.

## Scope completed

### Admin API responses no longer fail with HTTP 500 when client bodies contain invalid UTF-8

The v0.9.49 bug-hunt followed up on v0.9.48 by probing the admin port with the same Latin-1 `0xE9` byte that crashed `/mcp`. The admin port didn't crash (because `SimpleHttpServer::handleClient` has an outer try/catch returning 500 as a backstop), but it DID return:

```
POST /api/clients body:{"displayName":"bad-\xe9", ...}
  -> 500
  body: {"message": "[json.exception.type_error.316] invalid UTF-8 byte at index 187: 0x22"}
```

Same root cause as v0.9.48: `nlohmann::json::dump()` defaults to `error_handler_t::strict` which throws on invalid UTF-8 in any string field. The route's catch arm built a diagnostic message embedding `ex.what()` (which contained the bad byte from the parser's "last read" tail), and then `jsonResponse() -> dump()` threw on that bad byte itself, bouncing up to the outer 500 handler.

Wrong status (RFC 7231 §6.5.1: client-supplied bad bytes are 400, not 500), and the response body exposed nlohmann internal exception identifiers — a small information-leak.

The fix patches `MasterControlApplication::Impl::jsonResponse()` — the helper every admin API route uses — to call `dump()` with `error_handler_t::replace`. Invalid UTF-8 bytes become `�` (U+FFFD `�`) in the output. Same fix the v0.9.48 gateway crash needed; applied at the helper level so every admin route benefits without touching individual handlers. RFC 8259 §8.1 endorses this lenient behavior for tolerant JSON serializers.

**Verified live**:

The two v0.9.48 fixes (gateway crash) and v0.9.49 fix (admin 500→400) close the entire UTF-8-via-dump() class of issues across both HTTP surfaces.
