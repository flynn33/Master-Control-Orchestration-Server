# Release Completion Report — v0.9.54

Maintenance release. `/api/setup/dependencies/{id}/{not-install}` returned a misleading 405 instead of an explanatory 400.

## Scope completed

### Setup-dependencies bad-verb returns 400 with the supported-actions list

The v0.9.54 bug-hunt sent `POST /api/setup/dependencies/nodejs/preflight` (and `/uninstall`, `/detect`, `/zap`) and got back **405 Method Not Allowed** with `Allow: POST, OPTIONS`. That's misleading: POST IS the right method — it's the URL action segment that's unknown. The route handler at `if (POST && startsWith("/api/setup/dependencies/") && endsWith("/install"))` only matches `/install`; everything else falls through to the v0.9.28 `supportedMethodsForPath` fallback, which sees the prefix as POST-allowed and emits the 405.

The fix adds a sibling fallback handler immediately after the install route: it matches any `POST /api/setup/dependencies/{...}` that didn't reach `/install`, extracts the unknown action segment, and returns 400 with `"Unknown dependency action '<verb>'. Supported actions: install."`. A bare `/api/setup/dependencies/{id}` (no action segment) gets a route-shape 400 with the canonical URL.

`install` was the only action this gateway actually implements; `preflight`, `uninstall`, `detect` were on the bug-hunt's wishlist but never wired up. The 400 message tells the operator that explicitly so they don't waste cycles trying every verb.
