# Release Completion Report — v0.9.8

Maintenance release. Closes the three top items deferred from v0.9.7.

## Scope completed

### 1. Per-pool crash circuit breaker

Pre-v0.9.8 the watchdog respawned a worker every 2s indefinitely. A misbehaving worker that crashed in a tight loop would fill `/api/telemetry/events` with hundreds of error events per minute and consume a Job Object slot on every cycle. v0.9.7 verified this directly in the bug-hunt: killing `sequential-thinking` 3 times in 9 seconds produced 6 lifecycle events with no quarantine.

v0.9.8 adds per-pool failure tracking:

```cpp
static constexpr int kCrashThreshold        = 5;   // failures
static constexpr int kCrashWindowSeconds    = 60;  // rolling window
static constexpr int kQuarantineCooldownSeconds = 60;
std::map<std::string, std::vector<std::chrono::steady_clock::time_point>>
    poolFailureTimestamps_;
std::map<std::string, std::chrono::steady_clock::time_point>
    poolQuarantineUntil_;
```

When a pool crosses 5 worker deaths within 60 seconds, the auto-respawn loop in `reapDeadInstancesLocked()` skips it for 60 seconds. The watchdog tick prunes aged-out timestamps and expired quarantines so the bookkeeping is bounded. **`POST /api/pools/{id}/scale` is the explicit operator override** — it does not consult the quarantine map.

Verified live with a 6-kill storm:

```
Kill #1 : sequential-thinking#4  pid=8684    -> auto-respawn to #6
Kill #2 : sequential-thinking#6  pid=7116    -> auto-respawn to #7
Kill #3 : sequential-thinking#7  pid=15096   -> auto-respawn to #8
Kill #4 : sequential-thinking#8  pid=7060    -> auto-respawn to #9
Kill #5 : sequential-thinking#9  pid=13844   -> QUARANTINED
Kill #6 : pool already at 0 instances (quarantine in effect)
```

The operator-override path is verified: `POST /api/pools/sequential-thinking/scale` after the quarantine triggered immediate respawn to `sequential-thinking#10 state=ready`.

### 2. Humanized exit codes

Pre-v0.9.8 the death-event message read `exitCode=4294967295`. Operators reading the activity log had no signal whether that was a forced kill, an access violation, or something else. v0.9.8 adds `humanizeExitCode(code)` mapping the common Windows / NTSTATUS codes:

| Code | Label |
|---|---|
| `0x00000000` | STATUS_SUCCESS — normal exit |
| `0xC0000005` | STATUS_ACCESS_VIOLATION — segfault / null deref / bad pointer |
| `0xC000001D` | STATUS_ILLEGAL_INSTRUCTION |
| `0xC0000094` | STATUS_INTEGER_DIVIDE_BY_ZERO |
| `0xC0000409` | STATUS_STACK_BUFFER_OVERRUN — /GS triggered |
| `0xC000013A` | STATUS_CONTROL_C_EXIT |
| `0xC0000142` | STATUS_DLL_INIT_FAILED |
| `0xC0000374` | STATUS_HEAP_CORRUPTION |
| `0xC0000017` | STATUS_NO_MEMORY |
| `0xFFFFFFFF` | STATUS_PROCESS_KILLED — forced kill / taskkill /f / Stop-Process -Force |

Death-event message now reads:

```
Pool worker 'sequential-thinking#9' exited unexpectedly
(was state=ready,
 exitCode=4294967295 (STATUS_PROCESS_KILLED -- forced kill / taskkill /f / Stop-Process -Force))
```

Codes outside the table fall through and emit the bare numeric value.

### 3. Dual-emit pool-lifecycle events to `/api/activity`

Pre-v0.9.8 `/api/telemetry/events` (the PHASE-08 events ring) carried pool death/respawn events; the older `/api/activity` audit log surface (used by the dashboard's Recent Activity panel and operator audits) was blind to them. v0.9.8 makes `drainAndEmitSupervisorEvents` ALSO append to the global activity ring via a new `appendPoolLifecycleActivity(kind, poolId, message, statusCodeHint)` helper:

```
[pool_worker_death sc=500]   supervisor-watchdog  target=sequential-thinking
[pool_worker_respawn sc=200] supervisor-watchdog  target=sequential-thinking
[pool_quarantine sc=500]     supervisor-watchdog  target=sequential-thinking
```

The `target` field carries `poolId` for joins with `/api/pools`. Operators reading `/api/activity` see the same lifecycle story they see in `/api/telemetry/events`, plus the quarantine kind that is unique to this surface.

The helper is forward-declared near the supervisor class and defined alongside `globalActivityRing()` later in the same TU, so the supervisor can dual-emit without seeing the `ActivityEventRing` class definition (which is declared after the supervisor for layout reasons).

## Risks and blockers

- **Service shutdown wedged** during the v0.9.8 deploy. The OLD v0.9.7 binary held the SCM stop signal for >60 seconds; I had to force-kill it. Root cause not pinned down — could be the v0.9.7 watchdog thread holding `mutex_` during shutdown when shutdownAll runs concurrently. Watch-item; reproducible test would help. Defer to v0.9.9.
- **Quarantine event uses an ad-hoc encoding**: `DeathLogEntry` with empty `instanceId` and `exitCode = failure count`. Distinguishable in the consumer but cleaner is a separate `QuarantineLogEntry` struct. Defer to v0.9.9.
- **`mcpInitialized` reset on every respawn**: still true, still appropriate. Note for tracking.

## Ready for next phase?

Answer: **yes.** The five iterations (v0.9.4 → v0.9.5 → v0.9.6 → v0.9.7 → v0.9.8) collectively turn MCOS from "smoke-test surface only" into "production-functional 5-pool gateway with proactive supervision, operator-visible lifecycle telemetry, and back-pressure against runaway worker crashes". MCOS now:

1. Lets LAN AI clients reach the gateway and use 33 tools across 5 supervised pools end-to-end (memory, sequential-thinking, filesystem, sqlite, baseline-tools).
2. Auto-recovers from worker crashes within 2s (watchdog) regardless of LAN traffic.
3. Quarantines a pool that's crashing in a tight loop instead of respawning it forever.
4. Reports worker deaths, auto-respawns, and quarantines with human-readable exit-code labels in BOTH `/api/telemetry/events` AND `/api/activity`.
5. Reflects operator pool changes in LAN-client `tools/list` immediately (cache invalidation on upsert/remove/scale/drain).
6. Has a self-test that's robust to multi-host LAN environments.
7. Auto-reconfigures CMake on VERSION.json bumps.
8. Has 14 generic + 10 mcos-* sub-agents under `.claude/agents/` for Claude Code workflow use (separate from MCOS-supervised pool sub-agents — that gap is the remaining big-rock architectural item).
