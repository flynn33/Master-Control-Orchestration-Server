# Release Completion Report — v0.9.12

Maintenance release. Symmetric refactor: watchdog uses the same condition-variable wait pattern that BeaconService got in v0.9.11.

## Scope completed

### Watchdog uses `condition_variable::wait_for` (symmetric with BeaconService)

Pre-v0.9.12 the WorkerSupervisor watchdog observed `watchdogRunning_=false` within at most 100ms (it slept in 100ms steps). Functional but inconsistent with v0.9.11's BeaconService cv-based interruptible sleep. v0.9.12 replaces the polled sleep with `std::condition_variable::wait_for` guarded by `watchdogSleepMutex_` + `watchdogSleepCv_`. The destructor now calls `notify_all()` before falling into the bounded-join path, so the watchdog wakes within microseconds of `watchdogRunning_=false` instead of waiting for the next 100ms tick.

Two small benefits:
- **Symmetry**: BeaconService and WorkerSupervisor now use the same shutdown idiom — easier to reason about.
- **Tighter bound**: worst-case shutdown is now bounded by one in-flight reap+drain cycle (a few ms) instead of "100ms sleep + reap + drain".

**Verified live:** v0.9.11 stop = 0.3s, v0.9.12 stop = 0.3s (no regression; the BeaconService change in v0.9.11 was already dominating).
