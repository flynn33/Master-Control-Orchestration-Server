# Release Completion Report — v0.9.11

Maintenance release. Closes the BeaconService stop blocker observed in the v0.9.10 deploy.

## Scope completed

### Interruptible BeaconService::stop()

The v0.9.10 deploy measured service stop at **14.8 seconds** (and a follow-up measurement at 11.7s). The v0.9.10 watchdog-detach diagnostic helped rule out the watchdog as the cause: no `supervisor_shutdown_wedge` events appeared after restart. The likely culprit was `BeaconService::stop()` — the worker thread slept for `beaconBroadcastIntervalSeconds` (15s default) between broadcasts using `std::this_thread::sleep_for`, so when `stop()` flipped `running_=false` mid-sleep, the worker didn't observe the flag until the sleep completed.

v0.9.11 makes the sleep interruptible:

```cpp
// new private members
std::mutex sleepMutex_;
std::condition_variable sleepCv_;

// in worker loop
{
    std::unique_lock<std::mutex> lock(sleepMutex_);
    sleepCv_.wait_for(lock,
        std::chrono::seconds(configuration.beaconBroadcastIntervalSeconds),
        [this]() { return !running_.load(); });
}

// in stop()
{
    std::lock_guard<std::mutex> lock(sleepMutex_);
    sleepCv_.notify_all();
}
if (worker_.joinable()) worker_.join();
```

`wait_for` returns when either the timeout elapses OR the predicate is satisfied (the worker reads `running_=false`). Notifying under the lock prevents the missed-wakeup race. The same pattern is applied to the disabled-state poll loop so toggling `beaconEnabled` off/on doesn't block stop either.

**Verified live with a back-to-back measurement:**

| Binary | Stop elapsed |
|---|---|
| v0.9.10 (sleep_for) | **11.7 seconds** |
| v0.9.11 (wait_for + notify_all) | **0.3 seconds** |

That's a 39× shutdown speedup. SCM no longer hits StopPending; in-place binary upgrades complete in well under 5 seconds total wall time.

## Risks and blockers

- **One sleep-loop site remaining** in `WorkerSupervisor`'s watchdog. The watchdog already uses 100ms steps + atomic check, so the destructor sees `watchdogRunning_=false` within 100ms. That's not a blocker, but for symmetry a future v0.9.x could convert that to the same `wait_for + cv` pattern for cleaner consistency.
