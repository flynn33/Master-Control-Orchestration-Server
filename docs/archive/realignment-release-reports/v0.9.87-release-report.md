# Release Completion Report — v0.9.87

`/api/health/summary` gains a supervisor block + supervisor error state contributes to the overall `ok` flag.

## Scope completed

### Supervisor block in /api/health/summary

Pre-v0.9.87 the summary returned `gateway`, `pools`, `activityRing`, `hostTelemetry`, and a top-level `ok` flag aggregating them. The supervisor surface (added v0.9.76, watchdog'd v0.9.78) was missing entirely. An operator polling `/api/health/summary` for a green light wouldn't see a supervisor in `error` state — even though that state indicates a real handshake failure that the operator probably wants to know about.

v0.9.87 adds:

```json
"supervisor": {
    "state": "config_generated",
    "activeProviderId": "claude",
    "active": true,
    "lastErrorMessage": ""
}
```

And tightens the overall `ok`:

```cpp
out["ok"] = gatewayOk && poolsOk && ringOk && teleOk && supervisorOk;
// supervisorOk = (status.state != SupervisorState::Error)
```

State map for the supervisor contribution:
* `off` / `revoked` → ok contributor: `true` (operator intent — no supervisor by design)
* `config_generated` / `pending_connection` / `connected` → ok=true (normal lifecycle)
* `disconnected` → ok=true (heartbeat gap — the watchdog already audit-logs this, but it's a transient remote-client issue, not a server health failure)
* `error` → ok=false (handshake / capability / expiration failure)

## Public contracts changed

`/api/health/summary` response gains an optional `supervisor` object (present iff `supervisorAssignmentService_` is initialized, which is always in practice post-v0.9.76). The `ok` field's semantics tighten: a supervisor in `error` state now flips `ok` to `false`. Pre-v0.9.87 supervisor errors had no effect on this field.

No schema is published for `/api/health/summary` so the change isn't schema-breaking.
