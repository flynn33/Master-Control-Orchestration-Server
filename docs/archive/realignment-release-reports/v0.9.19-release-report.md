# Release Completion Report — v0.9.19

Maintenance release. Closes a real gap that's been latent for fifteen releases.

## Scope completed

### POST `/api/governance/decisions` — wires CLU `enforceAction`

The v0.9.4 GET stub on `/api/governance/decisions` advertised:

```json
{
  "method": "POST",
  "expects": "GovernanceEnforcementRequest",
  "returns": "GovernanceEnforcementDecision",
  "note": "POST handler lands in PHASE-07 alongside the lease router."
}
```

PHASE-07 then shipped (lease router live since v0.9.4 + matured through every iteration since), but the POST handler was never added. POST returned `{"message":"Not found"}` for v0.9.4 → v0.9.18.

The v0.9.19 bug-hunt found this: testing `POST /api/governance/decisions` with a valid-shape body got 404 instead of an enforcement decision. The fix is straightforward — `commandLogicUnitService_->enforceAction(GovernanceEnforcementRequest)` has existed since PHASE-05 and is already used by `installPackageJson`, `installRepoJson`, and `executeForsettiToolJson`. v0.9.19 wires the same call into the route:

```cpp
if (request.method == "POST" && request.path == "/api/governance/decisions") {
    if (!commandLogicUnitService_) {
        return jsonResponse(OperationResult{ false, false, "CLU service is not running." });
    }
    try {
        const auto enforcementRequest =
            nlohmann::json::parse(request.body).get<GovernanceEnforcementRequest>();
        const auto decision = commandLogicUnitService_->enforceAction(enforcementRequest);
        return jsonResponse(decision);
    } catch (const std::exception& ex) {
        return jsonResponse(OperationResult{ false, false,
            std::string("Invalid GovernanceEnforcementRequest JSON: ") + ex.what() });
    }
}
```

**Verified live:**

```
POST /api/governance/decisions
{ "action":"remote_install", "actor":"test-client",
  "subjectId":"test-package", "allowUntrustedExecution":false }

→ { "action":"remote_install", "allowed":true, "blockingFindings":[],
    "deferredActionId":"", "message":"", "outcome":"allow",
    "posture":"warning", "ruleId":"" }
```

Same call with `action:"client_register"` → also `allow / warning posture`.

The GET stub stays put (its body is now a plain "POST evaluates a request through CLU's enforceAction" hint without the obsolete "lands in PHASE-07" promise).

## Risks and blockers

- **Action-enum string format isn't documented in the discovery doc**. Clients have to know that `action` is `snake_case` (`remote_install`, `client_register`, etc.) rather than `PascalCase`. Watch-item; the enforcement-decision response shape includes `action: "remote_install"` so a client can round-trip what comes back, but the FIRST request needs the right string. Future v0.9.X candidate: surface the valid action-enum strings on the GET stub.
