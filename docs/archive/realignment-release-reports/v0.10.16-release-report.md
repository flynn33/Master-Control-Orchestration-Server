# Release Completion Report — v0.10.16

SSE resume on `/api/events`. v0.10.15 carried over from the debugger sub-agent scan that an `EventSource` client reconnecting to `/api/events` had no way to resume from the last event it observed: the handler initialised `lastActivityId = ""` unconditionally, so reconnecting clients replayed from the live ring head and any activity event that landed during the disconnect window was silently dropped. v0.10.16 honors the standard `Last-Event-ID` request header (the value the browser auto-echoes on reconnect) and the same `?since=` alias set v0.10.15 added. The handler now emits an SSE `id:` field on every activity event so `EventSource` can record it.

## 3. Public contracts changed

- **`GET /api/events` request surface (additive).** The handler now reads `Last-Event-ID` and the v0.10.15 alias set on `?since=`. Clients that send neither continue to start from the live tail exactly as before. No header or query name was renamed; no error response was introduced.
- **`GET /api/events` response surface (additive).** Each `event: activity` frame now contains an `id: <evt.id>` line. Browsers, `curl`, and any conforming SSE parser will store the value and echo it on reconnect; non-SSE parsers ignore the new line. Dashboard frames are unchanged (no `id:` field, deliberately).
- **No removed or renamed HTTP routes, no removed or renamed Shell controls.**

## 5. Risks and blockers

- **Activity event id ordering.** The resume contract assumes the global activity ring's `read(sinceId, maxCount)` returns events with ids strictly greater than `sinceId`. That contract has held since the ring's introduction (the v0.10.x test suite includes round-trip cases on `ActivityEvent.id`); if a future change to the ring's id generation breaks monotonicity, the resume could replay or skip. Not changed in v0.10.16; called out for awareness.
- **No unit test of the SSE resume.** Streaming the SSE channel from inside the in-process ctest binary would require a running HTTP server fixture, which the project's lightweight test harness deliberately avoids. Live integration probe via PowerShell `Invoke-WebRequest -Method Get` with a streaming reader is the verification path used.
