# Release Completion Report — v0.10.2

Iteration-3 follow-up to v0.10.1. While auditing the boot self-test wiring against the operator directive *"If a test fails, then it MUST be reported in the Overview Error Frame and reported to the diagnostic module"*, I confirmed the activity-ring path covers the Error Frame but found the persistent Diagnostics log was not receiving any self-test events. After a service restart the in-memory activity ring rolls over and any historical self-test failure becomes unrecoverable.

## Scope completed

`emitSelfTestActivity()` now writes one persistent log row per **failed** probe via `MasterControl::Diagnostics::appendEvent` with `severity = "error"`. The data block carries `name`, `category`, `durationMs`, and `ranAtUtc` so an external auditor can reconstruct the timeline.

`runBootSelfTestsNow()` now also writes one summary row per boot via `MasterControl::Diagnostics::appendEvent` with `severity = "info"` (all-pass) or `severity = "warn"` (any failed). The summary data block carries `passed`, `failed`, `total`, `startedAtUtc`, `finishedAtUtc`.

PASS rows are NOT written to the persistent log to avoid unbounded growth (~30+ lines per boot indefinitely with no rotation). The per-pass detail remains available from the activity ring while the service is running and from `/api/self-tests` for the latest snapshot. If a future operator wants a permanent per-pass audit trail, that needs an opt-in flag and a rotation policy.

The persistent log lives at `<PUBLIC>\Documents\Master Control Orchestration Server\logs\runtime\events.jsonl`, alongside the other runtime diagnostic events (supervisor, telemetry, etc.). Each line is one JSON object; `jq` and `Get-Content -Tail` both work for inspection.

## Risks and blockers

If the persistent log destination (`<PUBLIC>\Documents\...`) is unwritable (locked-down account, missing PUBLIC env var, etc.), `appendJsonLine` returns silently per the existing pattern. The activity ring still gets the event so the Overview Error Frame still surfaces failures. This matches the existing behavior for supervisor + gateway events.
