# Release Completion Report — v0.9.58

Canonical install-command surfacing for catalog placeholder MCPs. Pre-v0.9.58 the v0.9.56/57 diagnostic block told operators *"register a pool with poolId=X"* but didn't carry the install path for entries that map to a published npm MCP package. v0.9.58 adds an `installCommand` field, populates it from a static map for the entries we know the canonical package for, and surfaces it as a copy-paste-able code block in the dashboard card.

## Scope completed

### `installCommand` + `installPackageDetected` fields

Both `SubAgentRuntimeStat` and `McpServerRuntimeStat` gain:

| Field | Type | Purpose |
|---|---|---|
| `installCommand` | string | Operator-runnable install command (e.g. `npm install -g @playwright/mcp`). Empty when no canonical install path is known. |
| `installPackageDetected` | bool | Reserved for boot-time detection of pre-installed packages. Always `false` in v0.9.58; the field is plumbed end-to-end so a future iteration can flip it without further model or UI work. |

### Static known-package map for 6 entries

`enrichRuntimeStatDiagnostics` consults a private `kKnownInstallCommand` map in the `awaiting_pool_registration` path:

| Entry id | Canonical install command |
|---|---|
| `playwright` | `npm install -g @playwright/mcp` |
| `chrome-devtools` | `npm install -g chrome-devtools-mcp` |
| `filesystem` | `npm install -g @modelcontextprotocol/server-filesystem` |
| `memory` | `npm install -g @modelcontextprotocol/server-memory` |
| `sequential-thinking` | `npm install -g @modelcontextprotocol/server-sequential-thinking` |
| `sqlite` | `npm install -g mcp-server-sqlite-npx` |

The four already-pooled MCPs still carry their canonical command for re-deployment / migration scenarios. Entries without an entry in the map (computer-use, desktop-control, terminal-shell, local-git, file-search, code-execution-repl, etc.) leave `installCommand=""` and the dashboard simply omits the install line — no fabricated guesses.

### Dashboard card renders the install line

`renderRuntimeStatDiagnostic` in `app.js` now appends an `<p class="subagent-diag-install-cmd">` line when `stat.installCommand` is non-empty. The styling uses `user-select: all` so a single click selects the whole command for copy-paste.

### `installHint` enriched with the install command

When `installCommand` is populated, the `installHint` text gets `Known install command: \`<cmd>\`.` appended to the operator-readable next-step line, so consumers reading just the hint (e.g. plugin-side scripts) see the install path inline.

## Risks and blockers

* **Static map staleness.** The canonical npm package names are baked into source. A future package rename would require a code update. Mitigation: only six entries; trivial to update.
* **`installPackageDetected` always false.** Plumbed but not wired to filesystem detection yet. Operator UI shows the install command but cannot disambiguate "package available — just register a pool" from "package not yet installed." Next iteration adds the boot-time scan.

## Deferred work

* **Boot-time npm package detection** to flip `installPackageDetected` for installed entries (this is the v0.9.59 target).
* **Auto-spawn supervised pools** for detected packages (carries to v0.9.60+).
* MCOS sub-agent runner architectural decision (carried).
* PHASE-13 Win2D rendering (carried).
* SSE upgrade for Streamable HTTP (carried).
