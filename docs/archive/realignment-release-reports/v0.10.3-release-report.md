# Release Completion Report — v0.10.3

Iteration-4 follow-up to v0.10.2. v0.10.2 added a `MasterControl::Diagnostics::appendEvent` call that writes self-test failures to a persistent log file under `<PUBLIC>\Documents\Master Control Orchestration Server\logs\runtime\events.jsonl`. The failure mode was silent: if the destination directory was unwritable (locked-down account, missing PUBLIC env var, ACL change), the rows would be dropped without any visible warning and the operator's audit trail would degrade invisibly.

v0.10.3 turns that into an explicit boot self-test probe.

## Scope completed

`runBootSelfTestsNow()` gains probe number 7: `diagnostics.log_writable` in the `telemetry` category. Sequence:

1. Resolve the persistent log paths via `MasterControl::Diagnostics::resolvePersistentLogPaths(L"runtime")` — this returns `<root>/runtime/{events.jsonl,telemetry.jsonl}`.
2. `std::filesystem::create_directories(componentDirectory)` — fail if create errored.
3. Open `<componentDirectory>/.write-probe` with truncate, write `"{}"`, close. Fail on any open/write error.
4. `std::filesystem::remove(probe)` to leave the directory clean.
5. Record the probe with the resolved path in the message so the operator sees *exactly* which directory passed/failed.

Failure surfaces in:

- the Overview Error Frame (via the existing `self_test` activity-ring path)
- the persistent Diagnostics log via the v0.10.2 routing (or the activity ring only, if the log itself is what failed)

The probe runs on every boot and on every operator-initiated `POST /api/self-tests/run`. Total boot self-test count rises from 37 to 38.

## Risks and blockers

The probe depends on `<PUBLIC>\Documents\` being resolvable. On highly-locked-down accounts where the PUBLIC env var is missing entirely, `MasterControl::Diagnostics::resolvePersistentLogPaths` falls back to LOCALAPPDATA, then to current_path()/logs. The probe still works in those fallback paths, but the operator's `events.jsonl` ends up somewhere different than the documented PUBLIC location. If the operator wants strict enforcement of the PUBLIC location, a future probe variant could check the resolved path against an expected anchor.
