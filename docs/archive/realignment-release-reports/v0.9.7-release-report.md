# Release Completion Report — v0.9.7

Maintenance release. Closes the top item deferred from v0.9.5 / v0.9.6: worker death and auto-respawn now surface as TelemetryEvents in `/api/telemetry/events` and the operator dashboard.

## Scope completed

### Activity-ring telemetry for worker death + auto-respawn

Pre-v0.9.7 the supervisor's `deathLog_` / `respawnLog_` vectors collected events when reaps happened — but no consumer drained them. Operators saw nothing in any operator-facing surface when a worker crashed and got auto-restarted in the background; the only signal was a pid change in `/api/pools/{poolId}` if the operator happened to query during/after the event.

v0.9.7 adds:

- **`WorkerSupervisor::setTelemetryAggregator(std::shared_ptr<ITelemetryAggregator>)`** — setter wired by the runtime composition root after both the supervisor and the aggregator are constructed.
- **`WorkerSupervisor::drainAndEmitSupervisorEvents()`** — two-phase helper. Phase 1 (under `mutex_`): swap-out `deathLog_` / `respawnLog_` to local vectors and copy the aggregator pointer. Phase 2 (mutex_ released): emit one TelemetryEvent per entry through the aggregator. The two-phase pattern is deliberate — calling `recordEvent` while holding `mutex_` would invert lock ordering with the aggregator's own internal lock.
- **Watchdog drain** — the 2-second background reap loop now calls `drainAndEmitSupervisorEvents()` after every `reapDeadInstancesLocked()` pass.
- **Operator-API drain** — `listPools()` and `findPool()` also call `drainAndEmitSupervisorEvents()` at the end of their reap pass so operator-API-driven reaps surface events immediately rather than waiting for the next watchdog tick.

Event shapes:
- **Death**: `Telemetry::Worker::Error`, message `"Pool worker '{instanceId}' exited unexpectedly (was state={prevState}, exitCode={code})"`, `poolId` set.
- **Respawn**: `Telemetry::Worker::Info`, message `"Pool '{poolId}' auto-respawned to instance '{newInstanceId}' (watchdog or reap-driven)"`, `poolId` set.

**Verified live** with a kill+wait probe:
```
$ curl -s http://localhost:7300/api/telemetry/events | grep '"category":"worker"' ...
{"category":"worker","message":"Pool worker 'memory#3' exited unexpectedly (was state=ready, exitCode=4294967295)","poolId":"memory","severity":"error", ...}
{"category":"worker","message":"Pool 'memory' auto-respawned to instance 'memory#6' (watchdog or reap-driven)","poolId":"memory","severity":"info", ...}
```

## Risks and blockers

- **Per-instance crash counter / circuit breaker not yet implemented.** Today a worker that crashes repeatedly gets respawned every 2s indefinitely. v0.9.8 candidate: N-failures-in-M-seconds gate that quarantines a pool after a threshold. Without it, a misbehaving worker could fill the activity log with thousands of error events over hours.
- **Activity-ring (`/api/activity`) does NOT receive death/respawn events** — only `/api/telemetry/events` does. The two surfaces have different producers; the gateway-lifecycle events landed in both via separate calls (see v0.9.3 fix). Adding pool death/respawn to `/api/activity` requires giving the supervisor access to `globalActivityRing()`, which means either passing it through the constructor or referencing the global. Watch-item; defer to v0.9.8.
- **`exitCode=4294967295` (0xFFFFFFFF) in the death event message.** That's `STATUS_PROCESS_KILLED` mapped to UINT32_MAX when the parent `Stop-Process -Force`s a child. Useful signal for operators ("killed externally") but raw — could humanize in a future release.

## Ready for next phase?

Answer: **yes.** v0.9.7 closes the operator-visibility loop on worker lifecycle. Combined with v0.9.4-v0.9.6, MCOS now:

1. Lets LAN AI clients reach the gateway and use 33 tools across 5 pools end-to-end.
2. Auto-recovers from worker crashes within 2 seconds (watchdog-driven), regardless of LAN traffic.
3. Reflects operator pool changes in LAN-client `tools/list` immediately (no TTL drift).
4. Reports worker deaths and auto-respawns to operators via `/api/telemetry/events` and the dashboard.
5. Has a self-test that's robust to multi-host LAN environments (skips foreign-host beacons).
6. Auto-reconfigures CMake on VERSION.json bumps so the running binary's reported version is always in sync.

The four iterations (v0.9.4 → v0.9.5 → v0.9.6 → v0.9.7) collectively turned MCOS from "smoke-test surface only (baseline-tools)" into "production-functional 5-pool gateway with proactive supervision and operator-visible lifecycle telemetry". The remaining deferred items are either polish (PHASE-13 UI, exit-code humanizing, activity-ring duplication) or large architectural decisions (sub-agent runner) — none block client-facing operability.
