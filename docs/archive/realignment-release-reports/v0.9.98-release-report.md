# Release Completion Report — v0.9.98

**Regression fix.** Operator-reported: "The MCP server cards are missing from telemetry view." Root cause: `Deploy-LocalLive.ps1` (shipped v0.9.78, wrapped by the `DEPLOY_LOCAL_LIVE` cmake target in v0.9.80) never copied the compiled XAML resources (`.xbf`), the merged resource index (`.pri`), or the WinRT metadata (`.winmd`) alongside `MasterControlShell.exe`. The shell binary at the live install was fresh (v0.9.97 from 5/10 8:25 PM), but the `TelemetrySectionControl.xbf` it loaded was from **5/8/2026 6:52 AM** — predating v0.9.76's addition of the MCP server card grid.

## Scope of the regression

Every XAML/WinRT change shipped between v0.9.76 and v0.9.97 was invisible to the operator on the WinUI Shell surface:

| Change | Release | Visibility before v0.9.98 |
|---|---|---|
| MCP Servers + Sub-Agents card grids on Telemetry view | v0.9.76 | **invisible** (`FindName('TelemetryMcpServersCardStack')` failed against stale .xbf → `PopulateMcpServerCards` was a no-op) |
| Supervisor Agent card on Overview surface | v0.9.76 | **invisible** |
| Supervisor toggle group + Generate/Revoke buttons | v0.9.76 | **invisible** |
| Bind-address resolution on visible TextBlocks (existing x:Name) | v0.9.76 | visible (TextBlock was already on the 5/8 .xbf; only the text content changed via C++) |
| Tray icon menu + Open Browser entry | v0.9.79 / v0.9.82 | visible (WndProc-driven, not XAML) |
| Compact MCP/Sub-Agent card density | v0.9.76 | invisible (the StackPanel target didn't exist) |
| Telemetry sidebar Supervisor card | v0.9.79 | **invisible** |
| Supervisor detail row (Assignment/Config/Expires/Heartbeat/Client/Issued) | v0.9.79 + v0.9.96 | **invisible** |
| Relative-time supervisor heartbeat | v0.9.90 | invisible inside the supervisor card (since the card wasn't there) |

Browser dashboard surfaces were *not* affected — `app.js` and `styles.css` were always being copied. Any operator using `http://127.0.0.1:7300/` saw every UX change correctly.

The C++ code, the routes, the persistence, the tests — all correct. The deploy script was the single missing link.

## Root cause

`Deploy-LocalLive.ps1` (v0.9.78) was designed to copy `MasterControlServiceHost.exe`, `MasterControlShell.exe`, `MasterControlShell.pdb`, `app.js`, `styles.css`, and `index.html`. The .xbf / .pri / .winmd resources WinUI 3 needs alongside the shell were never in the list. The WiX MSI installer's original install populated those at 5/8 with the then-current XAML; subsequent hot-deploys never refreshed them.

`FindName(x:Name)` on a missing XAML element returns `nullptr` in WinUI 3 — the C++ code then silently no-ops. No exception, no error log, no visible failure. Just a missing card.

The bug was self-cloaking: every release report listed "Service answers version=0.9.X" + "Shell relaunched as PID Y" as proof of successful deploy. Those probes confirmed the .exe was updated. They didn't catch that the XAML wasn't.

## Fix

One block in `scripts/Deploy-LocalLive.ps1`:

```powershell
$ShellBuildDir = Join-Path $BuildDir 'src\MasterControlShell\Debug'
foreach ($pattern in '*.xbf', '*.pri', '*.winmd') {
    Get-ChildItem $ShellBuildDir -Filter $pattern -ErrorAction SilentlyContinue | ForEach-Object {
        Copy-Item $_.FullName (Join-Path $LivePath $_.Name) -Force
        Write-Host ("  + " + $_.Name) -ForegroundColor DarkGray
    }
}
```

The DEPLOY_LOCAL_LIVE cmake target (v0.9.80) needs no change — it just invokes the script.

## Risks and blockers

None — the fix is additive (copying extra files). The same .xbf set is what the WiX MSI installer ships; identical bits land at the same path.

## Notes for the operator

* If you launched the shell BEFORE this fix landed, **close that shell window and reopen** (or use the tray's `Close MCOS Shell` and relaunch). The new shell process needs to load the fresh .xbf files.
* The Supervisor Agent card on the Overview surface is what you should now see at the top-third of the Overview pane, below the Self-Tests card.
* The MCP server cards live at the bottom of the Telemetry view (after the Host Pressure / Resource Allocation / Operational Activity grids). Scroll down on the Telemetry tab.
