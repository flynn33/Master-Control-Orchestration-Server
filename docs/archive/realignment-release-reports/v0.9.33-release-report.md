# Release Completion Report — v0.9.33

Maintenance release. Two more "resource does not exist" routes that were returning 200 with succeeded:false.

## Scope completed

### Lease-route 404s

The v0.9.33 bug-hunt continued the v0.9.30 "200-with-succeeded:false → real HTTP status" cleanup and surfaced two more sites in the lease subsystem:

| Route                                                  | pre-v0.9.33 | v0.9.33 |
|---|---|---|
| `POST /api/leases/{unknown-id}/release`                | 200 + `{ succeeded:false, message:"Unknown lease id." }` | **404** + same body |
| `POST /api/pools/{unknown-pool}/leases`                | 200 + `{ state:"failed", statusMessage:"Unknown pool." }` | **404** + same body |

Both routes already had the right *body*; they just hadn't been wired through to the right HTTP status code. The acquire-lease path returns an `EndpointLease` JSON object (not an `OperationResult`), which is why the v0.9.30 sweep didn't catch it — the status check there has to inspect `lease.state == LeaseState::Failed && lease.statusMessage == "Unknown pool."` rather than `result.succeeded == false`.

The release-lease path was a simpler fix: extract the result before serializing, branch on `result.message == "Unknown lease id."`, return 404. Pre-v0.9.33 it was a one-liner `return jsonResponse(leaseRouter_->releaseLease(...))` which had no place to inspect the result.

**Verified live**:

```

POST /api/pools/no-such-pool/leases              -> 404  body: { state:"failed", "Unknown pool." }

=== regression: real lease lifecycle ===
```
