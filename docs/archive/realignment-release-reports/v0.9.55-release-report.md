# Release Completion Report — v0.9.55

Maintenance release. Companion to v0.9.54 — `/api/clu/approvals/{id}/{not-approve-not-reject}` now returns 400 with action list instead of generic 404.

## Scope completed

### `/api/clu/approvals/{id}/{verb}` bad-verb returns 400 with supported-actions list

The v0.9.55 bug-hunt followed v0.9.54 across the other URL families and found the same misleading-fallback issue at `/api/clu/approvals`. POST to any verb that isn't `/approve` or `/reject` fell through to `staticFileResponse` and returned a generic 404 "Not found" — technically correct (no handler matched), but the operator can't tell whether the URL family was wrong or the verb segment was unrecognized.

The fix mirrors v0.9.54: a sibling fallback handler immediately after the `/reject` route catches any other `POST /api/clu/approvals/{...}` and returns 400 with `"Unknown approval action '<verb>'. Supported actions: approve, reject."`. Bare `/api/clu/approvals/{id}` (no action segment) returns 400 with the route shape.

**Verified live**:

```
=== v0.9.55 ===
POST /api/clu/approvals/x/badverb        -> 400 "Unknown approval action 'badverb'. Supported actions: approve, reject."
POST /api/clu/approvals/x/zap            -> 400 (same shape)
POST /api/clu/approvals/x                 -> 400 "Approval action route is POST /api/clu/approvals/{id}/{approve|reject}."

=== regression: real verbs still work ===
POST /api/clu/approvals/never/approve    -> 404 (v0.9.43, "Deferred action not found.")
POST /api/clu/approvals/never/reject     -> 404 (v0.9.43)
GET  /api/clu/approvals                   -> 200 (lists empty queue)

=== regression: prior v0.9.x preserved ===
GET /api/version                                    -> 200
POST /api/setup/dependencies/nodejs/preflight       -> 400 (v0.9.54)
/api/health/summary                                  -> ok:true, toolCount:33
```
