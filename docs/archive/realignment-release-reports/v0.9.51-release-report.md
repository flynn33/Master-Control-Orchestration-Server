# Release Completion Report — v0.9.51

Maintenance release. JSON-RPC batch + non-object top-level requests now return -32600 instead of HTTP 500.

## Scope completed

### `[{...}, {...}]` batch returns proper -32600 instead of HTTP 500

Per JSON-RPC 2.0 §6, a client may send a batch as an array of request objects. A server can either process the batch (returning an array of responses) or refuse it with a single Invalid Request error. This MCP gateway processes single-request envelopes only; the v0.9.51 bug-hunt sent `[{...},{...}]` and got back:

```
HTTP 500
{"jsonrpc":"2.0","id":null,"error":{"code":-32603,"message":"Internal error: [json.exception.type_error.306] cannot use value() with array"}}
```

Root cause: `req.value("method", ...)` throws `type_error.306` ("cannot use value() with array") when `req` is an array. The v0.9.48 `serveLoop` catch picked it up — service didn't crash — but the response was HTTP 500 with the nlohmann internal exception type leaking through.

Same issue for non-object top-level JSON (string `"hello"`, number `42`, `null`, etc.) — `req.value()` throws on them.

The fix adds two type checks right after `json::parse()`:

```cpp
if (req.is_array()) {
    return buildJsonRpcError(-32600,
        "Invalid Request: JSON-RPC batches are not supported by this gateway. "
        "Send a single request object per HTTP POST.");
}
if (!req.is_object()) {
    return buildJsonRpcError(-32600,
        "Invalid Request: top-level JSON must be a request object.");
}
```

The response is now a proper JSON-RPC error envelope at HTTP 200, with the spec-correct -32600 error code.
