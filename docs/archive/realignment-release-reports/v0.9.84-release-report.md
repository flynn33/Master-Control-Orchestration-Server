# Release Completion Report — v0.9.84

Wildcard-bind URL leak on `/api/health/summary`.

## Scope completed

### Bug found during v0.9.83 bug-hunt probe

During the post-deploy probe for v0.9.83, `/api/health/summary` returned:

```json
"gateway": {
    "adapterType": "native",
    "mcpUrl": "http://0.0.0.0:8080/mcp",
    "state": "running",
    "toolCount": 97
}
```

`http://0.0.0.0:8080/mcp` is the raw `composeMcpUrl(listenHost)` from `NativeHttpSysGatewayAdapter`, embedding the wildcard bind directly in the URL. The wildcard is the correct value for the actual HTTP.sys listen call (the kernel binds to all interfaces), but it cannot be routed by an external AI supervisor or LAN client.

The discovery handler (`/api/discovery`, `/.well-known/mcos.json`) already applies a substitution chain (bindAddress → activeProfile.preferredBindAddress → primaryIpAddress → 127.0.0.1) to produce a routable mcpUrl in its document. That substitution was added in v0.9.0 alongside the in-process HTTP.sys adapter. The `/api/health/summary` route (added in a later release) was missed; its gateway block surfaced `status.mcpUrl` unmodified.

### Fix

Inline the substitution chain in the `/api/health/summary` handler. Wildcard binds (`0.0.0.0`, `[::]`, `[::0]`) are replaced with the resolved LAN IP before the URL hits the response.

Operator-side debug field: `gateway.mcpUrlRaw` exposes the unsubstituted URL so an operator UI can render both the listening-on value (the wildcard) and the reachable-at value (the LAN IP), matching the bind-address pair the WinUI Shell already renders via the v0.9.76 helper.

## Public contracts changed

`/api/health/summary` `gateway` object gains a new optional field `mcpUrlRaw`. The `mcpUrl` field changes from "raw listen URL with wildcard host" to "resolved client-routable URL with LAN IP". Existing consumers that read `mcpUrl` get a more useful value automatically; consumers that specifically wanted the raw listen value (none observed in the codebase) can read `mcpUrlRaw`.

No schema is published for `/api/health/summary` so the addition isn't a schema-breaking change.
