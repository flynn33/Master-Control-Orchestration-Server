# Release Completion Report — v0.9.23

Maintenance release. Adds an atomic `pool restart` operation.

## Scope completed

### `POST /api/pools/{poolId}/restart` — atomic worker rotation

Pre-v0.9.23 an operator wanting to hot-rotate a pool's workers had to send two separate calls in sequence — `POST .../remove` + `POST /api/pools` (re-register from cached config) + `POST .../scale` — racy if monitoring polls between them. v0.9.23 wires those steps as one operation:

1. Snapshot the pool's current definition via `findPool` (preserves executable, args, environment, scale + drain policies).
2. `removePool` (terminates supervised children, drops the pool record).
3. `upsertPool` with the snapshotted definition (re-registers identical config).
4. `ensureMinInstances` (re-spawns workers).
5. `persistSupervisedPoolsToConfiguration` + `mcpGateway_->InvalidateToolCatalog()` + `Telemetry::Worker::Info` event (matches the existing remove/upsert/scale handlers).

No new behavior in `WorkerSupervisor`; pure convenience composition of existing primitives.

**Verified live**:

```
before: instanceId=memory#3 pid=6440
POST /api/pools/memory/restart
→ { "succeeded": true,
    "message": "Pool restarted (removed + re-registered + scaled to minInstances): Pool already at or above minInstances." }
after:  instanceId=memory#6 pid=11260 state=ready
PASS: process rotated
PASS: memory.read_graph returned successfully against new instance
```

Pid changed (process really rotated), tool call still works (catalog correctly refreshed).

Use cases:
- A misbehaving worker that hasn't crossed the v0.9.8 quarantine threshold.
- A config-file change to the pool's environment that needs a fresh process to pick up.
- Precautionary rotation after a memory-leak suspicion.
