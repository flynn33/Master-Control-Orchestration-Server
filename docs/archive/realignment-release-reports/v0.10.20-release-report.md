## 1. Findings → resolution

| Route | Typed model | Change |
|---|---|---|
| `POST /api/runtime/mcp-servers` | `RuntimeEndpoint` | Added route-layer parse + `collectDroppedTopLevelKeys` + response body `droppedKeys` array. |
| `POST /api/runtime/subagents` | `RuntimeEndpoint` | Same; a `subAgentDroppedKeys` local captures the diagnostic at the top of the handler and the response emit at the bottom (after the v0.7.2 auto-promote-to-pool block) appends it. |

- **`POST /api/runtime/mcp-servers` response body (additive).** New `droppedKeys` array appears when the input payload included top-level keys not declared in `RuntimeEndpoint`. Status codes unchanged (200 on success, 400 on parse-error).

- **`POST /api/runtime/subagent-groups` response body (additive).** Same shape for `SubAgentGroupDefinition`.
- **`POST /api/clients/{clientId}/privileges` response body (additive).** Same shape for `LanClientPrivileges`. Both the 200 success path and the 400/404 failure paths now include `droppedKeys` when present.

No removed or renamed HTTP routes. No removed or renamed Shell controls. No status-code regressions.

## 5. Risks and blockers

- **Double-parse cost.** Each refactored route now parses the JSON body twice (once in the route layer to extract droppedKeys, once in the existing `adminApiService_->upsertXJson(rawBody)` call). For the small POST bodies these routes handle (sub-100 KB typically) the cost is negligible. If a future profiling pass flags it, the AdminApiService methods could grow typed overloads accepting the pre-parsed JSON to skip the second pass.
- **No new unit tests.** The `MasterControl::collectDroppedTopLevelKeys<T>(input, model)` helper has v0.10.17 unit tests against `LanClient`. The new call sites use `RuntimeEndpoint`, `SubAgentGroupDefinition`, and `LanClientPrivileges` — the helper is type-generic so the existing coverage transfers. Live POST probes verify each new call site. A future iteration could add per-model unit tests if a regression appears.
