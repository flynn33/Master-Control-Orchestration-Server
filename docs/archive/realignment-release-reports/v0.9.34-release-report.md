# Release Completion Report — v0.9.34

Maintenance release. Real production lease leak found and fixed.

## Scope completed

### Lease leak in `tools/call` — every gateway call leaked one lease

The v0.9.34 bug-hunt fired 10 parallel `tools/call` requests and watched `/api/pools/baseline-tools/saturation`:

```
pre  (fresh service):                  activeLeaseCount=0
post (10 parallel tools/call + 2s):    activeLeaseCount=10
post (5s longer settle):               activeLeaseCount=10  (still!)
post (30 more calls):                  would be 40
```

The leases sat in `/api/pools/baseline-tools/leases` with `state="active"` and `releasedAtUtc=""` — never released, never expired. Operator-visible consequences:

- The dashboard's per-pool **Active Leases** bar grew monotonically with every call.
- **Saturation calculations** treated phantom leases as real load; scale-out triggers fired prematurely.
- With `maxActiveLeasesPerInstance = 64`, a long-running gateway would eventually saturate and start rejecting calls with `"could not acquire instance lease"` — at which point the only recovery was a service restart (which clears the in-memory lease table).

**Root cause** in [`McpGatewayAdapters.cpp:1060-1066`](../../../src/MasterControlApp/McpGatewayAdapters.cpp): the source had a comment block that *promised* a `releaseLease` call:

```cpp
// Always release the lease. We acquired it for a single forward,
// and the simple least-loaded router does not require us to
// explicitly release short-lived leases for correctness, but
// honoring the contract keeps the lease accounting accurate
// when the operator inspects the dashboard.
// (LeaseRouter::releaseLease is idempotent on unknown ids.)
// Comment kept for clarity -- the call site is below.
```

…but no `releaseLease()` call followed. The comment must have been written alongside an intended implementation that was never added (or was deleted in a refactor and the comment was kept). Pre-v0.9.34 every single `tools/call` leaked one lease.

The fix calls `router->releaseLease(lease.leaseId, "tools/call complete")` unconditionally after `sendStdioJsonRpc` returns and before any of the success/error JSON-RPC return paths. `LeaseRouter::releaseLease` is idempotent on unknown ids, so racing scenarios (supervisor reaping the instance mid-call, double-release if a future refactor adds another path) are safe.

**Verified live**:

```
=== v0.9.34 ===
pre  (fresh service):                  activeLeaseCount=0
post (10 parallel tools/call + 1s):    activeLeaseCount=0  queueDepth=0
post (30 parallel tools/call + 1s):    activeLeaseCount=0  queueDepth=0
/api/pools/baseline-tools/leases       []  (empty)

=== regression: prior fixes ===
GET    /api/version                    -> 200
DELETE /api/version                    -> 405 (v0.9.28 still works)
POST   /api/leases/none/release        -> 404 (v0.9.33 still works)

```
