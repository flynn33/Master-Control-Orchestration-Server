# Release Completion Report — v0.9.45

Maintenance release. Closes the v0.9.44 carry: Apple remote host remove now returns 404 on unknown id.

## Scope completed

### `POST /api/platform-services/apple-hosts/remove` returns 404 on unknown hostId

The v0.9.44 release report carried this as deferred under the assumption that the underlying `removeAppleRemoteHost` helper was idempotent. The v0.9.45 source check found that wasn't true — the helper at [MasterControlRuntime.cpp:4583](../../../src/MasterControlApp/MasterControlRuntime.cpp#L4583) already distinguishes three outcomes:

| Helper return | Cause | v0.9.45 status |
|---|---|---|
| `succeeded: true, "Apple remote host removed."`               | hostId existed and was removed     | 200 |
| `succeeded: false, "Apple remote host '<id>' was not found."` | hostId not in the registry         | **404** |
| `succeeded: false, "Apple remote host removal requires a hostId."` | empty / missing hostId       | **400** |

The route was treating both failure cases as 400. The fix uses a substring match on `"was not found."` (substring because the message embeds the hostId itself in quotes) to distinguish them. Validation errors (empty hostId) and parse errors (malformed body, v0.9.44) continue to return 400.

**Verified live**:

```
=== v0.9.45 ===
POST /api/platform-services/apple-hosts/remove  body:{"hostId":"v0945-never-existed"}
  -> 404  "Apple remote host 'v0945-never-existed' was not found."

=== regression ===
POST /api/platform-services/apple-hosts/remove  body:{"hostId":""}
  -> 400  "Apple remote host removal requires a hostId."  (validation)
POST /api/platform-services/apple-hosts/remove  body:malformed
  -> 400  "Invalid Apple remote host removal payload: ..."  (v0.9.44 parse-error catch)

=== stable surfaces ===
GET /api/version                              -> 200
/api/health/summary                           -> ok:true, total:5, toolCount:33, darkOrQuarantined:0
tools/list via /mcp                           -> 33 tools (18991 bytes)
```
