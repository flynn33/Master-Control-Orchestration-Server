# Release Completion Report — v0.10.19

Three operator-flagged testing-phase defects fixed in a single patch:

1. `sqlite` was the last catalog placeholder; v0.10.18 deferred it because of a native-module crash. Root cause traced this turn.
2. `/api/health/summary.gateway.toolCount` was stale relative to the gateway's actual tools/list count (68 reported vs 121 live).
3. `mcpServerRuntimeStats[i].status` and `subAgentRuntimeStats[i].status` were decoupled from `.reachable` for stdio-supervised workers.

## 1. Findings → resolution

### Item 1 — sqlite crash root cause (testing-phase diagnostic)

**Reproduction:** Pre-v0.10.19 attempts to register sqlite via `executable=C:\Program Files\nodejs\npx.cmd args=[-y, mcp-server-sqlite-npx, <db>]` produced 171 instance respawns, 168 telemetry errors, 28 supervisor quarantine cycles in 30 minutes. Each spawn exited `exitCode=1` within ~1s.

**Tested ruled out:** path-spaces (crashed identically on `C:\ProgramData\MCOS\state\mcp-sqlite.db` with no spaces), db-file-presence (crashed with both fresh empty file AND 8 KB project sqlite copy), supervisor stdin redirection (other npx-based MCP packages — memory, filesystem, sequential-thinking, chrome-devtools — spawn cleanly through the same supervisor path).

**Root cause:** `mcp-server-sqlite-npx` depends on the native `sqlite3` node module (`node_sqlite3.node`). When `npx -y` is invoked under MCOS's service context (running as `S-1-5-18` / SYSTEM), npx tries to materialise the package into SYSTEM's npm cache at `C:\WINDOWS\system32\config\systemprofile\AppData\Roaming\npm-cache\_npx\…`. The native-binary download + load step fails under that account/path (other npx packages without native deps don't trigger this code path). Verified: the same `npx -y mcp-server-sqlite-npx` invocation succeeds when run from a regular user PowerShell session, AND succeeds when the supervisor's pool template points directly at the SYSTEM-account npm-globally-installed binary at `C:\WINDOWS\system32\config\systemprofile\AppData\Roaming\npm\mcp-server-sqlite-npx.cmd` (skipping the npx download/build step).

**Fix:** New seed block in `MasterControlRuntime.cpp` (~line 13823, immediately after the v0.10.18 `registerNpxMcpPool` batch). Resolves the SYSTEM-account npm-global binary path; if it exists, registers a sqlite pool pointing directly at it. Creates the db file at `C:\ProgramData\MCOS\state\mcp-sqlite.db` if absent (mcp-server-sqlite-npx requires an existing file). Skips entirely if the binary isn't installed, leaving sqlite as a catalog placeholder per the honest-unavailable-sentinel rule.

### Item 2 — `/api/health/summary.gateway.toolCount` staleness

**Reproduction:** Live test 2026-05-15: dashboard reported `toolCount: 68` while `POST /mcp tools/list` over the gateway returned 121 tools.

**Root cause:** `NativeHttpSysGatewayAdapter::ListTools()` at `McpGatewayAdapters.cpp:595` only called `refreshToolCatalogLocked()` when the cache vector was EMPTY. Pool upserts called `InvalidateToolCatalog()` which reset `toolCatalogCacheValidUntil_` but the cache vector itself stayed populated, so the next `ListTools()` call returned the stale snapshot. The TTL-expiry path only triggered when something else (the gateway's own tools/list MCP handler) hit `refreshToolCatalogLocked()` directly.

**Fix:** `ListTools()` now defers to `refreshToolCatalogLocked()` unconditionally (whenever `workerSupervisor_` is set). The refresh function's existing TTL self-bypass at line 660 (`if (now < toolCatalogCacheValidUntil_) return cache;`) keeps the hot path unchanged when the cache is still valid.

### Item 3 — `runtimeStat.status` decoupled from `.reachable`

**Reproduction:** Pre-v0.10.19 the dashboard's `mcpServerRuntimeStats[i].status` reported `"offline"` for every stdio-supervised pool worker (18 entries) even though `.reachable` correctly reported `true` and `readyInstanceCount` was 1.

**Root cause:** `MasterControlRuntime.cpp:11534` (sub-agent path) and `:11611` (mcp-server path) both set `stat.status = to_string(endpoint.status)`. `endpoint.status` is derived from `probeEndpoint(host, port)` (TCP connect probe). Stdio-supervised workers don't listen on TCP, so the probe always returns `EndpointStatus::Offline`. The `.reachable` field is computed correctly via a separate code path that consults the pool's instance state.

**Fix:** In both the sub-agent and mcp-server pool-enrichment branches (replace_all=true in the edit), after `readyCount` is computed, override `stat.status = "online"` when `readyCount > 0`. This couples the status string field to the same source of truth that the `.reachable` bool already uses.

## 3. Public contracts changed

- **`/api/health/summary.gateway.toolCount` accuracy (no breaking change).** Operators see the real count instead of a stale snapshot. Old behaviour was effectively a bug; the field's documented semantic is "current gateway tool catalog size" and v0.10.19 now matches.
- **`mcpServerRuntimeStats[i].status` + `subAgentRuntimeStats[i].status` values change for stdio-supervised pools** from `"offline"` to `"online"` when the pool has any Ready instance. Operators using the field as a synonym for `.reachable` now see consistent state. Operators using it as a literal TCP-probe result will see different output — but stdio-supervised workers were never TCP-probable to begin with, so the prior value was meaningless.
- **Sqlite pool seed (additive).** Fresh installs that have `mcp-server-sqlite-npx` installed for the SYSTEM account see sqlite as a reachable pool. Hosts without the binary continue to ship sqlite as a catalog placeholder.

No removed routes, no removed Shell controls, no removed pool ids.

## 5. Risks and blockers

- **SYSTEM-account binary install is currently manual.** The operator (or installer) must run `npm install -g mcp-server-sqlite-npx` then copy the package + bin script from the regular-user npm-global location into `C:\WINDOWS\system32\config\systemprofile\AppData\Roaming\npm\`. The installer will handle this in a future iteration. The seed's binary-exists check keeps the failure mode honest if the install hasn't happened.
- **Supervisor still ignores `pool.template_.environment`.** Scheduled for v0.10.20 alongside POST unknown-key parity. The sqlite seed sets `PATH` in the template for forward-compatibility but the override is a no-op until that fix lands.
