# Release Completion Report — v0.9.14

Maintenance release. Closes a real production-impact dashboard-latency bug surfaced by the v0.9.14 bug-hunt probe: `/api/dashboard` took **6.6 seconds warm, up to 20 seconds cold**.

## Scope completed

### Reachability probe TTL cache (180× dashboard speedup)

Pre-v0.9.14 the `DashboardSnapshot snapshot()` path made a serial 200ms TCP-connect probe to every sub-agent and every mcp-server endpoint on every call. With ~30 endpoints in a typical install (sub-agents + mcp-servers + variants per address family) the snapshot spent 6+ seconds on probes alone. Both the WinUI shell's snapshot loop and the browser dashboard's auto-refresh poll this endpoint, so every refresh blocked for that window.

v0.9.14 adds:
- **Per-`host:port` reachability cache**: `std::map<std::string, ReachabilityCacheEntry>` keyed by the operator-display string ("host:port" or "[ipv6]:port" per RFC 3986).
- **`probeReachabilityCached(host, port, &outHostPort)` helper**: cache lookup → hit returns immediately; miss does the same probe as before, stores the result, returns it.
- **30-second TTL**: initially set to 5s, but live measurement showed a cold sweep takes ~6s, so a 5s TTL was already stale by the time the sweep finished and EVERY subsequent snapshot re-probed everything. Bumped to 30s after the empirical measurement.

The probe code itself (non-blocking TCP connect, 200ms select timeout, getaddrinfo iteration over both IPv4 + IPv6) is unchanged. It just lives in one helper now instead of being duplicated ~60 lines × 2 inline at the sub-agent and mcp-server snapshot sites.

### Measured live impact

| Call | Latency |
|---|---|
| 1 (cold cache) | 6946 ms |
| 2 (cache populated mid-sweep) | 13492 ms |
| 3 (full cache hit) | **36 ms** |
| 4 (full cache hit) | **41 ms** |
| 5 (full cache hit) | **32 ms** |
| After 35s wait (TTL expired, re-probe) | 20046 ms |

Steady-state cache-hit latency ~36ms = **~180× faster than v0.9.13**. The periodic re-probe spike (every 30s) is a known limitation; v0.9.15 candidate is a background warm-up thread that probes async so the dashboard request is always cache-hit.

## Risks and blockers

- **Periodic re-probe spike** when TTL expires: the snapshot that lands ~30s after the previous probe sweep takes 6-20s while the cache rebuilds. v0.9.15 candidate: background async probe thread that keeps the cache warm so dashboard reads are always O(1).
- **`getaddrinfo` blocking time** isn't bounded by our 200ms select timeout — it's a separate phase that can take seconds for unresolvable hosts. The 30s TTL absorbs this in steady state but it dominates the periodic re-probe spike.
